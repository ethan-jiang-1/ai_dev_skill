# gstack 优缺点

## 先给总体判断

gstack 的优点不是“点子很多”，  
而是它把很多本来分散的 AI 工程能力做成了相互咬合的系统。

它的缺点也不是“写得不认真”，  
而是这套系统本身已经很厚，使用门槛、迁移门槛、维护门槛都会被抬高。

## 主要优点

## 1. 入口清晰，skill 信号非常强

很多 skill 仓库的问题是：

- 文件像文档，不像能力入口
- 工具权限不清
- 触发条件不清

gstack 在这点上做得很明确：

- `<skill>/SKILL.md` 结构统一
- front matter 明确写 name / description / allowed-tools
- README 和 `docs/skills.md` 又进一步补 usage map

这让使用者和 agent 都更容易识别能力边界。

## 2. 不是平铺 skill，而是 sprint 闭环

它真正的优势是把 skill 放在生命周期里：

- 想法 -> office hours
- 规划 -> CEO / Eng / Design / DX review
- 构建与排障 -> browse / investigate / design tools
- 验证 -> review / QA / security / benchmark
- 交付 -> ship / deploy / canary / docs / retro

这种组织方式比单纯 skill 列表更适合真实工程。

## 3. 浏览器 runtime 把 QA 从口号变成能力

`ARCHITECTURE.md` 里对 browser daemon 的设计非常关键。  
这让很多原本容易沦为空话的 skill 真正可执行：

- `/browse`
- `/qa`
- `/qa-only`
- `/design-review`
- `/benchmark`
- `/canary`

如果没有这层，很多“QA lead”“designer who codes”式 skill 只能停留在提示词层。

## 4. 模板系统明显降低了文档漂移

`SKILL.md.tmpl -> SKILL.md` 的生成链很值得肯定：

- 共享块能集中治理
- 命令参考能从代码同步生成
- skill 文档新鲜度能进 CI

这比完全手写 prompt 仓库稳很多。

## 5. 对 agent 失败模式有真实理解

从 preamble、Search Before Building、AskUserQuestion 规范、反跳步规则、错误重写这些细节看，  
gstack 很明显不是只在追求“更像人说话”，而是在认真处理 agent 的常见失败模式：

- 失焦
- 漏步骤
- 过度实现
- 不会恢复
- 不记上下文

这是高质量 AI 工程仓库很关键的特征。

## 6. 多宿主和长期使用意识很强

它支持：

- Claude Code
- OpenClaw
- Codex CLI
- Cursor
- OpenCode
- 其他 host

并且还加入：

- learnings
- timeline
- analytics
- upgrade
- team-init

这说明它关心的是“长期可用的 skill system”，不是一次性 demo。

## 7. 验证意识强，不只是“相信 prompt”

从 `package.json`、测试目录、evals 工作流能看出，它对 skill 的态度是：

- 要能测试
- 要能分层验证
- 要能跑 eval
- 要能查文档漂移

这点很加分，因为很多类似仓库最薄弱的就是验证层。

## 主要缺点

## 1. 系统很厚，小任务容易显得过重

这是 gstack 最直接的缺点。

对于：

- 小修小补
- 临时脚本
- 一次性试验
- 不想维护流程的个人项目

它很容易显得过度工程化。

## 2. 很多价值依赖 runtime，不是纯迁移型方法论

有些仓库的强项是“看完就能抄方法”。  
gstack 不是完全这样。

它很多高价值部分依赖：

- `browse/` runtime
- `design/` runtime
- `bin/*` 工具
- extension
- host 集成方式

所以你能学到理念，  
但不一定能低成本搬走全部能力。

## 3. preamble 很强，但也让 skill 变厚

统一 preamble 的好处明显，  
但它也带来副作用：

- skill 文件体量很大
- 新读者难快速抓住每个 skill 的独特部分
- 共用块一旦变复杂，理解成本会持续上升

这是一种典型的系统化收益与可读性成本交换。

## 4. workflow 关系有时偏隐式

相较于一些把 workflow 目录单列出来的仓库，  
gstack 的编排更多嵌在：

- skill 正文
- preamble
- runtime 脚本
- README 的方法叙述

这让它更自然，但也更容易让第一次分析的人低估其流程结构。

## 5. 对宿主和环境仍有要求

README 里已经写得很清楚，它需要：

- Claude Code 或其他宿主
- Git
- Bun
- 某些平台还要 Node
- 浏览器与 cookie 环境

这说明它不是那种零环境依赖、复制即用的知识仓库。

## 6. 规模扩大后，命名治理和演进治理会变得关键

现在仓库里已经有：

- 40 个 `SKILL.md`
- 37 个模板
- 多类辅助模块
- runtime 和 extension 代码

这种规模下，后续真正难的会是：

- skill 命名是否持续清晰
- 适用边界是否持续清晰
- 不同 skill 是否会重叠
- preamble 是否会继续膨胀

也就是说，它的维护难点已经从“有没有内容”升级成“怎么治理演进”。

## 7. 叙事很强，容易抬高预期

README 的叙事张力很强，这是优点，也是风险。

好处是：

- 用户会被完整工作流吸引

风险是：

- 初次接触者容易期待“装上就像一个 20 人团队”
- 如果没有相应基础设施，落差会比较明显

所以它更适合愿意配合流程、测试、浏览器基础设施的人，  
不太适合只想拿几句 prompt 马上出结果的人。

## 适用边界

## 最适合

- 有持续代码库的个人 builder
- 要并行跑多个任务的技术 founder
- 需要 review、QA、ship 闭环的团队
- 愿意维护规则、测试和部署流程的项目

## 一般适合

- 想学习 AI 工程流程设计的人
- 想借鉴 skill 模板化治理的人
- 想研究 browser-assisted agent 的人

## 不太适合

- 只想收集 prompt 模板的人
- 不愿意增加流程成本的人
- 环境很轻、任务很短的小脚本场景

## 最后一句优缺点总结

gstack 最大的优点，是它把 AI 工程从“写 prompt”推进到了“做系统”。  
gstack 最大的缺点，也是它已经是一个系统了，因此天然更重、更复杂、更依赖环境。
