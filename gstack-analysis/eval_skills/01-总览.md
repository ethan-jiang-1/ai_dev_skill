# gstack 总览

## 这套东西到底是什么

`gstack` 不是一个“给 Claude 多装几个 slash command”的轻量仓库。  
它更像一套把 AI 工程流程产品化后的技能系统：

- 前面有 product / strategy / architecture / design review
- 中间有 build / investigate / browse / QA
- 后面有 ship / land-and-deploy / canary / retro
- 旁边还挂着 memory、safety、upgrade、browser extension、多宿主接入

所以它的真正对象不是单个 skill，  
而是一个完整 sprint 的 AI 工作流。

## skill 版面里到底有什么

当前 snapshot 里最显式的 skill 单元有：

- 36 个顶层 skill 入口
- 4 个 OpenClaw 原生 skill
- 合计 40 个 `SKILL.md`
- 37 个 `SKILL.md.tmpl`

如果只看 Claude Code 主战场，顶层入口大致可分成下面几组。

- `/plan-ceo-review`
- `/plan-eng-review`

## 三、审查与验证层

## skill-like 单元的外围支撑

除了 `SKILL.md`，还有几类很关键的 skill-like 结构：

### 1. 模板层

- `<skill>/SKILL.md.tmpl`

这是技能内容的“可生成母版”，  
也是 gstack 降低文档漂移的核心手段。

### 2. 专题子模块

- `review/specialists/*.md`
- `qa/references/issue-taxonomy.md`
- `qa/templates/qa-report-template.md`

这说明部分 skill 已经从单文件 prompt 进化成复合技能包。

### 3. OpenClaw 原生技能

- `openclaw/skills/gstack-openclaw-office-hours/SKILL.md`
- `openclaw/skills/gstack-openclaw-ceo-review/SKILL.md`
- `openclaw/skills/gstack-openclaw-investigate/SKILL.md`
- `openclaw/skills/gstack-openclaw-retro/SKILL.md`

这层让 gstack 不只绑定 Claude Code。

## 它的结构重心在哪里

### 重心 1：浏览器能力

`ARCHITECTURE.md` 直接把 persistent browser daemon 视为核心。  
这说明 gstack 的底层假设是：

- agent 想真正做 QA 和 dogfooding，必须有眼睛
- 浏览器调用要快
- 浏览器状态要持久

### 重心 2：模板化技能治理

`SKILL.md.tmpl -> SKILL.md` 的链路说明它非常在意：

- 技能文档与代码一致
- 技能共享 preamble
- 公共块可复用

### 重心 3：多角色审查

从 CEO、Eng、Design、DX、Security、QA、Release 这些 skill 可以看出，  
它并不是在给单一“万能程序员 agent”加提示，  
而是在主动拆角色。

## 我对这套仓库的总体理解

如果只看表层，它像：

- 35 个左右的 slash command

如果看深一点，它其实是：

- 一套 AI 工程组织结构
- 一套浏览器 + 设计工具 + 脚本工具混编的 skill runtime
- 一套把 prompt、脚本、测试、CI、升级、学习系统绑在一起的产品化仓库

## 最值得学的地方

- skill 不是平铺，而是围绕 sprint 链路组织
- 浏览器 runtime 让 QA 和设计审查真正有操作性
- 模板系统减少 skill 文档漂移
- preamble 把 update / telemetry / learnings / session tracking 统一下沉
- OpenClaw 与其他 agent host 适配让仓库有分发意识

## 最大的阅读提醒

看 gstack 时不要只盯：

- README
- 命令数量
- 营销文案

真正该盯的是：

- `SKILL.md.tmpl`
- `ARCHITECTURE.md`
- `browse/src/*`
- `review/specialists/*`
- `package.json`
- `.github/workflows/*`

因为那几层才是它从“prompt 仓库”进化成“系统仓库”的关键。
