# gstack 结构拆解

## 先说结论

gstack 的结构不是“技能目录 + 一点脚本”，  
而是五层叠起来的：

- 入口 skill 层
- 模板与共享块层
- runtime 与可执行工具层
- 集成与分发层
- 测试、CI、评估层

它真正强的地方不在某一个单独目录，  
而在这些层之间的咬合。

## 1. 入口 skill 层

### 路径特征

- `<skill>/SKILL.md`
- 根级 `SKILL.md`

代表目录：

- `office-hours/`
- `plan-ceo-review/`
- `review/`
- `qa/`
- `ship/`
- `browse/`
- `cso/`
- `autoplan/`

### 这一层在做什么

- 对外暴露 skill 名称
- 写 description 和使用场景
- 声明 allowed-tools
- 定义执行纪律
- 作为 Claude Code 加载时直接读取的能力入口

### 共同结构

从 `office-hours/SKILL.md` 和根级 `SKILL.md` 可以看到，这类文件普遍包含：

- YAML front matter
- `name`
- `preamble-tier`
- `version`
- `description`
- `allowed-tools`
- 大段操作性正文

这说明它们不是“说明文档”，  
而是带工具权限和执行约束的能力声明。

## 2. 模板与共享块层

### 路径特征

- `<skill>/SKILL.md.tmpl`
- `scripts/gen-skill-docs.ts`

### 这一层的关键作用

这一层在解决一个 prompt 仓库最常见的问题：

- 人写的说明容易漂移
- 命令新增后文档忘了更新
- 多个 skill 里重复的流程块会走样

gstack 的解法是：

- 人写模板
- 代码生成可提交的 `SKILL.md`
- 用 placeholder 把共用块注入多个 skill

### 这层的重要 placeholder

`ARCHITECTURE.md` 里点名了几类关键块：

- `{{PREAMBLE}}`
- `{{COMMAND_REFERENCE}}`
- `{{SNAPSHOT_FLAGS}}`
- `{{QA_METHODOLOGY}}`
- `{{DESIGN_METHODOLOGY}}`
- `{{TEST_BOOTSTRAP}}`
- `{{REVIEW_DASHBOARD}}`
- `{{CODEX_PLAN_REVIEW}}`

这意味着：

- 技能共性被集中治理
- 技能差异仍保留在模板正文
- 生成后的 skill 又能直接被宿主消费

## 3. runtime 与可执行工具层

这层是 gstack 和许多纯 Markdown 仓库的最大分水岭。

### A. `browse/`

路径：

- `browse/src/*.ts`
- `browse/test/*.test.ts`

这一层负责：

- persistent Chromium daemon
- localhost HTTP server
- CLI 到 daemon 的通信
- refs 系统
- screenshot / snapshot / click / fill 等命令
- cookie 导入与安全处理

这部分让 `/browse`、`/qa`、`/design-review` 获得真实执行力。

### B. `design/`

路径：

- `design/src/*.ts`
- `design/test/*.test.ts`

这一层负责：

- 设计变体生成
- 视觉对比
- 设计 session 和记忆
- 设计到代码相关流程

所以设计类 skill 不是只有 prompt，而是有程序支撑。

### C. `bin/`

路径：

- `bin/gstack-config`
- `bin/gstack-update-check`
- `bin/gstack-team-init`
- `bin/gstack-learnings-log`
- `bin/gstack-learnings-search`
- `bin/gstack-timeline-log`
- `bin/gstack-timeline-read`
- `bin/gstack-analytics`

这一层负责：

- 配置管理
- 升级检查
- 团队模式 bootstrap
- 学习记录
- 会话轨迹
- 本地分析

它把很多“应该存在但经常没人做”的运维动作产品化了。

### D. `extension/`

这一层提供：

- 浏览器侧栏 UI
- popup / inspector / background 逻辑
- 浏览器侧辅助入口

这说明 gstack 想让浏览器不是单纯被动执行器，  
而是 agent 可操作界面的一部分。

## 4. 集成与分发层

### A. OpenClaw

路径：

- `openclaw/skills/*/SKILL.md`
- `docs/OPENCLAW.md`
- `openclaw/*.md`

作用：

- 把部分方法论抽成 OpenClaw 原生技能
- 让 Claude Code skill 与 OpenClaw session 可以协作

### B. 多 agent host

README 明确支持：

- Codex CLI
- OpenCode
- Cursor
- Factory Droid
- Slate
- Kiro

这说明 gstack 的分发单位不是单一 IDE，而是 skill pack。

### C. 项目级接入

路径：

- `setup`
- `bin/gstack-team-init`
- `CLAUDE.md`
- `AGENTS.md`

作用：

- 全局安装
- 团队模式
- 项目级路由规则注入
- 多宿主说明

## 5. 测试、CI 与评估层

### 测试层

`package.json` 可以看到它把验证分成多档：

- 普通单测
- e2e
- evals
- gate
- periodic
- codex / gemini 专项
- skill audit

这和普通仓库只跑一个 `npm test` 的思路不同，  
它明显在按 skill / eval / 模型评审来设计验证体系。

### CI 层

`.github/workflows/` 当前包括：

- `actionlint.yml`
- `ci-image.yml`
- `evals.yml`
- `evals-periodic.yml`
- `skill-docs.yml`

可以看出它在持续检查：

- workflow 合法性
- CI 镜像
- 评估集
- skill 文档是否过期

## 6. 结构上的真正重点

### 重点一：skill 不止一层

最表层是 `/review`、`/qa`、`/ship` 这些入口。  
但其下还有：

- specialist
- template
- runtime
- bin 工具
- extension

### 重点二：很多工作流被折叠进 skill 本身

和一些把 workflow 独立放在 `workflows/` 目录的仓库不同，  
gstack 更像把编排逻辑嵌在：

- 生成后的 SKILL 正文
- 共用 preamble
- runtime 工具

所以它的 workflow 更隐式，但不代表更弱。

### 重点三：仓库结构围绕“长期使用”设计

从 telemetry、learnings、timeline、upgrade、team-init 这些层面看，  
gstack 不是在服务一次性对话，  
而是在服务长期、多项目、多 session 的 AI 工程实践。

## 结构拆解后的总体判断

如果把 gstack 只按“prompt 仓库”看，它会显得过重。  
如果把它按“AI 工程系统”看，它的结构反而是合理的：

- prompt 负责决策与流程
- runtime 负责执行力
- templates 负责一致性
- tests / evals 负责约束
- integration 负责分发

这五层合在一起，才是 gstack 的完整结构。
