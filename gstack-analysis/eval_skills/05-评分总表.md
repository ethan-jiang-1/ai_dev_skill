# gstack 评分总表

## 评分说明

这里先用主体分析框架的 1-5 分做横向整理：

- 1 = 很低
- 3 = 中等
- 5 = 很高

本表不是最终收口评分，  
而是把 gstack 的关键 skill 类别分开盘点，方便后面在总评里再做更苛刻的 1-10 六维收口。

## 评估对象切分

我把 gstack 拆成五类对象：

- 顶层 skill 入口系统
- 模板化技能治理系统
- 浏览器与设计 runtime
- 审查与交付闭环
- 多宿主、学习与运维系统

## 分项解读

## 1. 顶层 skill 入口系统

### 评分

- 结构化程度：5
- 可验证性：4
- 复用性：4
- 工具依赖：3
- 协作价值：5
- 可维护性：4
- 学习成本：3
- 落地门槛：3

### 为什么这样打

这一层有最强的显式信号：

- 目录即 skill
- front matter 清晰
- 角色定位稳定
- README / docs/skills 又帮忙建立地图

它的验证性不如模板系统那么高，  
但作为入口设计已经很稳。

## 2. 模板化技能治理系统

### 评分

- 结构化程度：5
- 可验证性：5
- 复用性：5
- 工具依赖：3
- 协作价值：4
- 可维护性：4
- 学习成本：4
- 落地门槛：3

### 为什么这样打

这是 gstack 最工程化、也最可迁移的一层。  
它把“prompt 文档容易漂移”这个常见难题处理得很成熟。

之所以学习成本打 4，  
是因为理解模板、占位符、生成链仍然需要一定上下文。

## 3. 浏览器与设计 runtime

### 评分

- 结构化程度：5
- 可验证性：4
- 复用性：3
- 工具依赖：5
- 协作价值：4
- 可维护性：3
- 学习成本：4
- 落地门槛：5

### 为什么这样打

这层是最硬核、也最依赖环境的一层。

优点：

- 真正带来执行力
- 让浏览器和设计类 skill 不空

代价：

- 环境要求高
- 迁移成本高
- 维护复杂度也更高

## 4. 审查与交付闭环

### 评分

- 结构化程度：5
- 可验证性：5
- 复用性：4
- 工具依赖：4
- 协作价值：5
- 可维护性：4
- 学习成本：4
- 落地门槛：4

### 为什么这样打

这一层体现的是 gstack 的流程设计功底：

- 不是“有 review”
- 而是有完整 review pipeline

它对多角色协作和长期项目尤其有价值，  
但落地仍需要团队接受流程纪律。

## 5. 多宿主、学习与运维系统

### 评分

- 结构化程度：4
- 可验证性：4
- 复用性：4
- 工具依赖：4
- 协作价值：5
- 可维护性：3
- 学习成本：4
- 落地门槛：4

### 为什么这样打

这一层最能体现产品化意识：

- 不只考虑单次运行
- 还考虑升级、学习、团队模式、跨宿主

但越往这层走，系统治理难度也越高，  
所以可维护性我没有给很高。

## 横向比较后的结论

### 最强的两层

- 模板化技能治理系统
- 审查与交付闭环

原因是：

- 一个解决 skill 漂移与复用
- 一个解决 AI 工程真正的流程闭环

### 最有差异化的层

- 浏览器与设计 runtime

因为很多仓库会写流程，  
但很少把浏览器执行层做成长期可用的底座。

### 最容易被低估的层

- 多宿主、学习与运维系统

这层不如 `/qa`、`/review`、`/browse` 那么显眼，  
但它决定 gstack 是不是一个长期系统。

## 主体框架总分

为了便于横向感知，我把五类对象的 8 项分数做简单平均：

| 对象 | 平均分 |
|---|---:|
| 顶层 skill 入口系统 | 3.9 |
| 模板化技能治理系统 | 4.1 |
| 浏览器与设计 runtime | 4.1 |
| 审查与交付闭环 | 4.4 |
| 多宿主、学习与运维系统 | 4.0 |

## 主体框架结论

如果只看主体 1-5 分框架，  
gstack 整体已经是一个明显高于普通 skill 仓库的系统型项目。

但它的最终总评还不能只看平均分，  
因为：

- 工具依赖高不一定是坏事
- 差异化能力不能被简单平均冲淡
- AI agent 特异性和失败模式覆盖要在更苛刻框架里看

所以最终判断，放到 `gstack-全量评估.md` 里用 1-10 六维结构收口。
