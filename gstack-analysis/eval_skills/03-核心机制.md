# gstack 核心机制

## 先说最关键的三个机制

如果要抓 gstack 最核心的东西，我会优先抓这三条：

- `SKILL.md.tmpl -> SKILL.md` 的技能生成机制
- `/browse` 背后的 persistent browser daemon
- 围绕 preamble、review、QA、ship 形成的流程闭环

这三条合起来，决定了它不是普通 prompt 仓库。

## 1. 技能生成机制

### 它在解决什么问题

skill 仓库最大的问题之一是：

- 命令更新了，文档没更新
- 多个 skill 共享规则时，拷贝后逐渐分叉
- agent 实际读到的 skill 与开发者以为的 skill 不是同一个东西

gstack 的解法是：

- 人维护 `SKILL.md.tmpl`
- 代码根据 runtime 元数据生成 `SKILL.md`
- 生成产物直接提交进仓库

### 这带来的结果

- Claude 真正读取的文件是稳定的
- 文档新鲜度能被 CI 检查
- 命令参考、flag 说明、共用 methodology 可以自动注入

### 这机制为什么比“全手写 skill”强

因为它兼顾了：

- 人写的判断力
- 代码生成的一致性
- Git 可追踪性

它没有把 skill 完全 runtime 动态化，  
而是选择“生成后提交”，这是一个很工程化的折中。

## 2. preamble 统一引导机制

### 它在做什么

从根级 `SKILL.md` 和 `office-hours/SKILL.md` 可以看到，  
多个 skill 开头都共享一个很长的 preamble。

这一层统一处理：

- upgrade check
- session tracking
- proactive 行为开关
- routing 规则检查
- telemetry 询问与记录
- learnings 检索
- timeline 记录
- vendored 安装检测

### 这说明什么

这说明 gstack 已经不把 skill 当成“单轮 prompt”了。  
它把每次 skill 执行都当成一个带上下文、带状态、带后续影响的系统事件。

### 这层的价值

- skill 行为更一致
- 新能力能跨 skill 下发
- 运维型逻辑不会散落到每个 skill 自己维护

### 这层的代价

- 每个 skill 都会变厚
- 理解门槛会上升
- 一旦 preamble 设计失误，影响面会很大

## 3. `/browse` 的 daemon 机制

### 它为什么重要

`ARCHITECTURE.md` 说得非常明确：  
浏览器是硬部分，其他大多是 Markdown。

核心逻辑是：

- 第一次调用时拉起浏览器 server
- CLI 通过 localhost HTTP 跟 server 通信
- server 持续持有 Chromium
- cookie、tab、login、localStorage 都持续存在

### 它解决的根问题

如果每个浏览器命令都重新启动 Playwright：

- 慢
- 会丢状态
- QA 很难连续做

daemon 模式带来的是：

- 首次启动成本一次付清
- 后续命令子秒级
- 真实页面状态可连续追踪

### 它为什么是 skill 系统的底座

因为一旦浏览器可持续，很多 skill 才真正从“建议型”升级成“执行型”：

- `/browse`
- `/qa`
- `/qa-only`
- `/design-review`
- `/benchmark`
- `/canary`

它们的可信度都直接依赖这层。

## 4. ref system 与 agent 可操作性

### 核心思路

gstack 没走“往 DOM 注入 ref 属性”的路线，  
而是基于 accessibility tree 与 Playwright Locator 去映射 `@e1`、`@e2` 这类引用。

### 这个设计的好处

- 不碰 DOM
- 不怕 CSP
- 不容易被 React / Vue / Svelte hydration 冲掉
- 对 Shadow DOM 更稳

### 为什么这对 agent 友好

agent 不需要写 CSS selector，  
只要：

- 先 snapshot
- 再用 `@ref`

这极大降低了浏览器操作的脆弱性。

## 5. 错误信息重写机制

`ARCHITECTURE.md` 里有一条很关键：

- 错误是给 AI agent 看的，不是给人看的

这意味着它不是把原始 Playwright error 原封不动丢给模型，  
而是改写成：

- 更短
- 更可执行
- 直接附下一步动作

这个细节很值钱，因为 agent 的恢复能力很大程度取决于错误提示是否可操作。

## 6. 多角色 review pipeline

### 表层上看

README 里是多个 slash command：

- CEO review
- Eng review
- Design review
- DX review
- Security
- QA
- Ship

### 机制上看

它们不是并排平铺，而是一个阶段链：

- 先定义和重构问题
- 再锁架构和设计
- 再实现
- 再 review / QA / security
- 再 ship / deploy / canary
- 最后 retro / learn

### 这机制的意义

它把 AI 的价值从“生成代码”推到“分阶段降低决策风险”。

## 7. review 与 QA 的复合化机制

### `/review`

除了 `review/SKILL.md` 本体，它还有：

- specialist 模块
- checklist
- design checklist
- triage 文档

说明 `/review` 实际是一个可拆解的审查框架。

### `/qa`

除了 `qa/SKILL.md` 本体，它还有：

- issue taxonomy reference
- QA 报告模板

说明 `/qa` 不只是“去点点页面”，  
而是把问题分类和交付格式标准化了。

## 8. learnings、timeline、telemetry 机制

### learnings

- 记录项目级经验
- 下次 skill 运行时可检索
- 让多 session 能继承过去经验

### timeline

- 记录 skill 开始与过程事件
- 支撑 retro、analytics、状态追踪

### telemetry

- 默认关闭
- 明确区分 community / anonymous / off
- 只上报有限元信息

这三层合起来，让 gstack 不只是一次性技能调用器，  
而是一个会记忆、会统计、会自我改进的系统。

## 9. 测试分层机制

从 `package.json` 和 `ARCHITECTURE.md` 看，它至少有三层验证思想：

- 便宜的静态验证
- 贵但真实的 e2e
- LLM-as-judge 的补充判断

这比很多 prompt 仓库只做“人工点一下”强很多，  
因为它已经在认真思考 skill 如何被测试。

## 10. 为什么这些机制能拼成闭环

闭环大致是这样的：

- 模板系统保证 skill 结构稳定
- preamble 保证 skill 有统一系统行为
- browse / design runtime 提供真实执行力
- review / QA / ship 把风险分阶段压下去
- learnings / timeline / telemetry 让系统跨 session 累积经验
- evals / CI 防止技能系统悄悄退化

这套闭环最强的地方在于：  
它没有把“AI agent 提升”寄托在单个 prompt 写得多好，  
而是把 prompt、runtime、验证、记忆、分发一起做。

## 核心机制总判断

gstack 最强的核心机制不是某一个天才 prompt，  
而是下面这个组合：

- 模板化技能治理
- 可持续浏览器执行层
- 多角色评审流水线
- 跨 session 的学习与运维层

这也是它和普通 skill 仓库真正拉开差距的地方。
