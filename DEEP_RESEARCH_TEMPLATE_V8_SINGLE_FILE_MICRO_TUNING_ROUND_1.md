# DEEP_RESEARCH_TEMPLATE_V8 单文件微调 Round 1

> target_template: `DEEP_RESEARCH_PROGRESSIVE_PLAN_TEMPLATE_V8.md`
> round_goal: `在单文件前提下，先统一权威段落，消灭最重的重复块`
> round_scope: `只规划并实施单文件内的 authority consolidation，不扩方法论，不拆文件`

## 背景

这轮不是做 `V9`，也不是重写模板。

这轮只解决一个最大问题：

- `单文件内部的核心规则重复出现，导致权威源不唯一`

这个问题如果不先处理，后续任何“再加一点规则”“再补一点说明”都会继续把模板推向更重、更难维护、更容易漂移的状态。

单文件是硬约束，因此本轮策略不是拆分，而是：

- 在单文件内部建立清晰层级
- 让每类核心规则只有一个完整版本
- 让其他位置只保留最短可执行提醒

## 本轮为什么先打这个问题

这一步是上游问题，先于下面这些问题：

- skeleton 过重
- status 伪精确
- 写入同步摩擦过高
- 游戏性不足

原因很直接：

- 它同时伤害结构性与简洁性
- 它会放大后续所有新增规则的复杂度
- 它不要求推翻方法论，适合小步改
- 它对执行语义影响最小，但对阅读成本改善最大

## 本轮硬约束

- 保持单文件
- 不拆章节为多个文件
- 不修改 `Wave 0 / Wave 1 / Wave 2 / Readiness Check` 的总体方法论
- 不修改 `Status File Skeleton` 的字段集合
- 不修改 `证据采集协议` 的主体结构
- 不新增新的协议或新的波次
- 不做“顺手优化”式扩张

## 当前最大问题的具体表现

当前 `V8` 中，至少有 7 类核心规则存在“完整版本 + skeleton 内重复版本 + reminder 再重复”的问题。

这些重复不是普通重复举例，而是会导致：

- 维护时不知道哪一段是正式版本
- skeleton 看起来像“再读一遍整部宪法”
- 后续改一处容易漏改另一处
- 文档长度上升，但新增信息密度不升反降

## 本轮具体要改什么

### 1. 保留前部完整协议，不动其方法论语义

以下前部章节作为权威定义保留：

- `## 自主执行协议（Autonomous Execution Protocol，MUST READ）`
- `## Topology Formalization Gate`
- `## Exploration-Exploitation Decision Framework`
- `## Foundation Sufficiency Check（Wave 0 → Wave 1）`
- `## Early Saturation Protocol`

本轮只允许做极少量措辞压缩，不允许改变这些段落的规则语义。

### 2. skeleton 中的重复规则一律降级为“短提醒”

`## 可直接复制的 Plan Skeleton` 里的重复协议不再保留完整解释。

具体处理原则：

- 如果前文已有完整规则，skeleton 里只保留“这一轮必须遵守”的短提醒
- skeleton 中的提醒必须服务于实例化
- skeleton 中不再承担“再次定义协议”的职责

具体目标：

- `296-304` 保留为极短版自主执行提醒，明确“完整规则见前文”
- `436-450` 改为 topology 提醒，不再全文复述 gate
- `452-475` 改为探索 / 利用决策提醒，不再完整展开信号列表
- `587-596` 改为 Wave 0 进入 Wave 1 前的短检查
- `909-925` 改为 stop condition 中的引用式提醒

### 3. reminder 类章节只保留扫描价值，不保留第二份规则正文

这类块的功能应从“再讲一遍规则”改成“帮助扫读和执行”：

- `803-810` 的 `自主执行协议提醒`
- `863-873` 的 `Human-on-the-loop 原则`
- 底部 `1186-1206` 的“真正要守住的东西”

具体处理：

- `803-810` 保留 2 到 4 行的导航式提醒
- `863-873` 只保留与 `suspend` 决策强相关的最短说明，不再展开完整人工介入条件
- `1186-1206` 保留为高层 invariant 列表，但不再塞入长句式的完整执行协议复述

### 4. 本轮不移动大章节，只做“单文件内部降重”

本轮不做下面这些大动作：

- 不把后面的协议整体前移
- 不改变 skeleton 与 status skeleton 的边界
- 不重做目录顺序
- 不引入新的总览页或 mini index

原因：

- 这是 Round 1，不做大手术
- 先把单文件里的重复收掉，再决定后续是否需要更深的单文件重排

## 本轮明确不做什么

- 不重写 `总体策略`
- 不重写 `研究线的具体目标`
- 不改 `并行执行建议`
- 不改 `执行节奏`
- 不改 `Hard Gates`
- 不改 `Status File Skeleton`
- 不改 `Reference` 文档 schema
- 不新增游戏化字段
- 不讨论多文件化

## 编辑顺序

本轮执行时，严格按下面顺序编辑 `V8`：

1. 先锁定前部权威段落，不改标题，只确认保留位置。
2. 再进入 skeleton，把重复块压成短提醒。
3. 再压缩 reminder 区块，防止留下第二份正文。
4. 最后压缩底部 invariant 列表中的重复句子。
5. 通读一遍，确认没有某条规则被删到“只剩提醒，没有正文”。

不允许反向顺序：

- 不能先删 reminder，再发现前文也被改薄了
- 不能先大量压缩 skeleton，再回头补权威版

## 本轮的写法规则

Round 1 编辑时必须遵守下面这些写法规则：

- 一个规则族只能有一个“完整定义”
- 同一规则在其他位置最多保留：
  - 一句提醒
  - 一句“完整规则见前文某节”
  - 一句与当前局部上下文强相关的执行提示
- reminder 只负责导航，不负责再定义协议
- 不为了压行数而删掉真正关键的局部操作提醒
- 不改 heading 名称，除非改名能直接减少歧义且不影响可定位性

## 验收标准

Round 1 完成后，必须同时满足下面条件：

- 模板仍然是单文件
- 每类核心规则只有一个完整版本
- skeleton 内不再出现第二份近似全文的协议
- reader 能在 30 秒内定位到每类核心规则的正式版本
- 总篇幅有实质下降，而不是换位置重复
- `Plan Skeleton` 仍可直接复制实例化
- 执行语义不变

建议的量化目标：

- 净减少至少 `60` 行
- 不减少任何一个核心协议的可执行性

## 风险与防误伤

### 风险 1：压缩过度，导致 skeleton 失去独立可用性

防法：

- skeleton 里保留最短可执行提醒
- 不把所有协议都变成“自行理解”

### 风险 2：删掉重复时，把唯一权威段落也一起削薄

防法：

- 先锁定 canonical section，再删其他位置
- 删除前后逐项核对“哪一段是正式版本”

### 风险 3：底部 invariant 列表被压得失去总结价值

防法：

- 允许保留高层原则
- 只删“再次讲完整执行协议”的句子

## 完成后的检查清单

- `自主执行协议` 是否只剩一个完整版本
- `Topology Formalization Gate` 是否只剩一个完整版本
- `Exploration-Exploitation` 是否只剩一个完整版本
- `Foundation Sufficiency Check` 是否只剩一个完整版本
- `Early Saturation Protocol` 是否只剩一个完整版本
- `Suspended Branch / human-on-the-loop` 是否只剩一个详细协议版本
- skeleton 是否仍可复制
- status skeleton 是否完全未动
- 单文件约束是否完全满足

## 下一轮入口

只有本轮完成并稳定后，才进入 Round 2。

Round 2 的主题才是：

- `压缩 skeleton，只保留实例化必须内容`

Round 1 未完成前，不提前进入：

- status schema 精简
- 写入摩擦降低
- 游戏性增强

## 这轮真正的目标

这轮不是让模板“更强”，而是让模板先“更稳”。

如果 Round 1 做对了，后面所有微调都会更便宜；如果 Round 1 跳过，后面的每一次优化都还会继续在重复块上付利息。
