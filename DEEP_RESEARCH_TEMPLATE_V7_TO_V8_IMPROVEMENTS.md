## 这份文件现在负责什么

这份文件不再保存“某一轮应该加哪 9 条建议”这种短期提案。

它现在只保存对后续模板演进仍然稳定成立的原则，尤其是：

- 不要把 `简洁性` 误解成“尽量压缩”
- 模板优化的首要目标不是更短，而是让 LLM 更可控地执行
- 允许发挥的地方要给空间
- 不允许发挥的地方要有工程性的硬边界

以后如果这份文件与某一轮临时控制稿冲突：

- 具体实施顺序以当轮控制稿为准
- 长期取向与裁剪原则以本文件为准

## 最高原则

### 1. 不以“更短”作为目标

对这类长程执行模板来说，`简洁性` 不是压缩篇幅本身。

这里的 `简洁性` 只应该表示：

- 低噪音
- 少重复版本
- 低歧义
- 好消费

不表示：

- 规则越少越好
- 行数越短越好
- 说明越省越好

如果压缩会削弱硬约束、让规则变隐含、或让 LLM 需要自行补完关键边界，那么这种压缩就是负优化。

### 2. 模板优化的首要目标是 `可控执行`

真正的优先级是：

1. `约束强度`
2. `单点权威`
3. `可机械执行`
4. `低歧义`
5. `简洁性`

如果 `简洁性` 和前四项冲突，优先保留前四项。

### 3. 不能一刀切地反对重复

重复有两种：

- `坏重复`
  - 同一规则出现多个近义版本
  - 不同段落各讲一遍但措辞不完全一致
  - 模板自我解释、消费提示、元说明过多
- `好重复`
  - 对硬约束的局部加固
  - canonical 定义之外的必要局部提醒
  - 为了防止 LLM 漂移而保留的短提醒

结论：

- 不应把“所有重复”都视为问题
- 只应删除 `坏重复`
- `好重复` 只要能提升执行稳定性，就可以保留

## 三层控制模型

后续所有模板调优，默认都按这三层判断“这里到底该不该让 LLM 发挥”。

### A. Hard Contract

这里不允许自由发挥，必须有硬边界。

典型对象：

- object boundary
- `Source of Record`
- enum allowed values
- mode / gate transition
- skeleton copy boundary
- `QUEUE_PATH` authority
- interrupt conditions
- stop / readiness / success / hard-gate 的层次关系

对这一层的要求：

- 规则必须显式
- 语义必须稳定
- 可以有适度重复加固
- 不能把关键约束压成“理解一下”

如果某次改动让 Hard Contract 更依赖推断，而不是更依赖显式定义，这次改动就是退步。

### B. Bounded Flexibility

这里允许发挥，但必须在边界内。

典型对象：

- topic formalization
- early saturation judgment
- branch disposition after evidence review
- artifact 组织方式
- evidence-summary / question-list 的结构化表达

对这一层的要求：

- 给判断框架
- 给边界
- 不把每一步写死

如果某次改动把这层写成完全自由发挥，边界会丢。
如果某次改动把这层写成死板清单，模型会失去必要适应性。

### C. Open Heuristic

这里应该允许发挥，不要过度工程化。

典型对象：

- 搜索路径
- 资料探索顺序
- 综合表达顺序
- 具体分析展开方式
- 某条研究线内部的追问策略

对这一层的要求：

- 给目标
- 给评价方向
- 不预先替 LLM 决定所有动作

## 调优时的判断规则

### 1. 先问“这条规则属于哪一层”

每次想删、想压、想补之前，先判断它属于：

- `Hard Contract`
- `Bounded Flexibility`
- `Open Heuristic`

不同层不能用同一套裁剪标准。

### 2. 如果是 Hard Contract，优先保显式性

对于 Hard Contract：

- 宁可略重，不可过隐
- 宁可有短提醒，不可只剩暗示
- 宁可局部重复，不可多版本漂移

### 3. 如果是 Bounded Flexibility，优先保边界

对于 Bounded Flexibility：

- 可以压缩语言
- 但不能删掉判断边界
- 不能把“受控发挥”误压成“随便发挥”

### 4. 如果是 Open Heuristic，优先保空间

对于 Open Heuristic：

- 不要为了工程感把它写死
- 不要把策略层全部固化成 checklist
- 只要目标清楚、边界清楚，就允许模型在方法上发挥

### 5. 不以行数衡量质量

以后不要把“文档长”本身当成问题。

只有下面这些才算问题：

- 同一规则有两个以上完整版本
- 消费路径过长
- 模板自我解释太多
- canonical point 不够清楚
- LLM 容易读不出哪里能发挥、哪里不能发挥

## 明确禁止的错误优化

### 1. 为了压缩，把硬约束压成隐含规则

这是最应该避免的错误。

例如：

- object boundary 只剩暗示
- enum 不再集中定义
- queue authority 需要从多个段落拼出来
- mode transition 需要读者自己综合理解

### 2. 把“内容多”误判为“简洁性差”

内容多不自动等于差。

只有在内容多导致：

- 重复规则面
- 噪音过高
- 多版本语义
- 消费困难

时，才算问题。

### 3. 把所有重复都当成坏事

如果一条短提醒能有效防止 LLM 在高风险边界上漂移，这种重复可能是合理的。

### 4. 把 Bounded Flexibility 误做成 Open Heuristic

有些地方不是“完全自由发挥”，而是“在明确边界内发挥”。

例如：

- `early_saturation`
- `topic promotion`
- `suspend / archive / redirect`

这些都不该被压成纯感觉判断。

### 5. 把 Open Heuristic 误做成 Hard Contract

有些地方不该过度工程化。

例如：

- 资料搜集顺序
- 分析展开顺序
- 单条 topic 内的探索策略

这些地方如果写得太死，会降低真实执行质量。

## 允许保留适度重复的场景

下面这些场景，允许保留适度重复：

- canonical layer 已定义，但 skeleton 周边仍需要一句短提醒
- 某个高风险边界如果只在前文出现一次，LLM 很容易在执行时忘掉
- 某个字段虽然已有统一定义，但在使用点保留短注释能显著降低误填概率

保留重复时必须满足：

- 只有一个完整定义点
- 其他位置只是短提醒，不是第二份完整规则
- 提醒内容不能和 canonical point 打架

## 对 V8 及后续版本的具体含义

### 对 `简洁性` 的重新定义

以后再讨论 `V8` 的 `简洁性`，默认只是在问：

- 是否还有低增益说明层
- 是否还有第二份近义规则
- 是否还有模板自我解释污染实例消费
- 是否还有不必要的多段拼接语义

不是在问：

- 能不能再少 200 行
- 能不能再压得更短

### 对 `一致性` 的重新定义

以后再讨论 `一致性`，默认重点看：

- 是否仍只有一套 canonical enum
- 是否仍只有一套 object authority
- 是否仍能机械判断 mode / gate transition
- 是否仍清楚区分
  - `topic_stop_decision`
  - `primary_source_status`
  - `Suspended Branches.state`

### 对 future tuning 的默认操作

以后如果要继续调模板，优先顺序应该是：

1. 先查是不是 `坏重复`
2. 再查是不是 `hard boundary` 被写得太隐
3. 再查是不是 `bounded flexibility` 被写得过松或过死
4. 最后才考虑篇幅与版式

以后编辑这份文件时，遵守下面规则：

- 只保留长期稳定成立的原则
- 把一次性的迁移建议、已完成事项、过时提案删除

- 不把这里写成新的 backlog
- 不把这里写成某一轮实施清单

- 作为长期“调优取向记忆”
- 在后续轮次里提醒：不要为了形式上的简洁而牺牲工程性的可控执行

## 一句话记忆

- 不要少掉关键边界
