# Deep Research Progressive Plan Template V7 → V8 改进建议

> 文档创建时间：2026-04-16  
> 针对模板：`DEEP_RESEARCH_PROGRESSIVE_PLAN_TEMPLATE_V7.md`  
> 建议版本：V8  
> 核心改进目标：**静默执行长程任务** + **细节打磨**

---

## 执行摘要

你的 V7 模板已经非常成熟，核心方法论（Wave 0/1/2、状态文件、Suspended Branch、配额机制、Readiness Check）都很扎实。

本文档提供 **9 个改进建议**，分为 3 个优先级：

- **P0+（最高优先级）**：4 个建议，直接解决"静默执行长程任务"问题
- **P0（强烈建议）**：3 个建议，提升探索-利用平衡、评估机制、失败处理
- **P1-P2（建议考虑）**：2 个建议，进一步打磨细节

**关键发现**：你的 V7 模板已经有完整的"自主执行协议（Autonomous Execution Protocol）"，但可能因为位置靠后、指令性不够强、缺少执行检查点提醒，导致 agent 在实际执行时仍然频繁停下来汇报。

---

## P0+：静默执行长程任务（最高优先级）

### 改进 1：将"自主执行协议"提前到更显眼的位置

**当前问题**：
- "自主执行协议（Autonomous Execution Protocol）"在模板中出现得比较靠后
- Agent 在读到这个协议之前，可能已经按照默认的"每步汇报"模式开始执行了

**建议修改**：
- **调整位置**：将"自主执行协议"从当前位置移到"实例化规则"之后、"Wave 0"之前
- **理由**：让 agent 在开始执行前就看到这个协议，建立"静默推进"的心智模型

**建议的章节顺序**：
```
## 适用场景
## 先填这些参数
## 实例化规则
## 自主执行协议（Autonomous Execution Protocol）⚠️ 必读  ← 移到这里
## Wave 0：共享 Ground Truth 地基
## Wave 1：按研究线深挖
...
```

---

### 改进 2：强化协议的指令性语言

**当前问题**：
- 当前协议用的是"默认模式"、"允许"、"不允许"这样的描述性语言
- 对于 agent 来说，指令性不够强，容易被其他习惯性行为覆盖

**建议修改**：
在"自主执行协议"中，将描述性语言改为强指令性语言：

**"默认模式：静默自主推进"小节**：
```markdown
### 1. 默认模式：静默自主推进（MUST）

**强制要求**：
- **MUST**：按 Wave / Step 检查点持续推进，状态写入 status 文件
- **MUST NOT**：主动停下来向用户汇报或等待确认
- **MUST NOT**：在对话中重复 status 文件已经记录的内容
- **MUST**：每完成一个有意义的步骤（一份 reference 落库、一条研究线完成、一个波次达标），更新 status 文件后立刻继续下一步
```

**"允许中断用户的条件"小节**：
```markdown
### 2. 允许中断用户的条件（ONLY，穷举，仅此四种）

**ONLY interrupt when**（只有在下面四种情况之一成立时，才允许主动中断用户）：

1. 主线阻塞且无法绕开
2. 研究方向需要根本性调整
3. 涉及高风险、不可逆或难回滚操作
4. 用户明确要求实时协同
```

**"不允许中断用户的情况"小节**：
```markdown
### 3. 不允许中断用户的情况（MUST NOT，显式列出）

**MUST NOT interrupt for**（以下情况必须自行处理，不得停下来请示或汇报）：

- Wave 之间的过渡
- 单个研究线的配额达标或完成
- 发现某个分支需要 suspend
- 证据配额全部达标
- 遇到搜索困难但可以绕开
- 发现新的高价值方向
- 某条研究线的停止条件已满足
- 某个新对象已经明确应升格为独立 topic
- Readiness Check 某项未通过
```

### 改进 3：在每个 Wave 说明中增加"执行模式提醒"

**当前问题**：
- Agent 在每个关键节点（Wave 0 完成、Wave 1 开始）可能会习惯性地停下来
- 模板中没有在这些节点显式提醒"不要停下来"

**建议修改**：
在每个 Wave 的说明末尾，增加一个"执行模式提醒"小节：

**Wave 0 末尾增加**：
```markdown
### 执行模式提醒

- Wave 0 完成后，更新 `.status.md` 中的 Wave 0 状态字段
- 直接推进到 Wave 1，不需要停下来向用户汇报
- 如果 Wave 0 过程中发现需要调整拓扑，按 Topology Formalization Gate 处理后继续推进
```

**Wave 1 末尾增加**：
```markdown
### 执行模式提醒

- 每条研究线完成后，更新 `.status.md` 中对应的 topic 状态字段
- 所有研究线完成后，直接推进到 Wave 2，不需要停下来向用户汇报
- 如果某条研究线遇到需要 suspend 的分支，按 Suspended Branch Protocol 登记后继续推进
```

**Wave 2 末尾增加**：
```markdown
### 执行模式提醒

- Wave 2 完成后，更新 `.status.md` 中的 Wave 2 状态字段
- 直接推进到 Readiness Check，不需要停下来向用户汇报
- 如果 Wave 2 过程中发现需要回补 Wave 1 的证据，补完后继续推进
```

### 改进 4：在 Readiness Check 开头增加"自主判断"说明

**当前问题**：
- Readiness Check 的描述可能让 agent 认为"需要停下来让用户确认是否通过"
- 实际上，Readiness Check 应该是 agent 自行判断的质量门

**建议修改**：
在 Readiness Check 小节开头，增加一段"执行模式说明"：

```markdown
## Readiness Check

**目的**：判断是否已经可交付 / 可停止 / 可接手。

**执行模式说明**：
- Readiness Check 是 **agent 自行判断的质量门**，不是需要用户确认的里程碑
- Agent 应该逐项检查下面的维度，自行判断是否通过
- 如果某项未通过，agent 应该自行判断是继续补充证据、还是按 Suspended Branch Protocol 挂起
- **只有在所有维度都通过后**，才应该停下来向用户汇报"本轮研究已完成，deliverable ready"
- 如果某项未通过且无法通过自主努力解决（例如需要内部文档、需要人工判断的争议），才应该按"主线阻塞"条件中断用户

### 检查维度

（现有的 5 个检查维度保持不变）
```

## P0：探索-利用平衡、评估机制、失败处理（强烈建议）

### 改进 5：增加 Exploration-Exploitation Decision Framework

**当前问题**：
- 模板强调"配额"和"停止条件"，但缺少对"何时应该大胆探索新方向"vs"何时应该深挖已知线索"的显式指导
- Topology Formalization Gate 有程序性描述，但缺少判断启发式

**建议修改**：
在"停止条件"之前，增加一个独立小节"Exploration-Exploitation Decision Framework"：

```markdown
## Exploration-Exploitation Decision Framework

目的：在"探索新方向"和"深挖已知线索"之间做出明智的权衡。

### 探索信号（Exploration Signals）

当出现以下信号时，应该考虑探索新方向：

1. **高频未归类概念**：当前 topic 内出现高频但无法归类到现有对象清单的新概念（连续 3+ 份 reference 提到）
2. **未建模维度**：多个 reference 指向同一个未建模的维度
3. **知识缺口**：停止条件达成但仍有明显知识缺口（例如配额满了，但核心问题仍未回答）
4. **横向依赖**：发现某个方向会显著影响其他 topic 的结论或推荐

### 利用信号（Exploitation Signals）

当出现以下信号时，应该继续深挖已知线索：

1. **证据收敛**：新增材料 80%+ 重复已知事实
2. **问题清单收敛**：question list 中的问题逐步被回答，新问题产生速度明显下降
3. **反例搜索饱和**：专门搜索"限制 / 争议 / 失败模式"，连续 2 轮未发现新失败模式
4. **机制稳定**：核心机制已经能用 3 句话解释清楚，且多个 primary source 支撑

### 决策阈值

- **继续深挖当前线**：利用信号 < 2 个，探索信号 < 1 个
- **Suspend 当前线**：利用信号 >= 3 个，且探索信号 >= 1 个（说明当前线已经饱和，但有新方向值得追）
- **Fork 新 topic**：探索信号 >= 3 个，且满足 Topic Promotion Criteria（见下）
- **Archive 当前分支**：利用信号 >= 3 个，探索信号 = 0 个，且继续下钻主要指向非公开材料

### Topic Promotion Criteria（topic 晋升标准）

当一个方向满足以下 4 个标准中的 3 个时，应该考虑将其 formalize 为独立 topic：

1. **独立对象清单**：这个方向是否有 3+ 个独立的核心对象（工具/框架/模式）？
2. **独立问题簇**：这个方向是否有 5+ 个无法被现有 topic 回答的问题？
3. **独立工件需求**：这个方向是否需要独立的 evidence summary / question list / failure taxonomy？
4. **横向依赖**：这个方向是否会显著影响其他 topic 的结论或推荐？
```

### 改进 6：强化 Readiness Check 的可操作性

**当前问题**：
- Readiness Check 有 5 个维度，但每个维度的 pass/fail 判断标准仍然偏主观
- 例如"每线'机制+趋势+难点'检查"，什么叫"机制已固定"？

**建议修改**：
为每个 Readiness Check 维度增加可操作的验收清单：

**30 秒回指检查**：
```markdown
### 30 秒回指检查

**验收标准**：
- [ ] 任意一条核心判断，能否在 30 秒内回指到本地 reference 文档？
- [ ] 本地 reference 文档是否硬核内容自给自足，读者不回 URL 就能用本地 md 支撑推理？
- [ ] 随机抽取 3 条核心判断，实际测试回指时间是否 < 30 秒
```

**每线"机制 + 趋势 + 难点"检查**：
```markdown
### 每线"机制 + 趋势 + 难点"检查

**验收标准**（每条研究线都必须满足）：

**机制检查**：
- [ ] 能否用 3 句话解释核心工作原理？
- [ ] 能否画出关键依赖图（或用文字描述清楚依赖关系）？
- [ ] 能否指出 2 个常见误解？
- [ ] 是否有 3+ 个 primary source 支撑核心机制？

**趋势检查**：
- [ ] 能否指出最近 12 个月的 3 个关键变化？
- [ ] 能否预测下一个可能的演进方向（基于已有证据）？
- [ ] 是否有 1+ 个 recent source（近期）支撑趋势判断？

**难点检查**：
- [ ] 能否列出 3 个已知失败模式？
- [ ] 能否指出每个难点的根本约束（技术/组织/生态）？
- [ ] 是否有 1+ 个 limitation source 支撑难点判断？
```

**横向综合检查**：
```markdown
### 横向综合检查

**验收标准**：
- [ ] 能否用一张表比较所有 topic 的核心对象？（或在 Wave 2 artifacts 中已经有这样的表）
- [ ] 能否指出 topic 间的 2+ 个关键依赖或冲突？
- [ ] 是否有跨 topic 的综合 artifact（例如 cross-topic synthesis、comparison table、final recommendation）？
```

**拓扑稳定性检查**：
```markdown
### 拓扑稳定性检查

**验收标准**：
- [ ] 当前 topic 数量是否已经稳定（没有明显的"应该独立但还没独立"的方向）？
- [ ] 所有 topic 的边界是否清晰（没有明显的职责重叠或缺口）？
- [ ] Topic registry 是否与实际研究内容同步？
```

**接手可继续性检查**：
```markdown
### 接手可继续性检查

**验收标准**：
- [ ] 一个不熟悉项目的人，能否在 30 分钟内通过 README、status、registry 理解当前进度？
- [ ] 所有 suspended branch 是否都有清晰的"为什么挂起"和"什么条件下重开"？
- [ ] Worklog 是否记录了关键决策的 rationale（不只是"做了什么"，还有"为什么这样做"）？
```

### 改进 7：增加 Failed Explorations 记录机制

**当前问题**：
- Suspended Branch Protocol 很好地处理了"暂时无法解决的问题"，但缺少对"已经尝试但失败的探索路径"的记录机制
- 这导致后续轮次或其他研究者可能重复无效探索

**建议修改**：
在 `.status.md` 模板中，增加一个"Failed Explorations"小节：

```markdown
## Failed Explorations

记录已经尝试但失败的探索路径，避免重复无效探索。

- exploration: <尝试过的搜索方向或假设>
- why_tried: <为什么当时认为这个方向有价值>
- what_found: <实际搜到了什么>
- why_failed: <为什么这个方向没有产出有效 reference>
- lesson: <这次失败教会了我们什么>
```

**说明**：
- 这不是为了"记录失败"，而是为了**避免重复无效探索**，并且**从失败中提炼出对问题空间的更深理解**
- 只记录"有代表性的失败"（例如某个看似合理的搜索策略连续 3 轮无效产出），不需要记录每一次小的搜索尝试

## P1-P2：进一步打磨细节（建议考虑）

### 改进 8：增加 Foundation Sufficiency Check

**当前问题**：
- Wave 0 有配额，但没有显式检查"这些文档是否真的为后续 Wave 1 提供了足够的共同语言"
- 可能出现"Wave 0 配额达标，但进入 Wave 1 后发现地基不足"的情况

**建议修改**：
在 Wave 0 和 Wave 1 之间，增加一个"Foundation Sufficiency Check"：

```markdown
## Foundation Sufficiency Check（Wave 0 → Wave 1 过渡检查）

在进入 Wave 1 之前，检查 Wave 0 的地基是否充分：

**验收标准**：
- [ ] 所有 topic 的核心术语是否都能在 Wave 0 文档中找到定义？
- [ ] 所有 topic 的对象分类是否都能映射到 Wave 0 的共同层？
- [ ] 如果现在开始 Wave 1，每条研究线是否都有明确的"从哪里开始深挖"的锚点？

**如果任何一条答案是"否"**：
- 应该补充 Wave 0 文档，而不是进入 Wave 1 后再回头补
- 更新 `.status.md` 中的 Wave 0 状态，说明补充了哪些内容
```

### 改进 9：增加 Early Saturation Protocol

**当前问题**：
- 配额机制（8 份文档、4 份 primary source）是防止浅搜的下限
- 但模板中也提到"权威信源优先规则：宁缺毋滥"
- 这两个原则可能冲突：如果已经有 3 份高质量 primary source 足以支撑判断，是否还需要凑满 4 份？

**建议修改**：
在"停止条件"小节中，增加"Early Saturation Protocol"（提前饱和协议）：

```markdown
### Early Saturation Protocol（提前饱和协议）

**适用场景**：
- 某个维度（如 primary source）已经达到"证据收敛"状态（新增材料 90%+ 重复已知事实）
- 继续搜索只会引入低质量来源，而不会增加新信息

**处理方式**：
- 可以提前停止该维度的搜索，不必凑满配额
- 但必须在 `.status.md` 中显式记录：
  ```
  - `primary_count`: `3` (early saturation: 新增材料已连续 3 轮重复核心机制，判定为证据收敛)
  ```
- 这样既保留了"宁缺毋滥"的灵活性，又避免了"随意降低标准"的风险

**判断标准**：
- 连续 3 轮搜索，新增材料 90%+ 重复已知事实
- 已有的 reference 来自权威一手来源（官方文档、核心论文、主要维护者的说明）
- 核心机制已经能用 3 句话解释清楚，且有多个独立来源支撑
```

## 实施建议

### 优先级排序

**立即实施（P0+）**：
1. 改进 1：将"自主执行协议"提前
2. 改进 2：强化协议的指令性语言
3. 改进 3：在每个 Wave 增加"执行模式提醒"
4. 改进 4：在 Readiness Check 增加"自主判断"说明

**强烈建议（P0）**：
5. 改进 5：增加 Exploration-Exploitation Decision Framework
6. 改进 6：强化 Readiness Check 的可操作性
7. 改进 7：增加 Failed Explorations 记录机制

**建议考虑（P1-P2）**：
8. 改进 8：增加 Foundation Sufficiency Check
9. 改进 9：增加 Early Saturation Protocol

### 验证方式

**方式 1：用新任务测试**
- 用 V8 执行一个新的 Deep Research 任务
- 观察 agent 是否还会频繁停下来汇报
- 记录 agent 在哪些节点停下来、为什么停下来

**方式 2：回测现有案例**
- 用 V8 的标准重新审视已完成的案例
- 检查哪些"停下来汇报"是必要的、哪些是可以避免的
- 检查是否有"Failed Explorations"应该被记录但未记录

**方式 3：接手者测试**
- 让一个不熟悉项目的人读 V8 模板和一个 V8 实例
- 测试他们是否能在 30 分钟内理解"为什么这样设计"、"如何继续推进"

### 风险与权衡

**风险**：
- 增加的机制可能让模板变得更复杂，降低执行效率
- Failed Explorations 如果记录过细，可能变成负担

**缓解**：
- 所有新增机制都是"可选增强"，不是"强制约束"
- 在 V8 说明中明确：这些机制是"当你遇到对应问题时的工具"，不是"每次都必须用的清单"
- 只记录"有代表性的失败"，不记录每一次小的搜索尝试

---

## 总结

你的 V7 模板已经非常成熟，核心方法论都很扎实。

这 9 个改进建议主要集中在：
1. **静默执行**（P0+）：让 agent 真正做到"默认自主推进，只在关键时刻中断"
2. **探索-利用平衡**（P0）：显式化"何时探索新方向"vs"何时深挖已知线索"的判断
3. **评估机制**（P0）：让 Readiness Check 从"主观判断"变成"可操作清单"
4. **失败处理**（P0）：记录失败的探索路径，避免重复无效探索

这些改进都不会破坏现有结构，只是让模板在"执行模式"、"决策启发"、"质量门"、"失败学习"这四个维度上更加显式和可操作。
