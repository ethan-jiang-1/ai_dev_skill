# get-shit-done 总览

## 这是什么

`get-shit-done` 不是单纯的 prompt 包，也不是单纯的 skill 仓库。  
它更像一套给 AI coding runtime 安装的 **skill runtime / workflow operating system**。

它的核心卖点不是“单个 skill 很厉害”，  
而是把 skill-like 单元做成一整套可分发、可调用、可串联、可落状态、可验证的系统。

## 这次分析最关心什么

这次不是泛泛分析整个仓库，  
重点只看仓库里 **类似 skill 的东西**。

也就是：

- 带 front matter 的 command
- 带 front matter 的 agent
- 负责编排的 workflow
- 提供规则的 reference
- 提供工件骨架的 template

## 我对它的第一判断

如果拿它和 `agent-skills` 对比：

- `agent-skills` 更像 skill library
- `get-shit-done` 更像 skill runtime

前者侧重把“一个 skill 应该怎么做”讲清楚。  
后者侧重把“这些 skill-like 单元怎么被安装、调度、串成工程流水线”做成产品。

## 这套系统最关键的 skill-like 结构

### 1. commands

`commands/gsd/*.md` 是能力入口层。

特征：

- 有 front matter
- 定义 `name`
- 定义 `description`
- 定义 `allowed-tools`
- 再通过 `execution_context` 跳到 workflow

所以 command 本质上是在声明：

- 这个能力叫什么
- 它允许动什么工具
- 它的执行上下文在哪里

### 2. agents

`agents/*.md` 是专职角色层。

特征：

- 有 front matter
- 定义 `name`
- 定义 `description`
- 定义 `tools`
- 定义 `color`

这些 agent 不是简单的人设，而是强约束的 skill persona。

### 3. workflows

`get-shit-done/workflows/*.md` 是编排层。

它们虽然常常没有 YAML front matter，  
但结构极强：

- `purpose`
- `required_reading`
- `process`
- `step`
- 上下文加载
- agent 调度
- 状态推进

所以 workflow 是 skill 的“路由器 + 调度器”。

### 4. references 与 templates

这两层决定系统为什么不只是 prompt 拼接。

- `references` 负责稳定规则
- `templates` 负责稳定产物

这让 skill 不只是说法一致，连产物形状也被固定下来。

## 它真正解决的问题

它想解决的是：

- 上下文腐化
- agent 一路写下去后质量退化
- 没有项目记忆
- 不会持续验证
- 规划与执行脱节

所以它把工作过程拆成：

- 初始化
- 讨论
- 规划
- 执行
- 验证
- ship

每一步都对应到一组 skill-like 单元。

## 为什么这项目值得分析

这个仓库最有研究价值的，不是营销语言，  
而是它证明了一件事：

- skill 可以不长得像 `SKILL.md`
- skill 也可以长成 command、agent、workflow、reference、template 的组合系统

这正是它和传统 skill 仓库最不一样的地方。
