# get-shit-done 核心机制

## 核心机制不是“某个命令”

这个仓库真正的核心机制，是一条由 skill-like 单元串起来的闭环：

- command 接用户输入
- workflow 做流程编排
- agent 做专职执行
- reference 提供规则
- template 固定输出
- `.planning/` 持久化项目状态
- 验证阶段把结果再打回流程

## 一、front matter 作为能力声明层

这是我们最关心的 skill 识别点。

### command front matter

在 command 里，front matter 主要承担：

- 能力命名
- 能力描述
- 工具权限声明
- 参数提示

这让一个 Markdown 文件，不只是文档，  
而是一个可以被 runtime 识别的能力入口。

### agent front matter

在 agent 里，front matter 主要承担：

- 子 agent 命名
- 专业职责定义
- 工具边界声明
- 可视化或分组信息

这使 agent 既是 prompt，也是“角色 skill 注册项”。

## 二、execution_context 作为路由机制

command 文件本身往往不承载全部细节。  
它通过 `execution_context` 把执行跳转到 workflow、reference、template。

这点很关键，因为它避免了两个问题：

- 入口 prompt 过胖
- 规则在多个入口里重复复制

所以 command 更像 controller，  
workflow 才像真正的 use-case 实现。

## 三、workflow 作为 skill orchestration

workflow 的真正价值是把 skill 变成系统，而不是变成清单。

它做的事包括：

- 读取上下文
- 判定模式
- 组织问题收集
- 并行分派 agent
- 汇总结果
- 写入产物
- 推进下一阶段

这一步一旦存在，仓库就不再是“提示模板集合”，  
而是“有生命周期的执行系统”。

## 四、项目记忆机制

`get-shit-done` 的一个明显优势，是把项目记忆显式写入 `.planning/`。

核心工件包括：

- `PROJECT.md`
- `REQUIREMENTS.md`
- `ROADMAP.md`
- `STATE.md`
- `VALIDATION.md`

这让 skill-like 系统摆脱了一次性 prompt 的局限。

具体好处：

- 跨会话可延续
- 不同 agent 能共享状态
- 后续验证有依据
- 计划变更可追踪

## 五、验证闭环

这个仓库没有停在“会规划”。  
它还强调：

- verify work
- checker feedback
- gap closure
- revision mode
- ship gating

这说明它的目标不是“产出文本”，  
而是尽量把 agent 行为收束到可验收的工程流程里。

## 六、薄入口、厚中台

从设计取向看，它采用的是：

- 薄 command
- 厚 workflow
- 专职 agent
- 共享 reference
- 统一 template

这是个很成熟的 skill-system 设计。

因为它让：

- 新增入口更容易
- 新增角色更容易
- 修改规则影响面更可控
- 复用共享知识更容易

## 七、为什么它比普通 skill 仓库更“系统”

普通 skill 仓库常见的问题是：

- skill 很多，但互相之间关系弱
- 缺少流程编排
- 缺少持久状态
- 缺少产物骨架
- 缺少验证回路

`get-shit-done` 用下面几样东西补上了这些缺口：

- workflow
- `.planning/`
- gsd-tools.cjs
- templates
- verification-oriented commands

## 核心机制结论

如果只用一句话总结：

`get-shit-done` 的核心机制，就是把 skill 从“静态能力卡片”升级成“带状态、带编排、带验证的工程流水线单元”。

这也是它最值得我们研究的地方。
