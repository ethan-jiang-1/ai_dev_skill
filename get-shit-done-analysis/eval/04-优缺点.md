# get-shit-done 优缺点

## 优点

### 1. skill-like 单元非常清晰

这个仓库最强的一点是：  
你确实“一看 front matter 就知道它在干嘛”。

尤其是 command 和 agent：

- 能力名称清晰
- 描述清晰
- 工具边界清晰
- 调用方向清晰

这让仓库的可读性和可拆解性很高。

### 2. skill 不只是一层，而是完整体系

它不是把一堆 skill 平铺出来，  
而是做成了多层系统：

- command
- agent
- workflow
- reference
- template

这比单层 skill catalog 更接近可落地工程系统。

### 3. 强状态管理

`.planning/` 体系让整个流程有记忆。  
这是很多 skill 仓库没有真正做好的地方。

### 4. 强验证意识

它不是只强调“做”，  
还强调“验证、修正、补缺、再推进”。

这使它更适合中长任务，而不是只适合一次性 prompt。

### 5. 可迁移的方法论很多

即便不直接使用整个仓库，  
也能抽出很多可复用设计：

- front matter 做能力声明
- workflow 做流程编排
- reference 做共享规则
- template 做共享产物
- state 文件做项目记忆

## 缺点

### 1. 对初学者来说偏重

如果用户只想拿几个 skill 直接用，  
这套系统会显得太厚。

### 2. skill 边界有时不是显式统一的

command 和 agent 很好识别，  
但 workflow、reference、template 要靠结构判断才能认出来。

也就是说：

- 它们本质上是 skill-like
- 但仓库没有用统一术语显式标注这一点

这对第一次分析的人有理解门槛。

### 3. Markdown 驱动系统规模很大

随着 command、workflow、reference、template 增长，  
维护成本会上升：

- 命名一致性
- 交叉引用一致性
- 规则更新一致性

都需要较强治理。

### 4. 运行时耦合仍然存在

虽然它努力兼容多 runtime，  
但很多细节仍然围绕特定 agent 工具生态来设计。

这意味着完全迁移到别的宿主时，  
仍然需要适配成本。

### 5. 文档量大于代码量时，理解成本会上升

它的优势在于结构化文档。  
但这也导致：

- 入口很多
- 规则很多
- 需要花时间建立全局地图

## 最大优点

最大优点不是“命令多”，  
而是它把 skill 做成了一个有编排、有状态、有验证的工程系统。

## 最大缺点

最大缺点不是“Markdown 太多”，  
而是 skill-like 边界虽然客观存在，但概念层没有完全统一命名，  
需要分析者自己把 command / agent / workflow / reference / template 收敛成同一框架来理解。

## 总体判断

如果你的目标是：

- 学几个 prompt 技巧

那它可能偏重。

如果你的目标是：

- 学怎么把 skill 组织成一套持续可执行的工程系统

那它非常值得研究。
