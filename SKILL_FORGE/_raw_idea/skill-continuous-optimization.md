# 面向 Coding Agent 的 Skill Continuous Optimization

## 1. 这条 idea 想补什么缺口

当前关于 skill engineering 的讨论，已经开始从“怎么写一个 skill”转向“怎么把 skill 工程化地组织、审计、发布、分发”。

但这里仍然缺了一块很重要的中后期能力：

**一个 skill 在已经可用之后，如何被持续精调。**

这件事如果只叫 prompt optimization，会把问题说小。

因为一个真实的 skill，真正影响效果的部分，通常不只是 instruction 文本，还包括：

- skill 的命名和描述
- 触发词和触发条件
- 元数据和目录结构
- step-by-step workflow 的拆法
- 工具调用的边界约束
- 输入输出契约
- 示例和反例
- 评测样本和失败样本
- 发布后的反馈回流机制

也就是说，这里要研究的不是：

“怎么把 prompt 改得更像 prompt engineer 在调词。”

而是：

“怎么把 skill 当作一个完整 artifact，通过观测、评测、回放、对比和迭代，持续变得更稳定、更易触发、更可执行、更可治理。”

## 2. 为什么它值得成为独立 idea

原因不是“这个方向也挺有意思”，而是它正好补在现有 skill engineering 研究版图的断层上。

目前常见研究更容易覆盖下面三段：

- 定义和结构规范
- 工具链、审计、发布、分发
- 生态采用与可信度判断

但在真实使用里，skill 的生命周期并不会在“发布成功”那里结束。

很多 skill 的主要问题恰恰发生在发布之后：

- agent 没有在该触发的时候触发
- agent 触发了，但没有按步骤执行
- 描述写得很全，但 discoverability 很差
- workflow 太长，模型只吸收成背景知识，没有照着做
- 工具调用边界不清，输出结构不稳定
- 初版在 demo 里好用，但一遇到真实任务分布就失效
- 每次修改 skill 都只能靠主观感觉，缺少回归验证

如果没有一套“持续精调 skill”的视角，skill engineering 就会只覆盖：

- 前期建模
- 中期工程交付

却缺少：

- 后期行为优化
- 版本迭代闭环
- 失败样本驱动的持续改进机制

所以这条线不应该只是现有 topic 的一段补充，而更适合作为一个独立 raw idea 先沉淀出来。

## 3. 这里说的“调优对象”到底是什么

为了避免话题又滑回 prompt tuning，这里先把调优对象定清楚。

Skill continuous optimization 的对象，至少包括下面八类。

### 3.1 触发与可发现性

- 名称是否可被正确联想
- 描述是否覆盖真实触发入口
- 关键词是否和用户任务语言分布一致
- skill 何时该触发、何时不该触发是否足够清楚

这一类问题的核心不是“内容对不对”，而是：

**agent 在对的时候能不能找到它，在错的时候会不会误触发它。**

### 3.2 指令主体与行为约束

- 主 instruction 是否足够清晰
- 约束是否过松或过紧
- 目标是否明确
- 禁止事项是否会被模型忽略
- tool-use policy 是否清楚

这部分和 prompt 最接近，但它只是 skill 的一部分。

### 3.3 结构与分块方式

- skill 是不是拆得过长
- 是否应该把背景说明、执行步骤、边界条件分块
- workflow 是不是应该改成编号步骤
- 长说明是否压低了可执行性

很多 skill 的问题不是信息缺失，而是结构让 agent 不容易执行。

### 3.4 Workflow 的可执行性

- agent 会不会真正按步骤走
- 会不会跳步
- 会不会只给结论，不执行过程
- 多阶段任务是否需要显式检查点
- 是否需要中间产物要求

这部分和“技能是否被真正遵守”直接相关。

### 3.5 工具使用契约

- 什么时候必须调用工具
- 什么时候禁止调用工具
- 工具结果如何影响后续步骤
- schema / format / output contract 是否明确

这一类问题如果不调优，skill 往往表面上“会做”，但执行结果并不稳定。

### 3.6 示例、反例与边界案例

- 是否提供了足够贴近真实任务的例子
- 是否提供“不该这样做”的反例
- 是否覆盖常见失败场景
- 是否帮助 agent 理解边界而不是只理解理想路径

### 3.7 评测集与失败样本集

- skill 有没有最小 eval set
- 是否有稳定的 regression cases
- 失败样本是否能被归类
- 版本前后能否客观比较

如果没有这一层，所谓优化基本都只是主观修改。

### 3.8 反馈回流与版本治理

- 线上失败能否回流
- 用户修正能否沉淀成样本
- 每次修改能否记录原因
- 是否知道这次改动到底修好了什么，又破坏了什么

这一层让 skill 从“改文档”变成“管理一个会演化的工程资产”。

## 4. 这条线真正要回答的问题

如果把这个 idea 做成后续研究线，它至少要回答下面几个问题。

### 4.1 Skill 的最小优化闭环是什么

一个实用的最小闭环应该不是“想到哪里改哪里”，而应该至少包含：

1. 观测到失败或偏差
2. 对失败进行分类
3. 判断问题落在哪一层
4. 形成修订假设
5. 对 skill 做局部修改
6. 用 eval / replay / regression 重新验证
7. 判断是否提升，是否引入副作用

### 4.2 哪些失败类型最值得优先优化

不是所有问题都值得先调。

优先级更高的通常是：

- 高误触发或高漏触发
- 明显不按 workflow 执行
- 工具调用错误代价高
- 结构导致大面积偏航
- 在真实使用中高频出现的失败模式

### 4.3 如何把“问题归因”从拍脑袋变成可复查

skill 失败的根因可能来自很多层：

- 触发元数据问题
- instruction 问题
- 结构编排问题
- 示例覆盖问题
- 工具约束问题
- 评测样本本身不对

如果不能稳定归因，优化就会一直停留在凭感觉反复改文本。

### 4.4 自动化到底能做到哪一步

这里要研究的不是“能不能全自动改 skill”，而是：

**哪些环节自动化最有价值，哪些环节必须人工把关。**

### 4.5 如何判断一个 skill 真的变好了

判断标准不能只是“作者更满意”。

至少应该考虑：

- 触发更准了没有
- 执行更稳定了没有
- 工具误用减少了没有
- 回归样本通过率提高了没有
- 是否引入新的副作用

## 5. 可以借鉴的外部模式，不应只搜 prompt tuning

这条线确实和 prompt optimization 有交集，但如果只沿着这个词搜，很容易把结论收窄。

更有价值的是把外部模式拆成四类来看。

### 5.1 Eval-driven optimization

这类方法强调：

- 先有测试任务和评测标准
- 再根据评测结果修改 prompt / policy / workflow
- 修改后重新运行验证

它的价值不在“自动写得更漂亮”，而在“修改是被评测驱动的”。

这套思路可以直接迁移到 skill：

- 先建立 skill eval set
- 再基于失败类型改 skill artifact
- 再做回归比较

### 5.2 Dataset + Trace + Replay

这类方法强调：

- 收集真实调用轨迹
- 识别失败样本
- 回放关键 case
- 比较版本前后的行为差异

它特别适合解决 skill 发布后才暴露的问题。

### 5.3 Optimizer / Search 式自动改进

这类方法更进一步，会尝试：

- 自动生成多个候选版本
- 基于指标进行筛选
- 在约束下搜索更优版本

但这里要很小心：

它不一定应该直接改完整 skill；

更合理的做法可能是让它先作用在局部部件上，例如：

- 描述改写
- step 结构重写
- 示例补齐
- 工具边界文字修正

### 5.4 Human-in-the-loop refinement

这类方法强调：

- 自动化系统负责发现问题、提出候选修订
- 人工负责设目标、做最终判断、处理高风险边界

对于 skill 来说，这往往比“全自动自我优化”更现实。

## 6. autoresearch 思路可以怎么衍生到 skill 调优

如果把 autoresearch 抽象掉名字，它的核心其实不是“自己搜索资料”。

它更底层的思想是：

- 先建立目标
- 再通过循环不断发现不足
- 生成新的探索或修订动作
- 根据外部反馈更新当前对象

这套模式可以迁移到 skill continuous optimization，但要把研究对象从“报告内容”改成“skill artifact”。

一个可能的映射是：

### 6.1 Observe

- 收集失败对话
- 收集误触发 / 漏触发样本
- 收集工具误用样本
- 收集用户修正记录

### 6.2 Classify

- 按失败类型聚类
- 区分触发问题、执行问题、结构问题、工具问题、样本问题

### 6.3 Localize

- 判断根因落在哪个 artifact 部位
- 决定是改 metadata、改描述、改步骤、改约束，还是补案例

### 6.4 Propose

- 生成一个或多个局部修订版本
- 显式写出每个版本试图修复的失败类型

### 6.5 Verify

- 跑回归样本
- 跑新失败样本
- 对比版本前后结果

### 6.6 Promote or Reject

- 通过则升级为新版本
- 不通过则回退候选，不污染主 skill

这比“让模型自己把 prompt 改一改再试试”要更接近工程闭环。

## 7. 这个研究方向最有价值的四个子题

如果后面真要把它扩成 topic，我认为最值得深挖的是下面四条。

### 7.1 Trigger / Discoverability Tuning

研究：

- skill 为什么漏触发
- skill 为什么误触发
- 描述、关键词、命名、例子如何影响匹配命中

### 7.2 Workflow / Executability Tuning

研究：

- skill 为什么“看过但不照做”
- 怎样的步骤结构更容易被 agent 执行
- 是否需要检查点、中间产物、显式停止条件

### 7.3 Tool Contract Tuning

研究：

- 工具调用的说明怎么写更稳
- 什么时候应把工具使用要求变成强约束
- schema / output contract 如何减少执行漂移

### 7.4 Feedback / Eval Loop Automation

研究：

- 怎么建立最小 eval set
- 怎么沉淀失败样本
- 怎样做版本前后回归
- 自动化在这个闭环里该占多大比例

## 8. 可能的研究边界与风险

这条线虽然重要，但也很容易做歪。

### 8.1 风险一：又退化回 prompt tuning

只要一不小心，研究就会滑回“怎么把指令写漂亮一点”。

所以必须反复强调：

**skill optimization 的对象是完整 artifact，而不是单段 prompt。**

### 8.2 风险二：过度追求自动化

如果太快追求 self-improving skill，很容易忽略：

- eval 本身可能不完整
- 自动修改可能引入隐性副作用
- 高风险 skill 不适合完全自动放行

### 8.3 风险三：没有 failure taxonomy

如果失败样本没有分类，后续所有优化都容易混成一团，最后只剩“继续改文案”。

### 8.4 风险四：没有版本对比与回归

如果每次改完都没有前后对照，就无法判断优化是真提升还是只是换了一种失败方式。

## 9. 这份 raw idea 未来怎么接到主研究里

短期内，它适合先留在 `_raw_idea`，作为一个明确的新方向。

后续如果并入主研究，最自然的方式不是把它塞进现有 topic 的某个小节，而是单列成新的研究线，例如：

- `skill-optimization-and-feedback-loops`

它将和现有三条线形成互补：

- `skill-methodology-and-spec`
  负责定义和结构边界
- `skill-toolchain-and-lifecycle`
  负责工程链路和工具角色
- `ecosystem-signals-and-adoption`
  负责可信度和采用判断
- `skill-optimization-and-feedback-loops`
  负责发布后的持续精调和版本闭环

这样整个版图才完整：

- 有定义
- 有工程链路
- 有生态判断
- 也有持续优化机制

## 10. 一句话总结

这条 idea 的核心不是：

“怎么让模型继续润色 prompt。”

而是：

**怎么把 skill 当成一个可观测、可评测、可回放、可迭代的工程资产，在上线之后继续稳定变好。**
