# 附录 C：Cross-Surface 兼容边界与 Portable Core

这份附录处理的是一个很容易被说得太抽象、也最容易被误会的问题：

> 如果同一个 skill 要在不同 surface（宿主表面）里使用，到底哪些部分算共通，哪些部分根本不能想当然地当成“统一标准”？

这件事如果不单独讲清，实践者最容易掉进两个坑：

- 以为三家平台都差不多，所以可以一把梭
- 或者反过来，觉得差异太多，所以根本没有可迁移写法

当前证据支持的，其实是中间路线：  
有共同层，但共同层比很多人想的要小；也有平台扩展层，但扩展层不该伪装成通用标准。[E04][E09]

## C1. 先讲最核心的判断

当前最稳的 authoring baseline（写作基线）不是：

- 把所有平台都支持的字段凑成一个“超级标准”

而是：

- 先写 portable core（可移植核心）
- 再把平台特有能力写成 surface appendix（宿主附加层）

`portable core（可移植核心）` 指的是：  
不严重依赖某一个平台，换个宿主仍然大体能沿用的核心部分。

对大多数 skill 来说，这通常包括：

- `SKILL.md`
- `name`
- `description`
- 核心步骤
- 指向支撑文件的导航

这套理解会比“试图把所有平台特性都揉成一坨”稳定得多。[E04][E09]

## C2. 三家 surface 目前最稳的共同层

当前在 Codex、GitHub、Claude 之间，最稳的共同层大致是：

| 共同层 | 为什么稳 |
| --- | --- |
| 目录级 skill 对象 | skill 不再只是单段 prompt，而是目录级能力包 |
| `SKILL.md` | 入口文件已经非常稳定 |
| `name` | 作为识别和显示的核心字段 |
| `description` | 接近路由字段，决定什么时候该考虑调用 |
| progressive disclosure（渐进展开） | 主文放主线，细节下沉到支撑文件 |
| supporting files on demand（按需支撑文件） | 长说明、示例、脚本不必全塞正文 |

如果你的目标是跨平台尽量稳，这一层最值得优先写好。[E04][E09][E13]

## C3. 当前最不该默认视为通用标准的层

很多问题都出在把平台特有能力误读成通用能力。

当前最不稳、最不该默认视为 portable baseline（可移植基线）的，包括：

- `allowed-tools`
- 平台特有的 invocation model（调用方式）
- plugin packaging（插件打包）
- `agents/openai.yaml`
- runtime / container assumptions（运行时或容器假设）
- Claude richer frontmatter（Claude 更丰富的前置信息）

这些东西不是不能用，而是：

- 可以用
- 但应该被写成扩展层
- 不能写成“skill 标准就是这样”

这是跨 surface authoring（跨宿主写作）最重要的边界之一。[E09]

## C4. 三家 surface 的简化对照

下面这张表不是完整支持矩阵，只是帮助你建立正确的心智模型。

| Surface | 最像的共同层写法 | 更明显的扩展层 | 最容易踩的坑 |
| --- | --- | --- | --- |
| `Codex` | 目录 + `SKILL.md` + 支撑文件 | `agents/openai.yaml`、plugins、`.agents/skills` 体系 | 把 Codex 的 distribution 语义误写成全平台默认 |
| `GitHub` | 目录 + `SKILL.md` + examples / resources | `allowed-tools`、GitHub 特定路径语义 | 把 GitHub 的路径或字段外推成所有平台标准 |
| `Claude` | 目录 + `SKILL.md` + supporting files | 更丰富的 frontmatter、hooks、agent、shell 等 | 把 Claude 的强扩展能力误当成“skill 本来就该这么写” |

如果你只记一个判断，就记这个：

> 功能最丰富的平台，往往也是最容易让作者误把扩展层当成共通层的平台。[E09]

## C5. 一个最稳的写作顺序

如果你打算写一个以后可能会迁移的 skill，最稳的顺序是：

### Step 1：先写 portable core

先只写：

- skill 是干什么的
- 什么时候该用
- 主线步骤是什么
- 细节去哪里看

### Step 2：再标出你依赖了哪些平台特性

例如：

- 某个字段只在 GitHub 有明确语义
- 某个调用方式只在 Claude 里更自然
- 某个打包方式明显偏 Codex

### Step 3：把这些平台特性写成兼容性说明

最简单的做法就是写一节：

- 本 skill 的 portable core 是什么
- 对 Codex 的扩展是什么
- 对 GitHub 的扩展是什么
- 对 Claude 的扩展是什么

这样后续维护时，你至少知道哪里是共通层，哪里是宿主锁定层。[E07][E09]

## C6. 什么情况下应该单独写 compatibility note

只要出现下面任意一种情况，就建议单独写兼容性说明：

1. 你开始使用平台特有 frontmatter。
2. 你开始依赖平台特有的调用方式。
3. 你开始接插件、运行时桥接或容器假设。
4. 你知道这个 skill 会在两个以上宿主里被尝试。
5. 你希望未来别人维护时，不需要重新猜“这部分到底是不是通用写法”。

兼容性说明不一定很长，但它能极大降低后续误读成本。

## C7. 跨 surface 时最容易犯的 4 个错

### 1. 一开始就把平台特性写满

这样最容易得到一个在单一平台里看起来很强、但稍微迁移就很乱的 skill。

### 2. 看见共同层，就误以为运行语义也一样

三家都支持 `SKILL.md`，不等于三家的加载、调用、扩展和发布语义都一样。

### 3. 因为怕差异，就放弃写可移植核心

差异存在，不等于没有共同层。  
真正稳的做法不是放弃共同层，而是把共同层写小、写稳。

### 4. 只贴字段表，不解释边界

字段表可以查，但它不能直接替代理解。  
真正有用的是告诉读者：

- 这字段为什么重要
- 它是不是 portable
- 如果不是，应该怎么处理

## C8. 如果你现在就要起草一个跨平台 skill

最推荐的最小起草方式是：

```md
1. 先写清 task、触发条件和主线步骤
2. 把长内容下沉到 supporting files
3. 给平台特有能力单独写一节 compatibility note
4. 明确哪些能力依赖单一宿主
```

只要你按这个顺序来，大多数跨平台混乱都能提前规避掉。

## C9. 读完这份附录后，下一步做什么

拿你手头一个已有 skill，检查它有没有下面三样东西：

- 哪部分是 portable core
- 哪部分依赖特定宿主
- 有没有一眼就能看懂的兼容性说明

如果三者都没有，这个 skill 以后只要一迁移，很可能就会开始混乱。
