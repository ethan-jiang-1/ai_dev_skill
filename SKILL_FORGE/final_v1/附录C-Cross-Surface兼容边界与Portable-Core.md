# 附录 C：Cross-Surface 兼容边界与 Portable Core

如果你的 skill 只在一个平台上用，这份附录可以先跳过。但如果你打算让同一个 skill 在 Codex、GitHub、Claude 等不同 surface（宿主平台）上都能用，这里讲的就是你最容易踩的坑。

最常见的两个极端：一是以为三家平台都差不多，直接一把梭全写满；二是觉得差异太大、完全没有可迁移的部分。实际情况在中间——**有共同层，但比你以为的要小；也有平台特有层，但不该伪装成通用标准。**

---

## C1. 菜谱类比：什么是 portable core

想象你在写一份烤鸡的菜谱。核心步骤——"先腌 2 小时、再 200 度烤 40 分钟"——不管你用什么牌子的烤箱都是一样的。但具体到"用哪个烤箱档位"、"要不要开热风循环"，那就取决于你的烤箱型号了。

skill 也是一样。`portable core（可移植核心）` 就是那些"换个平台也不变"的部分。对大多数 skill 来说，这包括：

- **`SKILL.md`** 作为入口文件
- **`name`** 和 **`description`** 作为身份标识和路由
- **核心步骤** —— 先做什么、后做什么
- **指向支撑文件的导航** —— 详细内容去哪里看

而"烤箱设置"——也就是平台特有的部分——包括 `allowed-tools`、`agents/openai.yaml`、Claude 更丰富的 frontmatter、特定的插件打包方式等等。这些东西可以用，但应该被写成**扩展层**，而不是混在核心里。

---

## C2. 三家平台目前的共同层

在 Codex、GitHub 和 Claude 之间，当前最稳定的共同层大致是这些：

| 共同点 | 为什么稳 |
| --- | --- |
| skill 是目录级对象 | 三家都已经从"一段 prompt"演化到了"一个目录" |
| `SKILL.md` 作为入口 | 入口文件的形式已经非常收敛 |
| `name` 和 `description` | 身份标识和路由字段是三家共享的基本需求 |
| 渐进展开 | 主文放主线、细节下沉到支撑文件，这个设计理念三家都支持 |
| 按需加载支撑文件 | 长说明、示例、脚本不必全塞正文 |

如果你的目标是跨平台尽量稳，**先把这一层写好**就够了。

---

## C3. 最不该默认视为通用标准的东西

很多跨平台问题都出在把某个平台的特有能力误当成"skill 标准就是这样"。一个值得注意的规律是：**功能最丰富的平台，往往最容易让作者把扩展层和共通层搞混。**

比如 Claude 支持很丰富的 frontmatter、hooks 和 shell 集成——这些都是强大的能力，但如果你写 skill 的时候默认"这些应该是 skill 标准配置"，那你的 skill 换到 Codex 或 GitHub 上就会出问题。

| Surface | 共同层写法 | 扩展层 | 最容易踩的坑 |
| --- | --- | --- | --- |
| Codex | 目录 + `SKILL.md` + 支撑文件 | `agents/openai.yaml`、plugins 体系 | 把 Codex 的分发语义误写成全平台默认 |
| GitHub | 目录 + `SKILL.md` + examples | `allowed-tools`、GitHub 特定路径 | 把 GitHub 的路径或字段外推成所有平台标准 |
| Claude | 目录 + `SKILL.md` + supporting files | 丰富 frontmatter、hooks、shell | 把 Claude 的强扩展误当成"skill 本来就该这么写" |

---

## C4. 最稳的写作顺序

如果你打算写一个以后可能会迁移的 skill，按这个顺序来：

**第一步：先写 portable core。** 只写四件事——这个 skill 干什么、什么时候用、主线步骤是什么、细节去哪看。不加任何平台特有的东西。

**第二步：标出你用了哪些平台特性。** 写完之后回头看一遍，把依赖特定平台的部分标出来。比如"这个 frontmatter 字段只在 Claude 有用"、"这个调用方式只在 GitHub 里有效"。

**第三步：把平台特性单独写成兼容性说明。** 在 skill 里加一节，说清楚：portable core 是什么、对 Codex 的扩展是什么、对 GitHub 的扩展是什么、对 Claude 的扩展是什么。以后维护的人一看就知道哪些是共通的、哪些是平台锁定的。

这个顺序的好处是：你永远知道哪些部分可以迁移，哪些部分需要适配。不会出现"换了个平台，不知道哪里会坏"的混乱。

---

## C5. 什么时候该写 compatibility note

只要出现下面任何一种情况，就建议单独写一节兼容性说明：你开始用平台特有的 frontmatter 了；你的调用方式依赖特定平台的机制了；你知道这个 skill 会在两个以上平台被使用；或者你希望未来别人维护的时候不需要猜"这部分到底是不是通用写法"。

兼容性说明不一定很长——哪怕只有三五行——但它能极大降低后续的误读成本。

---

## C6. 跨平台最容易犯的错

**一开始就把平台特性写满。** 结果是得到一个在单一平台里看起来很强、但换个平台就一团乱的 skill。更稳的做法是先写小、写稳的 portable core，平台特性后加。

**看见三家都支持 SKILL.md，就以为运行语义也一样。** 三家都认 `SKILL.md` 这个入口，但加载方式、调用机制、扩展能力都不同。共同层在文件格式上，不在运行语义上。

**因为怕差异，干脆放弃写可移植核心。** 差异存在，不代表没有共同层。把共同层写小、写稳，然后平台差异单独处理——这比"要么全通用、要么完全放弃"要实际得多。

---

## C7. 一个典型的跨 surface 失败案例

你写了一个代码审查 skill，在 Claude CLI 上跑得很好。`SKILL.md` 的 frontmatter 里用了 `allowed-tools` 限制只能调用 `Read` 和 `Grep`，还用了 `hooks/session-start` 在启动时自动注入审查纪律。

一个同事把这个 skill 复制到了 Codex 环境里。结果：`allowed-tools` 没生效（Codex 不认这个字段），agent 调用了不该调的工具；hooks 也没触发（Codex 没有对等的 hook 机制），审查纪律完全缺失。skill 的核心步骤其实是好的，但平台特有的约束全部失效了。

问题出在哪？你把 Claude 特有的能力（`allowed-tools`、hooks）混在了 portable core 里，没有单独标记为平台扩展。如果当初写了一节 compatibility note，说明"工具限制依赖 Claude CLI 的 `allowed-tools`，在其他平台需要用其他方式实现"，这个问题就不会悄悄发生。

---

## C8. 写作层面之外：运行时差异

这份附录聚焦的是**写作层面**的兼容策略——写 skill 时怎么区分 portable core 和平台扩展。但跨平台还有一层**运行时差异**是写作时看不到的：SDK 和 CLI 的行为不一致、API 通道的硬限制、各 surface 的 skill 发现路径不同等。

如果你的 skill 需要在多个 surface 上实际运行（不只是"写完存着"），建议读完这份附录后继续看 [附录G-进阶话题与深度参考](./附录G-进阶话题与深度参考.md) 的 G6 节，那里详细讲了运行时层面的差异和限制。

---

## C9. 读完这份附录，下一步做什么

拿你手头的一个 skill，做一件事：把它的内容分成两份——哪些是 portable core（换平台也不变的核心），哪些依赖特定平台。如果你发现分不清，那说明你在写的时候把两层混在了一起——这恰恰是最值得现在理清的。
