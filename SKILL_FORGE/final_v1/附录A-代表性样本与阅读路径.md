# 附录 A：代表性样本与阅读路径

这份附录不追求"列得越多越好"。  
它只做一件事：帮你把"先读后编"变成真正有顺序的动作。

如果主文讲的是"为什么先读后编"，这份附录讲的就是：

- 先看谁
- 看什么
- 怎么看
- 最容易误读成什么

## A1. 何时打开这份附录

最适合在下面几个时刻打开：

- 你已经接受"先读后编"是更省时间的路径，但不知道先读哪个对象
- 你准备开始拆样本，不想继续"随便翻"
- 你想训练自己看到结构、边界和路由，而不是只看内容热闹

## A2. 先看这 5 个生态入口样本

这 5 个对象足够构成一条高杠杆阅读路线。

| 样本 | 对象类型 | 最适合学什么 | 先看什么 | 不要误读成什么 |
| --- | --- | --- | --- | --- |
| `skills.sh` | `registry / directory` | 现成 skill 去哪里找、公开入口长什么样 | 首页定位、分类、`Official / Audits / Docs`、安装命令 | 质量背书系统 |
| `github/awesome-copilot` | `community learning hub` | 教程层和社区入口如何组织 | Learning Hub、分类方式、第三方提醒 | 工程基座 |
| `vercel-labs/agent-skills` | `sample-library` | 高质量 skill 样板怎么组织 | `SKILL.md`、`Use when`、`references/`、`scripts/` | installer 或治理层 |
| `vercel-labs/skills` | `installer / manager` | skill 怎样被受控安装、列出、更新 | project scope、global scope、single source of truth | 自动可信或自动有效 |
| `skill-forge` | `governance / publish` | skill 写完后治理层长什么样 | audit、publish、quality gate、artifact 检查 | 冷启动唯一模板 |

这 5 个对象最好不要乱看。  
更省时间的顺序是：

1. 先用 `skills.sh` 和 `awesome-copilot` 扩视野。
2. 再用 `agent-skills` 学样板。
3. 再看 `skills` 和 `skill-forge` 补工程层与治理层。

## A3. 最值得拆的 4 个进阶系统样本

如果你已经不满足于"看生态入口"，而是想训练自己看系统骨架，最值得拆的是下面 4 个进阶样本。这 4 个样本代表了 skill 从"一个文件"长成"一套系统"的不同演化路径，可以按你当前的阶段选择性深入。

### 1. 生命周期驱动的样板库（代表案例：多阶段 skill 集合）

**它是什么：**  
一套把 skill 按开发生命周期阶段组织的样板集合。典型的阶段划分是 Define（定义）/ Plan（规划）/ Build（构建）/ Verify（验证）/ Review（审查）/ Ship（发布）。每个阶段下挂对应的 skill，每个 skill 有清晰的 `Use when`。

**最适合学：**

- 如何按生命周期切分 skill，而不是把所有东西塞进一个大 skill
- `Use when` 怎么写才能让系统在对的时机调用对的 skill
- skill 之间如何形成组合，而不是互相打架

**先看什么：**

- 整体目录结构，观察它是怎么按阶段组织的
- 选一个阶段（比如 Review），看它下面的 skill 是如何界定边界的
- 比较相邻阶段的 skill，理解"阶段切分"的判断依据

**最容易误读：**

- 以为重点是"技能条目数量多"
- 漏掉真正值钱的地方：流程纪律和阶段结构的清晰度

---

### 2. 从单一 skill 长成多层系统（代表案例：command + agent + workflow 五层结构）

**它是什么：**  
一个从简单 skill 文件演化成五层结构系统的例子。五层分别是：command（命令层）、agent（智能体层）、workflow（工作流层）、reference（参考资料层）、template（模板层）。每层有自己明确的职责，互相配合而不重叠。

**最适合学：**

- 一个 skill 系统在真实使用中会如何自然地"长出"多层结构
- command、workflow、template 三层各自负责什么，为什么不能合并
- 编排层（workflow 层）是怎么把多个能力组织成一套有顺序的流程的

**先看什么：**

- 顶层目录结构，观察五层分别叫什么、放什么
- 选一个具体工作流，追踪它是如何跨层调用的
- 看 template 层，理解它为什么要单独存在

**最容易误读：**

- 只盯 front matter（前置信息）或命令名，忽略编排层
- 以为这套系统的重点是"功能多"，而不是"分层清晰"

---

### 3. 把 skill 做成工作流强制插件（代表案例：hooks + bootstrap + 行为测试）

**它是什么：**  
一套把 skill 进一步演化为"工作流强制插件"的做法。核心特点是引入了 hooks（钩子机制，在特定事件时自动触发动作）、bootstrap（启动引导，session 开始时自动注入必要上下文）和行为测试（用来验证 skill 的注入和触发行为是否符合预期）。

**最适合学：**

- 如何让 skill 从"人工调用"变成"在特定事件下自动触发"
- hooks 和 bootstrap 机制是如何让宿主环境更可控的
- 行为测试是怎么用来验证 skill 的触发逻辑的

**先看什么：**

- hooks 的触发点和触发条件
- bootstrap 注入了什么、什么时候注入
- 行为测试的结构，理解它在测什么

**最容易误读：**

- 以为只是"多了几个 skill"
- 漏掉它其实在加强宿主控制和注入纪律这个核心

---

### 4. 高产品化的系统骨架（代表案例：专门角色 + eval harness + 模板治理）

**它是什么：**  
一个更接近"产品级"复杂度的系统样本。典型特征包括：specialist review（专门角色审查，不同任务由不同角色负责）、eval harness（评测框架，系统化地对 skill 效果做回归测试）、模板治理（template 的版本和结构有明确的管理方式），以及对浏览器运行时的接入。

**最适合学：**

- 复杂系统如何用"专门角色"来避免单一 skill 过载
- eval harness 的基本结构：它测什么、怎么触发、怎么记录结果
- 模板治理的意义：为什么不能让模板随意修改

**先看什么：**

- 总体架构，理解专门角色是如何分工的
- eval harness 相关部分，理解它是怎么把回归检查变成系统性流程的
- 模板的组织方式，理解治理层在这里扮演什么角色

**最容易误读：**

- 只看 README 或功能介绍，漏掉真正的系统化工作流设计
- 以为这种复杂度是"起步就该有的"——它是 skill 体系成熟后的演化结果

## A4. 两条推荐阅读路线

### 路线 1：时间很紧，只想先建立感觉

顺序：

1. `skills.sh`
2. `github/awesome-copilot`
3. `vercel-labs/agent-skills`

这条路线最适合：

- 你刚想把"先读后编"跑起来
- 你最需要的是先建立生态感觉和样板感觉

### 路线 2：已经会写一点，想升级成系统视角

顺序：

1. `vercel-labs/agent-skills`
2. 生命周期驱动样板（A3 第 1 类）
3. 多层结构系统（A3 第 2 类）
4. 工作流强制插件（A3 第 3 类）
5. 高产品化系统骨架（A3 第 4 类）

这条路线最适合：

- 你已经写过 skill
- 但你想知道 skill 怎么一步步长成更系统的工作流

## A5. 看样本时一定要问的 8 个问题

不管你看哪个样本，都先问自己下面 8 个问题。

1. 它解决的是一个任务、一个流程阶段，还是一个完整系统？
2. 它的入口层是什么？是 `SKILL.md`、命令、插件，还是更复杂的宿主结构？
3. 它的路由写法在哪里？是 `description`、`Use when`，还是别的机制？
4. 正文和 supporting files（支撑文件）是怎么分层的？
5. 哪些细节被脚本化了，哪些仍然留在文字里？
6. 它最值得借的是结构、边界、编排，还是治理？
7. 它最容易被误抄的是哪一层？
8. 如果把它装进真实 workflow，最先该补哪一道 guardrail（护栏）？

只要你每看一个样本都认真回答这 8 个问题，你的阅读质量就会和"随便翻一眼"拉开明显差距。

## A6. 一个最好用的读样本模板

每拆一个样本，建议都按下面模板记笔记：

```md
样本名：

它最稳定的角色是什么：

它最值得借的 3 个点：
1.
2.
3.

它最容易被误抄的 2 个点：
1.
2.

如果我要借它，我只会先借哪一层：

如果我要把它装进真实 workflow，我最先补的 guardrail 是什么：
```

这个模板的作用，是逼你把"看热闹"变成"形成判断"。

## A7. 先读后编时，最容易犯的 4 个错

### 1. 看目录站时误以为自己已经完成了判断

目录站最多帮你找到对象，不帮你替你做信任判断。  
看到很多 skill，不等于知道哪些值得装。

### 2. 看样板库时只学表面格式

真正值钱的，往往不是 YAML 长什么样，而是：

- 什么时候触发
- 结构怎么分层
- 为什么细节被下沉

### 3. 看系统级样本时只看"功能多"

复杂对象最值得学的，常常是约束和组织方式，而不是功能列表。  
如果只盯着"好多命令"，你会错过真正的系统骨架。

### 4. 一边看样本，一边机械抄模板

借鉴样本的重点不是复制文本，而是学会回答：

- 为什么这么分层
- 为什么这样触发
- 为什么某些东西不写进正文

## A8. 读完这份附录后，下一步做什么

最推荐的动作不是继续收藏更多链接，而是立刻选一个样本，按照 A5 的 8 个问题做一次拆解。

如果你还没有拆过任何样本，最推荐从这三个开始：

1. `vercel-labs/agent-skills`（最容易上手的样板层）
2. 生命周期驱动样板（理解阶段切分和 Use when）
3. 高产品化系统骨架（建立系统视角）

这三个样本足够把"基础样板感"和"系统视角"同时拉起来。
