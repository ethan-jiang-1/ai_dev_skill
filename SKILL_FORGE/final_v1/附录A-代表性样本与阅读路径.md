# 附录 A：代表性样本与阅读路径

这份附录不追求“列得越多越好”。  
它只做一件事：帮你把“先读后编”变成真正有顺序的动作。

如果主文讲的是“为什么先读后编”，这份附录讲的就是：

- 先看谁
- 看什么
- 怎么看
- 最容易误读成什么

## A1. 何时打开这份附录

最适合在下面几个时刻打开：

- 你已经接受“先读后编”是更省时间的路径，但不知道先读哪个对象
- 你准备开始拆样本，不想继续“随便翻”
- 你想训练自己看到结构、边界和路由，而不是只看内容热闹

## A2. 先看这 5 个生态入口样本

这 5 个对象足够构成一条高杠杆阅读路线。

| 样本 | 对象类型 | 最适合学什么 | 先看什么 | 不要误读成什么 | 证据 |
| --- | --- | --- | --- | --- | --- |
| `skills.sh` | `registry / directory` | 现成 skill 去哪里找、公开入口长什么样 | 首页定位、分类、`Official / Audits / Docs`、安装命令 | 质量背书系统 | `E15` |
| `github/awesome-copilot` | `community learning hub` | 教程层和社区入口如何组织 | Learning Hub、分类方式、第三方提醒 | 工程基座 | `E16` |
| `vercel-labs/agent-skills` | `sample-library` | 高质量 skill 样板怎么组织 | `SKILL.md`、`Use when`、`references/`、`scripts/` | installer 或治理层 | `E14` |
| `vercel-labs/skills` | `installer / manager` | skill 怎样被受控安装、列出、更新 | project scope、global scope、single source of truth | 自动可信或自动有效 | `E17` |
| `skill-forge` | `governance / publish` | skill 写完后治理层长什么样 | audit、publish、quality gate、artifact 检查 | 冷启动唯一模板 | `E18` |

这 5 个对象最好不要乱看。  
更省时间的顺序是：

1. 先用 `skills.sh` 和 `awesome-copilot` 扩视野。
2. 再用 `agent-skills` 学样板。
3. 再看 `skills` 和 `skill-forge` 补工程层与治理层。

## A3. 最值得拆的 4 组本地样本

如果你已经不满足于“看生态入口”，而是想训练自己看系统骨架，最值得拆的是下面 4 组本地样本。

### 1. `agent-skills`

最适合学：

- 生命周期切分
- `Use when`
- skill 之间的组合关系
- 为什么一个样板库的重点不在“命令多”，而在“阶段切分清楚”

先看：

- `eval_skills/01-总览`
- `Define / Plan / Build / Verify / Review / Ship` 这些阶段的组织方式

最容易误读：

- 以为重点是技能条目多
- 漏掉真正值钱的地方是流程纪律和阶段结构

对应证据：

- `E19`

### 2. `get-shit-done`

最适合学：

- 一个 skill-like system（类 skill 系统）怎样从单一 skill 长成五层结构
- command、agent、workflow、reference、template 怎样互相配合

先看：

- 总览
- 结构拆解
- 各层分别承担什么责任

最容易误读：

- 只盯 front matter（前置信息）或命令名
- 忽略了编排层和模板层

对应证据：

- `E20`

### 3. `superpowers`

最适合学：

- skill 怎样进一步变成 workflow-enforcing plugin（强制工作流的插件）
- hooks（钩子）、bootstrap（启动引导）和行为测试怎样进入体系

先看：

- 总览
- 注入点
- 行为测试

最容易误读：

- 以为只是“多了几个 skill”
- 漏掉它其实在加强宿主控制和注入纪律

对应证据：

- `E21`

### 4. `gstack`

最适合学：

- 高级系统样本怎样组织 specialist review（专门角色审查）
- 模板治理、浏览器运行时和评测模式怎样接进来

先看：

- 总览
- 系统骨架
- eval harness（评测框架）相关做法

最容易误读：

- 只看 README 或功能介绍
- 漏掉真正的系统化工作流设计

对应证据：

- `E24`

## A4. 两条推荐阅读路线

### 路线 1：时间很紧，只想先建立感觉

顺序：

1. `skills.sh`
2. `github/awesome-copilot`
3. `vercel-labs/agent-skills`

这条路线最适合：

- 你刚想把“先读后编”跑起来
- 你最需要的是先建立生态感觉和样板感觉

### 路线 2：已经会写一点，想升级成系统视角

顺序：

1. `vercel-labs/agent-skills`
2. `agent-skills` 本地拆解
3. `get-shit-done`
4. `superpowers`
5. `gstack`

这条路线最适合：

- 你已经写过 skill
- 但你想知道 skill 怎么一步步长成更系统的工作流

## A5. 看样本时一定要问的 8 个问题

不管你看哪个样本，都先问自己下面 8 个问题。

1. 它解决的是一个任务、一个流程阶段，还是一个完整系统？
2. 它的入口层是什么？是 `SKILL.md`、命令、插件，还是更复杂的宿主结构？
3. 它的路由写法在哪里？是 `description`、`Use when`，还是别的机制？
4. 正文和 supporting files（支撑文件）是怎么分层的？
5. 哪些细节被脚本化了，哪些仍然留在文字里？
6. 它最值得借的是结构、边界、编排，还是治理？
7. 它最容易被误抄的是哪一层？
8. 如果把它装进真实 workflow，最先该补哪一道 guardrail（护栏）？

只要你每看一个样本都认真回答这 8 个问题，你的阅读质量就会和“随便翻一眼”拉开明显差距。

## A6. 一个最好用的读样本模板

每拆一个样本，建议都按下面模板记笔记：

```md
样本名：

它最稳定的角色是什么：

它最值得借的 3 个点：
1.
2.
3.

它最容易被误抄的 2 个点：
1.
2.

如果我要借它，我只会先借哪一层：

如果我要把它装进真实 workflow，我最先补的 guardrail 是什么：
```

这个模板的作用，是逼你把“看热闹”变成“形成判断”。

## A7. 先读后编时，最容易犯的 4 个错

### 1. 看目录站时误以为自己已经完成了判断

目录站最多帮你找到对象，不帮你替你做信任判断。  
看到很多 skill，不等于知道哪些值得装。[E11][E15]

### 2. 看样板库时只学表面格式

真正值钱的，往往不是 YAML 长什么样，而是：

- 什么时候触发
- 结构怎么分层
- 为什么细节被下沉

### 3. 看系统级样本时只看“功能多”

复杂对象最值得学的，常常是约束和组织方式，而不是功能列表。  
如果只盯着“好多命令”，你会错过真正的系统骨架。[E20][E21][E22]

### 4. 一边看样本，一边机械抄模板

借鉴样本的重点不是复制文本，而是学会回答：

- 为什么这么分层
- 为什么这样触发
- 为什么某些东西不写进正文

## A8. 读完这份附录后，下一步做什么

最推荐的动作不是继续收藏更多链接，而是立刻选一个样本，按照 A5 的 8 个问题做一次拆解。

如果你还没有拆过任何样本，最推荐从这三个开始：

1. `vercel-labs/agent-skills`
2. `agent-skills` 本地拆解
3. `gstack`

这三个样本足够把“基础样板感”和“系统视角”同时拉起来。
