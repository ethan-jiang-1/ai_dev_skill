# 附录 A：代表性样本与阅读路径

这份附录帮你把"先读后编"变成有顺序的动作。如果主文讲的是"为什么先读后编"，这里讲的是：先看谁、看什么、怎么看，以及最容易掉的坑在哪。

最后，我们会**手把手做一次完整的样本拆解示范**，让你看到"拆完之后笔记长什么样"。

---

## A1. 先看这 5 个生态入口

这 5 个对象构成一条省时间的阅读路线。建议按顺序看，不要乱跳：

| 样本 | 它像什么 | 最适合学什么 | 先看什么 | 别误读成什么 |
| --- | --- | --- | --- | --- |
| `skills.sh` | 黄页 | 现成 skill 去哪找 | 首页定位、分类、安装命令 | 质量背书 |
| `github/awesome-copilot` | 社区学习中心 | 教程层怎么组织 | 分类方式、第三方提醒 | 工程基座 |
| `vercel-labs/agent-skills` | 样板间 | 高质量 skill 怎么写 | `SKILL.md`、`Use when`、文件分层 | 安装工具或治理层 |
| `vercel-labs/skills` | 快递系统 | skill 怎样被受控安装 | project scope、global scope | 质量保证系统 |
| `skill-forge` | 质检部门 | 治理层长什么样 | audit、quality gate、artifact 检查 | 冷启动的唯一模板 |

推荐的阅读顺序：

1. **先用 `skills.sh` 和 `awesome-copilot` 扩视野。** 就像去一个城市先打开地图看看有什么，不急着进任何一家店。
2. **再用 `agent-skills` 学样板。** 选一个和你工作相关的 skill，认真拆开看它的结构。
3. **最后看 `skills` 和 `skill-forge` 补工程层。** 等你学会了怎么写，再学怎么装、怎么查。

## A2. 4 个进阶系统样本

如果你已经对基础样板有了感觉，想看看 skill 怎么长成"系统"，下面四类样本代表了四条不同的演化路径。它们的复杂度递增——从最容易上手的开始。

### 第 1 类：按生命周期组织的样板集合

想象一下：你不是只有一个 skill，而是有一整套——Define（定义）、Plan（规划）、Build（构建）、Verify（验证）、Review（审查）、Ship（发布）。每个阶段对应一到两个 skill，每个 skill 的 `Use when` 写得很清楚，系统在不同阶段自动调用对应的 skill。

这类样本最值得学的是**阶段切分的思路**——为什么不把所有东西塞进一个大 skill？因为一个大 skill 什么时候触发都说不清，而拆成多个小 skill 后，每个的触发时机就变得非常精确。

看的时候重点关注：目录结构怎么按阶段组织的、相邻阶段的 skill 边界怎么划的、`Use when` 怎么写才能让系统不搞混。

### 第 2 类：从单文件长成五层结构的系统

有些 skill 系统是从一个简单的 `SKILL.md` 一步步演化出来的，最终长成了五层：command（命令层）、agent（智能体层）、workflow（工作流编排层）、reference（参考资料层）、template（模板层）。

这类样本最值得学的是**自然生长的过程**——不是一开始就设计五层，而是用着用着发现"这部分应该独立出来"，于是慢慢分化。理解这个演化过程，比直接学最终结构更有价值。

看的时候重点关注：哪些东西从正文里分出去了、为什么分出去、编排层（workflow 层）是怎么把多个能力组织起来的。

### 第 3 类：用 hooks 和 bootstrap 做工作流强制

进一步的演化：skill 不再是"人工决定什么时候调用"，而是通过 hooks（钩子，在特定事件时自动触发）和 bootstrap（启动引导，session 开始时自动注入上下文）来**主动介入**。

打个比方：前面的 skill 像"你需要的时候去工具箱取"，这一类 skill 更像"一上班，安全检查表就自动弹到你桌上"。

看的时候重点关注：hooks 在什么事件上触发、bootstrap 注入了什么、有没有行为测试来验证这些自动化是否正常工作。

### 第 4 类：产品级系统骨架

最复杂的一类：有专门角色分工（不同任务由不同"专家"负责）、有 eval harness（评测框架，系统化地做回归测试）、有模板治理（模板的版本和修改有严格管控）。

这种复杂度不是一开始就该追求的，它是 skill 体系成熟之后的结果。看它的价值不在于"现在就照着搭"，而在于理解"一个 skill 体系最终可以长成什么样"——知道天花板在哪，回过头来搭基础就更有方向感。

> 想看这四类样本对应的具体仓库名称和详细拆解？它们分别是：第 1 类 → `agent-skills`（生命周期切分）、第 2 类 → `get-shit-done`（五层组合演化）、第 3 类 → `superpowers`（hooks + bootstrap 强制执行）、第 4 类 → `gstack`（产品级 sprint 系统）。详见 [附录G-进阶话题与深度参考](./附录G-进阶话题与深度参考.md) 的 G1 节。

## A3. 两条推荐路线

**路线 1：时间紧，先建立感觉。** `skills.sh` → `awesome-copilot` → `vercel-labs/agent-skills`。半天就能走完，适合刚开始想试试"先读后编"的人。

**路线 2：已经会写一点，想升级。** `vercel-labs/agent-skills` → 第 1 类（生命周期切分）→ 第 2 类（多层演化）→ 第 3 类（hooks 和 bootstrap）→ 第 4 类（产品级骨架）。需要几天，适合想建立系统视角的人。

## A4. 完整拆解示范：手把手拆一个样本

很多人知道"要拆样本"，但不确定"拆出来的东西应该长什么样"。下面用 `vercel-labs/agent-skills` 里的一个样板 skill 做一次完整示范。

### 你看到的原始材料

假设你打开了一个叫 `code-review` 的 skill 目录，看到这样的结构：

```
skills/code-review/
├── SKILL.md
├── references/
│   └── review-checklist.md
├── scripts/
│   └── review-pass.sh
└── examples/
    └── good-review-example.md
```

你打开 `SKILL.md`，看到这样的开头：

```md
---
name: code-review
description: Use when reviewing a pull request and you need a structured
  risk-first review pass.
---

1. Identify the type of change: feature, fix, or refactor.
2. Assess scope, test coverage, and rollback risk first.
3. For detailed checks, see `references/review-checklist.md`.
4. For automated check sequence, run `scripts/review-pass.sh`.
```

### 你的拆解笔记应该长这样

```md
样本名：code-review (vercel-labs/agent-skills)

它最稳定的角色是什么：
结构样板。适合学习 skill 应该怎么组织，不适合直接当工程基座或治理工具。

它最值得借的 3 个点：
1. description 不只描述功能，而是在做路由——"Use when... and you need..."
   同时定义了"什么时候该进"和"什么时候不该进"。
2. 正文只有 4 行主线步骤。详细检查项被下沉到 references/ 目录，
   以后改检查项不需要碰主文。
3. 重复操作被脚本化了（review-pass.sh），不依赖人工记忆。

它最容易被误抄的 2 个点：
1. description 的具体措辞。直接抄 "risk-first review pass" 没意义，
   你要根据自己的任务写自己的路由条件。
2. 目录结构。不是所有 skill 都需要 examples/ 和 scripts/。
   如果你的 skill 很简单，强行凑齐四个子目录反而是噪音。

如果我要借它，我只会先借哪一层：
description 的路由写法。这是整个 skill 里最有杠杆的一行——
写对了，系统知道什么时候该调用它；写错了，后面所有步骤都白费。

如果我要装进真实 workflow，最先补的 guardrail 是什么：
一组代表性的 PR 任务（5-10 个），用来做有 skill / 无 skill 的对照评测。
确认它是真的减少了遗漏，而不只是让回答更像模板。
```

### 为什么这种笔记有用

注意这份笔记里**没有**什么：没有"这个 skill 看起来不错"这种模糊评价，没有抄过来的大段代码，没有"可以借鉴"这种笼统说法。

每一条都是**具体的判断**——判断它的角色是什么、值得借的是哪个层面、容易误抄的是哪里。这种判断才是你以后自己写 skill 时真正用得上的东西。

## A5. 看样本时一定要问的问题

不管你看哪个样本，都带着这些问题去看。不需要每个都回答，但至少过一遍脑子：

1. 它解决的是一个任务、一个流程阶段，还是一个完整系统？
2. 它的路由写法（`description`、`Use when`）是怎么决定"什么时候进、什么时候不进"的？
3. 正文和支撑文件是怎么分层的？哪些东西被分出去了？
4. 有没有脚本化的部分？为什么那些东西值得脚本化？
5. 它最值得借的是结构、边界、编排，还是治理？
6. 如果把它装进真实 workflow，最先该补什么 guardrail（护栏）？

## A6. 先读后编时最容易犯的 4 个错

**一、看目录站时以为自己已经完成了判断。** 你在 `skills.sh` 上翻了二十分钟，觉得"大概知道有什么了"。但其实你只完成了发现，信任判断还没开始。看到很多 skill，不等于知道哪些值得装。

**二、看样板库时只学了格式。** 你注意到 YAML 头部怎么写、目录怎么建，但没看到更底层的东西：为什么这么分层？为什么某些内容被下沉到支撑文件？触发条件是怎么设计的？格式是最容易学的，也是最不值钱的。

**三、看系统级样本时只看到"功能多"。** 复杂样本最值得学的，往往是它的约束和组织方式——它怎么决定"什么不做"、怎么划分角色之间的边界。如果你只盯着"好多命令"，你会错过真正的系统骨架。

**四、一边看，一边机械抄模板。** 借鉴不是复制文本。你要学会回答"为什么"——为什么这么分层？为什么这样触发？为什么某些东西不写进正文？这些"为什么"才是你真正需要带走的东西。

---

## A7. 读完这份附录，下一步做什么

立刻选一个样本，用 A4 里的方式做一次完整拆解。如果你还没拆过任何样本，从 `vercel-labs/agent-skills` 开始——它是最容易上手的样板层。

拆完一个，你就会发现"先读后编"不是一个抽象原则，而是一种实实在在能让你少走弯路的方法。
