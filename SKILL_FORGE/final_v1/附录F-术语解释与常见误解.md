# 附录 F：术语解释与常见误解

这不是一份只给你看“翻译”的 glossary（术语表）。  
它更像一份小型解释手册，专门照顾这样一种情况：

- 你已经在做 skill
- 你能大概猜到这些词是什么意思
- 但你不想一直靠猜

因此每个术语条目都会尽量回答五件事：

1. 这词直译成中文是什么
2. 它在本 Playbook 里到底是什么意思
3. 它最容易和什么混淆
4. 一个最小例子
5. 什么时候你其实不用太在意它

## F1. 基础对象术语

### `skill`

- 中文直觉：能力包
- 在本 Playbook 里的意思：
  - 围绕某类任务或流程阶段设计的、按需加载的目录级能力包
- 最容易混淆：
  - 和单段 prompt 混淆
  - 和常驻规则混淆
- 最小例子：
  - “数据库迁移检查” skill，只在做迁移时才加载
- 什么时候不用太纠结：
  - 当你只是在理解大方向时，先记住它不是散落提示词就够了

### `AGENTS.md`

- 中文直觉：常驻规则文件
- 在本 Playbook 里的意思：
  - repo-level（仓库层）、持续生效的 agent 指导
- 最容易混淆：
  - 和 task-level（任务层）的 skill 混淆
- 最小例子：
  - 仓库统一测试命令、禁止修改的目录、提交约定
- 什么时候不用太纠结：
  - 当你只是区分“常驻”还是“按需”时，先抓这个差别就够了

### `sample library`

- 中文直觉：样板库
- 在本 Playbook 里的意思：
  - 提供高质量 skill 写法、结构和组织方式的参考库
- 最容易混淆：
  - 和 installer 混淆
- 最小例子：
  - 一个仓库里放了很多范例 skill，适合阅读和拆解
- 什么时候不用太纠结：
  - 当你只是想先知道“去哪学写法”时

### `installer / manager`

- 中文直觉：安装与管理层
- 在本 Playbook 里的意思：
  - 负责把 skill 安装、列出、更新、映射到项目或个人环境中的那层工具
- 最容易混淆：
  - 和治理层、样板库混淆
- 最小例子：
  - 把一个 skill 从公开来源装进当前项目目录
- 什么时候不用太纠结：
  - 当你还停留在“只看样本”阶段时

### `governance / publish`

- 中文直觉：治理与发布层
- 在本 Playbook 里的意思：
  - 检查 skill 的结构、安全、可发现性和发布前质量门槛的那层能力
- 最容易混淆：
  - 和 installer 混淆
- 最小例子：
  - 安装之前先做审查，发布之前先过结构和安全检查
- 什么时候不用太纠结：
  - 当你还没有进入长期维护阶段时

### `directory / registry`

- 中文直觉：目录站、发现入口
- 在本 Playbook 里的意思：
  - 帮你找到 skill、看见生态入口和公开信号的地方
- 最容易混淆：
  - 和质量背书系统混淆
- 最小例子：
  - 一个网站列出大量公开 skill 和安装命令
- 什么时候不用太纠结：
  - 当你只是想知道“去哪里找”

## F2. 方法与结构术语

### `portable core（可移植核心）`

- 在本 Playbook 里的意思：
  - 不严重依赖某一个平台，换个宿主仍大体能继续用的 skill 核心部分
- 最容易混淆：
  - 和“所有平台都完全一样”混淆
- 最小例子：
  - `SKILL.md`、`name`、`description`、主线步骤和支撑文件导航
- 什么时候不用太纠结：
  - 当你只在单一平台内部使用 skill 时

### `surface-specific extensions（宿主特定扩展）`

- 在本 Playbook 里的意思：
  - 某个平台自己支持、但不该被误写成通用标准的能力
- 最容易混淆：
  - 和通用 skill 规范混淆
- 最小例子：
  - 某平台特有的 frontmatter 字段或插件打包方式
- 什么时候不用太纠结：
  - 当你短期内根本不考虑跨平台迁移时

### `progressive disclosure（渐进展开）`

- 在本 Playbook 里的意思：
  - 主文只放主线，细节分散到支撑文件，按需要再展开
- 最容易混淆：
  - 和“写得不完整”混淆
- 最小例子：
  - 主文只写 4 步，详细检查项放进 `references/checklist.md`
- 什么时候不用太纠结：
  - 当你的 skill 还很小、内容还不多时

### `artifact（工件）`

- 中文直觉：产物、构件
- 在本 Playbook 里的意思：
  - skill 作为一个完整对象，不只是正文，而是正文、结构、脚本、支撑文件、版本等一起构成的整体
- 最容易混淆：
  - 和“只是那几行 prompt”混淆
- 最小例子：
  - 一个 skill 包含 `SKILL.md`、脚本、引用文件和版本记录
- 什么时候不用太纠结：
  - 当你只是刚接触 skill 概念时

## F3. 工作流术语

### `baseline（基线）`

- 在本 Playbook 里的意思：
  - 当前最小但足够稳、可以作为默认起点的组合或写法
- 最容易混淆：
  - 和“最终答案”混淆
- 最小例子：
  - 先用样板库 + 安装层 + 信任门，构成第一版 baseline
- 什么时候不用太纠结：
  - 当你只是在理解生态，而还没准备自己搭 workflow 时

### `controlled install（受控安装）`

- 在本 Playbook 里的意思：
  - 先小范围、可回滚地安装 skill，而不是一上来全局铺开
- 最容易混淆：
  - 和“安装速度慢”混淆
- 最小例子：
  - 先只在一个项目里试装，而不是所有仓库全开
- 什么时候不用太纠结：
  - 当你还在纯阅读阶段

### `trust gate（信任门）`

- 在本 Playbook 里的意思：
  - 安装或启用前，先确认 skill 的来源、脚本、支撑文件和风险边界
- 最容易混淆：
  - 和“感觉靠谱”混淆
- 最小例子：
  - 读一遍 `SKILL.md` 和 `scripts/` 再决定装不装
- 什么时候不用太纠结：
  - 当 skill 完全是你自己写、自己用、没有外部来源时

### `role-based bundles（按角色打包）`

- 在本 Playbook 里的意思：
  - 不把所有 skill 一股脑暴露出去，而是按角色或任务组合成小包
- 最容易混淆：
  - 和“功能越多越强”混淆
- 最小例子：
  - 只给 review 角色暴露 review 相关 skills
- 什么时候不用太纠结：
  - 当你只有一两个 skill，还不存在选择过载时

### `version pinning（版本固定）`

- 在本 Playbook 里的意思：
  - 记录当前验证通过的版本，不默认追最新
- 最容易混淆：
  - 和“永远不更新”混淆
- 最小例子：
  - 先固定在某个 skill 版本，验证完再升级
- 什么时候不用太纠结：
  - 当 skill 还处在草稿阶段，尚未正式进入常用 workflow

## F4. 评测与优化术语

### `evaluation（评测）`

- 在本 Playbook 里的意思：
  - 用代表性任务比较 skill 是否真的带来改善
- 最容易混淆：
  - 和“能加载成功”混淆
- 最小例子：
  - 同一组任务，有 skill 和无 skill 各跑一次
- 什么时候不用太纠结：
  - 当你还在非常早期的概念探索阶段

### `failure taxonomy（失败分类）`

- 在本 Playbook 里的意思：
  - 把失败按类型分清，而不是一律写成“效果不好”
- 最容易混淆：
  - 和“复杂标签游戏”混淆
- 最小例子：
  - 把失败分成触发失败、工具失败、结构失败
- 什么时候不用太纠结：
  - 当你还没有任何真实失败样本时

### `trajectory regression（轨迹回归）`

- 在本 Playbook 里的意思：
  - 每次修改后，不只检查最终结果，还检查中间步骤和调用轨迹有没有跑偏
- 最容易混淆：
  - 和“只看最后答案”混淆
- 最小例子：
  - 看 skill 是否按顺序调用了该调用的工具
- 什么时候不用太纠结：
  - 当你的 skill 完全没有多步流程或工具调用时

### `feedback loop（反馈闭环）`

- 在本 Playbook 里的意思：
  - 把真实使用中暴露的问题，重新带回下一轮修改
- 最容易混淆：
  - 和“收集点意见”混淆
- 最小例子：
  - 用户修正过的失败案例，后续被整理成回放样本
- 什么时候不用太纠结：
  - 当 skill 还没进入真实高频使用时

### `candidate revision（候选修订）`

- 在本 Playbook 里的意思：
  - 先生成一个可能更好的版本，作为候选，再拿去比较
- 最容易混淆：
  - 和“自动晋级的新版本”混淆
- 最小例子：
  - 先改写 `description` 形成一个候选版本，跑完对照再决定是否采用
- 什么时候不用太纠结：
  - 当你还没有 baseline，也没有比较样本时

### `human promotion gate（人工晋级门槛）`

- 在本 Playbook 里的意思：
  - 最终决定候选版是否进入常用版本，仍要靠人工判断，不完全交给自动化
- 最容易混淆：
  - 和“人工全做，不需要自动化”混淆
- 最小例子：
  - 自动系统先给一个候选版，人工看回放结果后再批准升级
- 什么时候不用太纠结：
  - 当你还处在非常早期的手工修改阶段

## F5. 最容易混在一起的 6 组词

### 1. `skill` vs `prompt`

- `prompt` 更像一段指令文本
- `skill` 更像一个按需能力包

### 2. `AGENTS.md` vs `skill`

- 前者偏常驻
- 后者偏按需

### 3. `sample library` vs `installer`

- 前者教你怎么写
- 后者帮你怎么装

### 4. `directory` vs `trust signal`

- 前者说明找得到
- 后者才说明能不能信

### 5. `baseline` vs `best`

- 前者是当前最稳的起点
- 后者像在追唯一最优解

### 6. `candidate revision` vs `promotion`

- 前者是候选版本
- 后者是正式晋级

## F6. 如果你觉得术语还是多，先记这 5 句话

1. skill 是按需能力包，不是单段 prompt。
2. 先读样本，再写自己的版本。
3. 目录站帮你找，不帮你替你信。
4. baseline 是起点，不是终点。
5. skill 第一次能用，不代表它已经成熟。

## F7. 读完这份附录后，下一步做什么

选你最容易混淆的三个术语，把它们各自写成一句自己的话。  
只要你能用自己的话解释它们，后面读主文和其他附录时，就不会总觉得在追着名词跑。
