# 附录 F：术语解释与常见误解

这不是一份只给翻译的术语表。它更像一份"把你从似懂非懂带到真正理解"的小手册。

你已经在做 skill 了，你大概能猜到这些词是什么意思——但"大概猜到"和"真正理解"之间经常隔着一些微妙的误解。这份附录专门处理这些误解。

每个术语我会尽量讲清楚五件事：它在本 Playbook 里到底指什么、一个日常类比、一个最小的 skill 场景例子、它最容易和什么混淆、什么时候你其实不用太在意它。

---

## 基础对象术语

### skill（技能包 / 能力包）

**在本 Playbook 里的意思：** 围绕某类任务设计的、按需加载的目录级能力包。

**日常类比：** 工具箱，不是便利贴。便利贴上写了"记得检查"就贴在墙上了；工具箱里有扳手、螺丝刀和说明书，需要的时候拿出来用。

**最小例子：** 你建了一个 `skills/code-review/` 目录，里面有 `SKILL.md`（入口，写了什么时候该用它）、`references/review-checklist.md`（详细检查项）、`scripts/review-pass.sh`（自动检查脚本）。这就是一个 skill——不是一段话，而是一个有结构的目录。

**最容易混淆：** 和单段 prompt 混淆（"写了一段话给 AI 就算 skill 了吧？"——不算，那是便利贴）；和常驻规则混淆（`AGENTS.md` 是一直生效的规章制度，skill 是只在特定任务时才需要的工具箱）。

**什么时候不用太纠结：** 如果你只是在理解大方向，先记住"它比一段话复杂"就够了。

---

### AGENTS.md（常驻规则文件）

**在本 Playbook 里的意思：** 仓库级的常驻规则文件，放着测试命令、目录禁区、提交约定这些"一直要遵守"的东西。

**日常类比：** 公司规章制度。你入职第一天就要遵守，不是做某个特定任务的时候才看。

**最小例子：** 你的项目根目录有一个 `AGENTS.md`，写着"测试用 `npm test`""不要修改 `dist/` 目录""提交信息用中文"。不管 agent 在做什么任务，这些规则始终生效。

**最容易混淆：** 和 skill 混淆。区分方法很简单：`AGENTS.md` 是"任何时候都生效"，skill 是"只在特定任务时才加载"。

**什么时候不用太纠结：** 如果你现在只写 skill，还没接触过仓库级规则配置，先跳过这个概念。等你开始管整个 repo 的 agent 行为时再来。

---

### installer / manager（安装器 / 管理器）

**在本 Playbook 里的意思：** 负责把 skill 安装、列出、更新、映射到你的项目或个人环境的工具层。

**日常类比：** 快递系统。它负责把东西送到你家，但它不负责这个东西好不好用——你拆包之后仍然要自己检查。

**最小例子：** 你运行 `skills install code-review`，installer 把 skill 下载到 `.cursor/skills/code-review/` 里。安装成功了——但 skill 好不好用，你还没验证。

**最容易混淆：** 和治理层混淆（"能装进去说明质量没问题"——不对，快递员不负责质量检查）；和样板库混淆（"从这里安装的 skill 结构一定好"——不一定）。

**什么时候不用太纠结：** 如果你的 skill 都是手动复制到项目目录里的，installer 暂时和你无关。等你需要管理多个 skill 的版本和更新时再来了解。

---

### directory / registry（目录站 / 注册表）

**在本 Playbook 里的意思：** 帮你找到 skill 的入口。像黄页一样，列出大量对象供你浏览和搜索。

**日常类比：** 黄页。黄页上有一家餐厅不代表它好吃，只代表它登记了。

**最小例子：** 你打开 `skills.sh`，搜索 "code review"，找到了 15 个相关 skill。你现在知道它们存在了——但哪个值得用，你还不知道。

**最容易混淆：** 和质量背书混淆。"在目录站上找得到"只能说明它被收录了，不能说明它值得信任。

**什么时候不用太纠结：** 如果你已经知道要用哪个 skill 了，目录站对你就不重要了。它主要服务于"还不知道有什么"的探索阶段。

---

### governance / publish（治理 / 发布）

**在本 Playbook 里的意思：** 检查 skill 的结构、安全和质量，负责发布前的把关。

**日常类比：** 质检部门。产品出厂前要过质检——结构完整吗？有安全隐患吗？满足发布标准吗？

**最小例子：** 你的代码审查 skill 准备发到团队共享仓库。`skill-forge` 检查了它：`SKILL.md` 有没有 `description`？有没有不安全的脚本？结构完整吗？通过了才能发。

**最容易混淆：** 和 installer 混淆（"质检和快递是两回事"）。

**什么时候不用太纠结：** 如果你的 skill 还在早期草稿阶段、只是自己用，先不用管治理。等准备分享或发布的时候再来。

## 方法与结构术语

### portable core（可移植核心）

**在本 Playbook 里的意思：** 不严重依赖某个平台，换个地方也能沿用的 skill 核心部分。

**日常类比：** 菜谱的核心步骤。"先腌 2 小时、再 200 度烤 40 分钟"——不管你用什么牌子的烤箱，这个顺序和比例不变。但"用哪个档位"要看你的烤箱。

**最小例子：** 你的代码审查 skill 有一个 `SKILL.md`，写了 `name: code-review`、`description: Use when reviewing a PR`、四步主线流程。这四样东西换到 Codex、Claude、GitHub 都不用改——这就是 portable core。而 `allowed-tools` 字段只在某些平台有效，不属于 portable core。

**最容易混淆：** 和"所有平台都完全一样"混淆。有 portable core 不代表三家平台毫无差异——差异存在，但核心可以迁移。

**什么时候不用太纠结：** 如果你只在一个平台上用 skill，先不用管跨平台。

---

### progressive disclosure（渐进展开）

**在本 Playbook 里的意思：** 主文只放主线步骤，细节按需分散到支撑文件里。

**日常类比：** 好的菜谱不会在第一页就把所有酱料的化学成分写出来。先告诉你"放两勺酱油"，想知道为什么是两勺的人可以翻到后面的备注看。

**最小例子：** 你的 `SKILL.md` 正文只写"详细检查项见 `references/review-checklist.md`"。agent 平时只看正文的四行主线步骤；真正执行到检查环节时才去加载那份 20 行的检查清单。

**最容易混淆：** 和"写得不完整"混淆。不是偷懒没写完，而是刻意把细节分层了。

**什么时候不用太纠结：** 如果你的 skill 很简单，一个 `SKILL.md` 就说清楚了，没有必要为了"渐进展开"硬拆文件。只有当内容多到读者会迷失主线时，才值得分层。

---

### artifact（工件）

**在本 Playbook 里的意思：** skill 作为一个完整对象——不只是正文那几行字，而是正文、结构、脚本、支撑文件、版本记录加在一起的整体。

**日常类比：** 一辆车不只是发动机。它还有底盘、刹车、导航和保养记录。优化一辆车不是只调发动机——skill 也一样。

**最小例子：** 你的代码审查 skill artifact 包括：`SKILL.md`（正文）、`references/review-checklist.md`（检查项）、`scripts/review-pass.sh`（脚本）、`CHANGELOG.md`（版本记录）。优化 skill 不是只改正文——可能需要改脚本、改检查项、或者更新版本记录。

**什么时候不用太纠结：** 刚开始写 skill 时，先把正文写好就行。等你的 skill 开始有 scripts、references、版本记录了，再回来理解 artifact 这个整体概念。

## 工作流术语

### baseline（基线）

**在本 Playbook 里的意思：** 当前最小但足够稳、可以作为起点的组合或写法。

**日常类比：** 做菜新手的第一套基础厨具——一把刀、一个砧板、一口锅。不是最终配置，但够你开始做菜了。

**最小例子：** 你选了 `agent-skills` 学结构 + `vercel-labs/skills` 做安装 + `skill-forge` 做质检。这三层加起来就是你当前的 baseline——够你开始搭 skill 体系了，以后再加评测工具。

**最容易混淆：** 和"最终答案"混淆。baseline 是起点，不是终点。

**什么时候不用太纠结：** 如果你还在探索阶段，先别纠结 baseline 到底包含什么。随便选一个能跑起来的组合就是你的 baseline，后面再慢慢调整。

---

### trust gate（信任门）

**在本 Playbook 里的意思：** 安装或启用 skill 之前，先确认它的来源、脚本和风险边界。

**日常类比：** 搬进新家前检查水电。房东说"放心"你也不会真的不查——万一有漏水、漏电呢？

**最小例子：** 你从 `skills.sh` 找了一个 skill 准备装。装之前你先打开它的 `SKILL.md` 和 `scripts/` 目录通读一遍——发现有个脚本会 `curl` 一个外部 URL。你决定先不装，去弄清楚这个请求是干什么的。这就是 trust gate。

**最容易混淆：** 和"感觉靠谱"混淆。trust gate 是实际的检查动作，不是主观印象。

**什么时候不用太纠结：** 如果你用的 skill 是自己从零写的，trust gate 可以简化——毕竟内容是你自己写的。trust gate 主要防的是"从外部引入不可控内容"的风险。

---

### controlled install（受控安装）

**在本 Playbook 里的意思：** 先小范围、可回滚地安装 skill，不一上来全局铺开。

**日常类比：** 新药上市前先做小规模临床试验，不会直接让全国人民都吃。

**最小例子：** 你先在一个测试项目里装了代码审查 skill，跑了 5 个 PR 审查任务确认没问题，才在团队的主项目里正式装。

**什么时候不用太纠结：** 如果你的 skill 只是自己用或小团队用，直接装就行，不需要走 controlled install 的完整流程。等 skill 要在大团队或生产环境铺开时再考虑。

---

### role-based bundles（按角色打包）

**在本 Playbook 里的意思：** 不把所有 skill 一股脑暴露出去，而是按角色或任务组合成小包。

**日常类比：** 医院的急救推车和手术推车上放的东西不同——不是因为有些工具不好，而是因为不同场景需要不同的组合。

**最小例子：** 你有 12 个 skill。你把 `code-review`、`code-simplification`、`security-hardening` 打成"审查员"包；把 `incremental-implementation`、`test-driven-development` 打成"开发者"包。做审查时只加载审查员包，agent 不会在 12 个里面乱选。

**什么时候不用太纠结：** 如果你的 skill 总共就三五个，按角色打包没意义。这个概念是给 skill 池增长到十几个以上、agent 开始选错 skill 的时候准备的。

---

## 评测与优化术语

### feedback loop（反馈闭环）

**在本 Playbook 里的意思：** 把使用过程中发现的问题带回来，改进下一版。

**日常类比：** 医生不是开完药就走。要回诊看疗效——好了就维持，没好就换方案。skill 也需要这种"回诊"。

**最小例子：** 你的代码审查 skill 上线后，第三周有同事说"它建议改的那段代码其实没问题"。你记录下来：这是一个误报。下次改 skill 时，你把这个 case 加到测试集里，确保修改后不再误报。

**什么时候不用太纠结：** 如果你的 skill 刚写好还没上线，先别想反馈闭环。等 skill 用了一阵子，积累了一些真实使用数据再来建闭环。

---

### failure taxonomy（失败分类）

**在本 Playbook 里的意思：** 把失败按类型分清，而不是统统写成"效果不好"。

**日常类比：** 你去医院不会说"我不舒服"就完了——是头疼、胃疼还是胸闷？不同症状指向不同的治疗方案。

**最小例子：** 你的 skill 最近出了三次问题。你分清楚：第一次是"该触发没触发"（触发失败），第二次是"步骤顺序反了"（工作流失败），第三次是"改了 description 之后旧任务坏了"（回归失败）。三类问题修法完全不同。

**什么时候不用太纠结：** 如果你的 skill 还处于"偶尔用用"的阶段，遇到问题直接修就行。等你开始做系统性的 skill 维护时，再回来学分类方法。

---

### trajectory regression（轨迹回归）

**中文直译：** 轨迹回归。

**在本 Playbook 里的意思：** 改了 skill 之后，不只检查最终输出，还检查中间步骤和调用轨迹有没有跑偏。

**日常类比：** 老师不只看考试答案，还看解题过程。答案碰巧对了但过程全错，下次一定出问题。

**最小例子：** 你改了代码审查 skill 的一步，新版本最终输出的审查意见看起来和旧版本一样好。但你检查中间步骤发现，新版本跳过了"先确认改动类型"直接开始检查——过程不对，只是碰巧答案没变差。

**什么时候不用太纠结：** 如果你的 skill 不涉及多步骤工具调用，轨迹回归暂时用不上。它主要服务于那些步骤链条长、中间过程容易跑偏的复杂 skill。

---

### candidate revision（候选修订）

**中文直译：** 候选修订。

**在本 Playbook 里的意思：** 先生成一个"可能更好"的版本作为候选，拿去和 baseline 做对照比较，再决定要不要采用。

**日常类比：** 医生先开一个新方案作为"试试看"，看了疗效再决定是否正式替换旧方案——不是开了就直接换。

**最小例子：** 你觉得代码审查 skill 的 `description` 可以写得更准。你写了一个新版本但不直接替换旧版——而是标为"候选版"，拿去和旧版对跑 8 个测试任务，看结果再决定要不要正式采用。

**最容易混淆：** 和"自动晋级的新版本"混淆。候选只是候选，是否正式采用还得人来判断。

---

### human promotion gate（人工晋级门槛）

**中文直译：** 人工晋级门槛。

**在本 Playbook 里的意思：** 最终决定候选版是否正式进入常用版本，仍然由人来判断，不完全交给自动化。

**日常类比：** 自动化帮你筛简历，但最终录不录用还是你来面试决定。

**最小例子：** 工具帮你自动生成了 3 个 `description` 候选版，并自动跑了回归测试，其中一个通过率最高。但你不是自动切换到它——你先看了一遍它的措辞，确认没有隐含的边界漏洞，才手动 promote（晋级）。

**什么时候不用太纠结：** 如果你的 skill 迭代还是手动的（自己改、自己试），这个概念暂时离你比较远。等你开始用自动化工具（如 DSPy、Promptfoo）生成候选版本时，human promotion gate 才变得关键。

---

## 如果你觉得术语还是多，先记这五句话

1. **skill 是工具箱，不是便利贴。**
2. **先找样本来拆，再写自己的版本。**
3. **黄页帮你找餐厅，但上了黄页不代表好吃。**
4. **baseline 是起点，不是终点。**
5. **开完药还要回诊看疗效。**

---

## 读完这份附录，下一步做什么

选你最容易混淆的三个术语，用自己的话各写一句解释。不用写得很正式——关键是能用**你自己的语言**说清楚它是什么。如果你做到了，后面读主文和其他附录的时候，就不会总觉得在追着名词跑。
