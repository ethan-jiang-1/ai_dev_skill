# 附录 B：对象分层与选择指南

主文第 4-5 节已经讲了生态分层的整体框架和 baseline 组合的逻辑。这份附录**不重复那些内容**，而是帮你解决一个更具体的问题：**面对每个对象，到底怎么判断它适不适合我现在用？**

---

## B1. 每个对象的详细拆解

主文里把对象按"层"来归类。这里换一个角度——按对象逐个展开，告诉你每个对象**到底擅长什么、边界在哪、什么场景下值得用**。

### `skills.sh`

**一句话定位：** skill 生态的黄页。

它做的事情很简单：帮你快速发现有哪些现成 skill 可以用。它有分类浏览、有安装命令、有 official/audits 标签。如果你刚开始接触 skill 生态，它能让你十分钟内就知道外面已经有什么。

**它的边界：** 它不审查 skill 的内容质量，也不做效果评测。就像黄页能帮你找到餐厅，但黄页上有名字不代表这家餐厅好吃。"在 skills.sh 上找得到"这件事提供的是发现便利，不是质量背书。

**什么时候用它：** 当你想快速扩视野、想知道某个任务领域有没有现成 skill 可以参考的时候。

**什么时候不该依赖它：** 当你需要判断一个 skill 值不值得装进生产环境的时候。那是 trust gate 和对照评测该做的事。

---

### `github/awesome-copilot`

**一句话定位：** 社区驱动的学习中心。

它把教程、样本、工具入口聚合到了一起，组织方式像一个精选书单。对"借鉴别人怎么写"特别有帮助——你可以看到不同人、不同团队是怎么理解和实践 skill 的。

**它的边界：** 它是学习资源，不是工程基座。里面聚合的内容质量参差不齐，有官方的也有第三方的。适合用来"看看别人怎么做"，但不适合直接拿来当你的工程起点。

**什么时候用它：** 当你想扩展思路、看看社区有什么新的实践方式的时候。

**什么时候不该依赖它：** 当你需要一个结构化的、可以直接跟着搭的 baseline 时。

---

### `vercel-labs/agent-skills`

**一句话定位：** 高质量的样板间。

这是当前最适合用来学结构的对象。它的 skill 样本有清晰的 `Use when`、合理的文件分层、恰当的 progressive disclosure（主文放主线、细节下沉到支撑文件）。如果你只看一个样板库来学"skill 应该长什么样"，看这个。

**它的边界：** 它是样板库，不是安装工具、不是治理系统、也不是评测平台。你能从它那里学到优秀的结构和写法，但"学完了"和"能用了"之间还隔着安装、审查和评测好几步。

**什么时候用它：** 当你想学 skill 的结构设计、触发写法和文件组织时。当你准备动手写自己的 skill，需要一个好样本来参考时。

**什么时候不该依赖它：** 当你以为"从这里复制一个 skill 过来就能直接用"的时候。

---

### `vercel-labs/skills`

**一句话定位：** skill 的快递系统和仓库管理。

它负责的是把 skill 安装到你的项目或个人环境里，并且提供列出、更新、路径映射和版本管理能力。可以理解成 npm 之于 JavaScript 包——它管的是"怎么装进去"和"怎么管起来"。

**它的边界：** 安装成功不等于效果好。就像 `npm install` 成功了不代表这个包真的适合你的项目。装完之后，你仍然需要独立做 trust gate（审查内容）和对照评测（有 skill vs 无 skill 的比较）。

**什么时候用它：** 当你准备正式把 skill 引入项目、需要受控安装和版本管理时。

**什么时候不该依赖它：** 当你把"能装进去"等同于"效果有保证"的时候。

---

### `skill-forge`

**一句话定位：** skill 的质检部门。

它做的是结构审计、质量门槛和发布前治理。你可以把它理解成代码世界里的 lint + CI check——它检查你的 skill 有没有满足某些结构和质量标准，在发布前帮你拦住一些明显的问题。

**它的边界：** 治理方向很重要，但不是所有实践都该从它起步。如果你还在"学怎么写"的阶段，先去看样板库；如果你已经写了一批 skill 准备正式发布了，再来用它。就像你不会在还在学做菜的时候就去考食品安全证——先把菜做好，再谈规范化发布。

**什么时候用它：** 当你的 skill 准备从"个人使用"走向"团队共享"或"公开发布"时。

**什么时候不该依赖它：** 当你把它当成冷启动的唯一模板，以为"按 skill-forge 的规范从头写就对了"的时候。规范是后期保障，不是起步方式。

---

### `open-skills`

**一句话定位：** 本地 LLM 和自托管场景的运行桥接。

它解决的是一个特定场景的问题：如果你用的不是云端大模型，而是本地 LLM、MCP 或自托管环境，你需要一个桥接层来让 skill 在这些环境里跑起来。

**它的边界：** 它解决的是特定运行时的适配，不是通用基座。如果你用的是主流云端平台，这个对象对你现在可能不那么相关。

---

### `Ai-Agent-Skills`

**一句话定位：** 团队或个人的精选 skill 书架。

类似于你自己维护的一个"我觉得好用的 skill 收藏夹"，适合团队内部管理或个人长期维护。

**它的边界：** 它是个人或团队级别的管理方式，不能替代通用的 installer 和治理层。

---

## B2. 选择决策树

面对"我该从哪里开始"这个问题，比起翻表格，用决策路径走一遍更快：

**你现在最缺的是什么？**

→ **"我不知道外面有什么"** → 先用 `skills.sh` 和 `awesome-copilot` 扩视野。花半天时间浏览，建立对生态的基本感觉。

→ **"我知道有什么了，但不知道怎么写"** → 去 `vercel-labs/agent-skills` 学结构。按 [附录A](./附录A-代表性样本与阅读路径.md) 的方式拆几个样本。

→ **"我会写了，想搭自己的工作流"** → 组合起来用：用 `agent-skills` 做结构参考 + `vercel-labs/skills` 做受控安装 + 别忘了 trust gate 和对照评测。

→ **"我已经有一批 skill 在用了，想正式管起来"** → 加入 `skill-forge` 做治理 + 开始用失败分类和回放表做持续优化（见 [附录D](./附录D-持续优化闭环与评测操作.md)）。

→ **"我的运行环境比较特殊"** → 根据具体场景看 `open-skills`（本地 LLM / MCP）或 `Ai-Agent-Skills`（团队内部管理）。

## B3. 五条防误读规则

这些误读非常常见，值得专门列出来：

**"在 skills.sh 上找得到，所以应该靠谱。"** 不对。目录站解决的是发现，不是质量判定。就像菜市场里什么菜都有，但你不会因为"摊位上有"就觉得新鲜。

**"vercel-labs/skills 能装进去，说明效果没问题。"** 不对。installer 解决的是装载，不是评测。`npm install` 成功不代表包没 bug。装完之后仍然要做对照测试。

**"agent-skills 样本最多、结构最好，直接当基座就行。"** 不对。样板库强在学习，但它不负责安装、不负责治理、也不负责评测。把样板间当成精装房住会出问题。

**"skill-forge 方向最对，现在就 all in。"** 不对。治理很重要，但如果你还在学怎么写 skill，先去看样板。就像还在学做菜就去考食品安全证，顺序反了。

**"找得到 + 下载多 + 结构像样 = 可以上生产。"** 不对。发现、分发、结构质量和任务效果是四件不同的事。前三个都满足了，第四个仍然需要独立验证。

> 一句话总结：能找到不等于能信，能安装不等于有效，很适合学也不等于应该直接重押。

## B4. 读完这份附录，下一步做什么

把你现在最常用的 3 个 skill 相关对象写下来，用 B1 的框架给它们各写一句"一句话定位"和"它的边界"。如果写的时候发现自己分不清它的边界在哪，那说明你对它的理解可能还停留在表面——这恰恰是最值得花时间想清楚的地方。
