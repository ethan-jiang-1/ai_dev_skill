# 附录 D：持续优化闭环与评测操作

主文第 7-8 节讲了"为什么 skill 上线不等于成熟"和优化闭环的基本思路。那里为了控制概念密度，只展开了三种最常见的失败类型。这份附录把**完整的七类失败**逐个拆开，每类配一个具体场景和解决路径。

如果你现在只是想快速上手闭环，主文第 8 节就够了。这份附录更像一本"失败诊断手册"——当你遇到具体问题时，翻到对应类型，看看解决思路。

---

## D1. 为什么需要完整的失败分类

主文里用代码审查 skill 的故事讲了三种典型失败：触发没命中、步骤顺序反了、改完之后旧任务坏了。但在真实使用中，你会遇到的失败类型不止这三种。

把失败分类想象成医生的"症状清单"：你去看病，医生不会只说"你不舒服"——他会问是头疼、胃疼还是胸闷，因为不同症状指向不同的病因和治疗方案。skill 的失败也一样：如果你总是说"这个 skill 不太行"，你永远找不到真正需要改的地方。

下面是七种主要的失败类型。

---

## D2. 触发与可发现性失败

### 场景

你写了一个"部署检查" skill，`description` 写的是 "Use when deploying to production"。你的同事在做 staging 环境部署的时候，这个 skill 完全没反应——因为 description 里只提了 production，系统不觉得 staging 部署算。

另一种情况更烦人：你的 skill 太容易被触发。同事只是在讨论部署计划（还没真的部署），skill 就跳出来了，搞得像狼来了一样，最后大家干脆忽略它。

### 根因

这类问题不在 skill 的步骤或内容，而在**路由**——`description` 写得太窄或太宽。

### 怎么修

调整 `description` 的覆盖范围。太窄就加同义表达（"deploying to production or staging"），太宽就加限定条件（"Use when actively deploying, not when planning a deployment"）。改完后拿几个正面案例和反面案例各跑一遍，确认触发更准了且没有误触发。

---

## D3. 工作流可执行性失败

### 场景

你的代码审查 skill 触发了，但同事反馈"它让我先改代码再做检查——这不是反了吗？"你去看一下，发现步骤确实写反了：step 1 是建议修复，step 2 才是做检查。对于审查流程来说，应该先检查、后修复。

另一种常见情况：步骤本身写得模糊。"请检查代码质量"——但怎么检查？先看什么？看到什么算有问题？步骤不清楚，agent 执行起来就会自由发挥，每次走的路径都不一样。

### 根因

步骤的**顺序不对**或者**粒度太粗**。skill 能被触发，但触发后的流程经不起推敲。

### 怎么修

先理清正确的步骤顺序（对审查来说：先定性改动类型 → 先做检查 → 再给建议），然后确保每一步都清晰到"agent 不需要猜下一步做什么"的程度。模糊的步骤要么细化，要么拆成子步骤并指向支撑文件。

---

## D4. 工具使用契约失败

### 场景

你的 skill 里写了"使用 search 工具查找相关文件"。agent 确实调了 search 工具——但传的参数是整个问题描述，而不是你预期的文件名关键词。结果搜出来一堆不相关的东西，后续步骤全跑偏了。

还有一种情况：你期望 agent 按"先搜索、再读取、再分析"的顺序调用三个工具，但实际上它跳过了搜索，直接猜了一个文件名去读取。

### 根因

skill 的步骤里对**该调什么工具、怎么调、按什么顺序调**的描述不够明确，导致 agent 自己做了（不一定对的）判断。

### 怎么修

在涉及工具调用的步骤里，写清楚三件事：调哪个工具、参数应该是什么格式（或者至少给一个示例）、这一步和上下步的顺序关系。不要留给 agent "自行判断"的空间——至少在关键调用上不要留。

---

## D5. 结构与打包失败

### 场景

你的 skill 把所有内容都堆在了 `SKILL.md` 里——200 行正文，既有主线步骤，又有详细检查项，还有示例和参考链接。结果 agent 的上下文窗口被这一大坨文字占满了，真正需要关注的关键步骤反而被淹没了。

另一种情况：你拆了支撑文件，但导航写得不清楚。`SKILL.md` 里写了"详细检查项见相关文件"——哪个文件？在哪个目录？agent 找不到，于是当这一步不存在。

### 根因

**正文太胖**（什么都往里塞）或者**支撑文件的导航断了**（拆了但没连好）。

### 怎么修

正文只留主线步骤，通常 5-10 行就够。详细检查项、长说明、示例都下沉到 `references/` 或 `examples/`。每个下沉的内容在正文里要有明确的导航路径——写清文件名和相对路径，不要写"见相关文件"。

---

## D6. 安全与治理失败

### 场景

你从某个公开目录站找了一个 skill，看起来写得不错就装了。用了几天后发现，它的 `scripts/` 目录里有一个脚本会自动执行 `curl` 命令往外发请求。skill 本身的审查步骤看起来正常，但附带的脚本做了你不知道的事情。

### 根因

装之前没做 trust gate（信任审查），或者审查的时候**只看了 SKILL.md，没看 scripts/ 和支撑文件**。

### 怎么修

装之前读一遍所有会被带入上下文的文件——不只是 `SKILL.md`，还有 `scripts/`、`references/`、以及任何 skill 里提到的外部资源。把 skill 当成一种像代码一样需要 review 的资产。来源不明确的 skill，先在隔离环境里跑一遍。

---

## D7. 版本与回归失败

### 场景

你的代码审查 skill 跑得挺好，你决定"优化一下措辞"，改了两行 description。改完之后新任务上看起来好了一点点，但你之前验证通过的三个旧任务全坏了——触发条件变了，导致其中两个根本没被触发。

这就是典型的**回归**：改了新的，坏了旧的。

### 根因

改动之前**没有留 baseline**（基线版本），改完之后**没有用旧任务回放验证**，所以坏了也不知道。

### 怎么修

这正是主文第 8 节讲的最小闭环。关键动作是：改之前记下当前版本和它通过的任务列表；改完之后用同一批任务重跑一次；新的变好了但旧的变差了，先回退到上一个版本，不硬上。版本控制对 skill 和对代码一样重要。

---

## D8. 反馈回流失败

### 场景

你的 skill 在线上用了三个月，中间出了不少问题——有时候触发不准，有时候步骤不对，有时候同事给了反馈说"这步建议改一下"。但这些问题每次出现的时候，你记了一下就过去了。三个月后你想优化 skill，回忆这些问题发现自己什么都记不清了。

### 根因

线上出了问题，但**没有系统性地回流到线下**。每次改 skill 都是"我好像记得有个问题"，凭感觉改，改完也不知道有没有真的变好。

### 怎么修

建一个哪怕最简单的记录机制：每次发现 skill 有问题，记下来——什么任务、什么类型的失败、当时的表现是什么。不需要很复杂，一个文本文件或表格就行。这样你下次改 skill 的时候，手里有一批真实的失败案例可以拿来做对照测试，而不是对着空气修改。

就像医生的诊疗记录——不是为了写了好看，而是为了每次回诊的时候能看到上次的情况，判断"到底好了没有"。

---

## D9. 一张完整的最小回放表

如果你已经读完了上面七种失败类型，现在最实用的下一步是给你最常用的那个 skill 建一张表。不需要很大——5-10 行就够：

| 任务 | 失败类型 | 基线版结果 | 候选版结果 | 是否变好 | 下一步 |
| --- | --- | --- | --- | --- | --- |
| staging 部署检查 | 触发失败 | 没触发 | 正确触发 | 是 | promote |
| PR review（大改动） | 步骤失败 | 先改后查 | 先查后改 | 是 | promote |
| PR review（小修复） | 回归 | 原来通过 | 新版没触发 | 否 | fallback |
| 安全审查任务 | 工具失败 | 参数错 | 参数仍错 | 否 | iterate |

这张表的价值在于：它逼你把"感觉"变成"判断"，把"修了什么"和"到底有没有变好"连起来。凭感觉改和对照着改，差距会越来越大。

---

## D10. 最容易犯的错

**把所有失败都归因到正文措辞。** 很多失败其实是触发问题、结构问题或工具问题。如果你每次都"润色一下正文"，很可能修的不是真正坏掉的那层。

**每次同时改很多东西。** 一次改了 description、改了步骤、改了支撑文件——结果确实变好了，但你不知道是哪个改动起了作用。下次遇到类似问题，还是不知道该改什么。一次只改一类，比较才有意义。

**只看最终输出。** workflow skill 经常是过程先坏掉——步骤顺序错了、工具没调对——但最终输出碰巧看起来还行。就像考试碰巧答对了但解题过程全错，下次一定会出问题。要看中间步骤。

**线上踩坑不记录。** 没有记录，改 skill 就只能凭记忆和感觉。哪怕是最简陋的记录——一行文字写下"什么任务、什么类型、什么表现"——也比什么都不记好得多。

---

## D11. 读完这份附录，下一步做什么

给你最常用的一个 skill 做两件事：

第一，写一份失败分类草稿。回忆最近它出过的问题，试着按照上面七种类型归个类。如果某个问题不确定属于哪类，先标"不确定"——重要的是开始分类这个动作，而不是分得百分百准确。

第二，建一张最小回放表。留 5-10 个代表性任务，以后每次改 skill，就用这批任务做对照。你会发现，有了这张表之后，"改完了到底好了没有"这个问题不再是一种感觉，而是一个可以看到答案的比较。
