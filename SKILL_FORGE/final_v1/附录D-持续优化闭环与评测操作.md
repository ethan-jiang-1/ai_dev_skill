# 附录 D：持续优化闭环与评测操作

这份附录讲的是一件在很多 skill 讨论里经常被低估、但实际上决定长期质量的事情：

> skill 第一次能用，不代表它已经成熟。  
> 真正稳定的 skill，需要一套持续优化闭环。

如果主文的重点是“为什么 skill 上线不等于成熟”，这份附录讲的就是：

- skill 到底会在哪些地方坏
- 怎么把失败整理成可改的对象
- 怎么做最小回放
- 怎么决定继续改、先回退，还是晋级为常用版本

## D1. 先给结论

当前最稳的判断不是“有没有一个现成 skill optimizer（skill 优化器）”，而是：

> skill 的持续优化更像一套组合闭环，而不是单个工具。

这个闭环通常至少包括五层：

1. artifact governance（工件治理）
2. trigger / discoverability tuning（触发与可发现性调优）
3. trajectory regression（过程轨迹回归）
4. offline / online feedback loop（离线 / 在线反馈回流）
5. candidate revision + human promotion gate（候选修订 + 人工晋级门槛）

如果这五层完全缺席，skill 很容易停留在“能用一次”的阶段。[E22][E23]

## D2. 为什么 final answer pass 还不够

很多 workflow skill 的失败，并不是输在最后一句回答，而是输在过程里。

例如：

- 该触发时没触发
- 不该触发时误触发
- 中间步骤顺序乱了
- 工具调用漏了
- 工具参数错了
- 本来要三步，结果只走了一步

所以：

- `final answer pass（最终答案通过）`
  - 只说明最后表面看起来还行
- `workflow pass（工作流通过）`
  - 才说明关键步骤、调用顺序和边界没有跑偏

这就是为什么 skill 评测不能只看结果，不看过程。[E25][E26]

## D3. 先把失败分类，而不是一律说“效果不稳”

最常见的低效做法，是把所有问题都写成“这个 skill 最近不太行”。  
更有效的做法，是先建立最小 failure taxonomy（失败分类）。

建议的最低分类包括：

| 失败类型 | 它通常意味着什么 |
| --- | --- |
| trigger / discoverability failure | 该触发时没触发，或错误触发 |
| workflow executability failure | 步骤本身不清楚，导致流程走不下去 |
| tool-use contract failure | 工具调用方式、参数或顺序出了问题 |
| structural / packaging failure | 正文、支撑文件或结构组织有缺陷 |
| safety / governance failure | 权限边界、风险控制或安全审查不够 |
| versioning / regression failure | 改完新版本后把旧的好效果改坏了 |
| feedback loop failure | 线上失败没有回流到线下，改动变成凭感觉 |

只要失败分类更清楚，修订就更容易集中火力，而不是每次都重写一遍 skill。[E22][E23][E25]

## D4. skill 真正会在哪些层被优化

有些人一听“优化 skill”，就只想到改 prompt。  
但当前证据更支持把 skill 优化看成 artifact-level optimization（工件级优化）。[E22]

真实会被优化的层包括：

### 1. 触发与描述

- `description`
- 触发条件
- 路由边界

这层优化解决的是：

- 什么时候该进
- 什么时候不该进

### 2. 工作流与步骤

- 步骤是否清楚
- 顺序是否合理
- 是否少了前置检查
- 是否漏了失败分支

### 3. 工具契约

- 什么时候该调工具
- 调哪个工具
- 参数该怎么传
- 哪些工具不该乱用

### 4. 结构与支撑文件

- 正文是不是太胖
- 细节是否被正确下沉
- 支撑文件是不是可导航、可维护

### 5. 版本与评测

- 有没有 baseline（基线）
- 有没有代表性任务
- 改完后有没有比较
- 出问题时能不能退回去

如果你把 skill 优化只理解成“把正文措辞改得更顺”，那你实际上漏掉了最重要的一半。[E22][E25][E27]

## D5. 一个最小可执行的优化闭环

你不需要一上来就搭一个很大的系统。  
一个最小可执行闭环就已经很有用。

### Step 1：留下代表性任务

先准备一小组任务，最好是：

- 高频任务
- 曾经失败过的任务
- 容易看出差异的任务

建议起步数量：

- `5-10` 个就够

### Step 2：每次只修一类失败

不要一口气同时改：

- 触发
- 步骤
- 工具
- 结构

一次只针对一类问题做候选修订，比较才有意义。

### Step 3：做 baseline vs candidate compare

`baseline vs candidate compare（基线版对候选版比较）` 指的是：

- 基线版先跑一次
- 候选版再跑一次
- 对比两者在哪些任务上变好，哪些任务上变差

这一步会明显降低“我感觉应该更好了”的自我错觉。[E27][E28]

### Step 4：记录过程，不只记录最后一句

至少记：

- 是否触发
- 中间调用了什么
- 参数有没有异常
- 步骤顺序是否跑偏
- 最终结果是不是改善

### Step 5：决定 promote、fallback 或继续修改

最稳的动作不是“看起来差不多就上线”，而是明确分三种结果：

- `promote`
  - 候选版明显更稳，可以晋级
- `fallback`
  - 候选版把旧效果改坏了，先回退
- `iterate`
  - 候选版局部有改善，但还不够稳，继续修

## D6. trajectory regression 到底在检查什么

`trajectory regression（轨迹回归）` 这个词听起来很工程，但意思其实很朴素：

> 每次改 skill 后，不只看最后输出，还要看执行过程有没有偏。

它最少可以检查下面这些维度：

- trigger / no-trigger（是否触发 / 是否不该触发）
- tool calls（调用了哪些工具）
- tool args（参数是否合理）
- tool sequence（调用顺序是否合理）
- step count（步骤数量是否异常）
- output contract（输出是不是还符合约定）
- safety boundary（有没有越过风险边界）

只要 workflow 里出现工具调用或多步操作，这套检查就会非常值钱。[E25]

## D7. online / offline feedback loop 要怎么理解

`online feedback（在线反馈）` 可以理解成真实使用现场暴露的问题。  
`offline feedback（离线反馈）` 则是把这些问题整理成可以重复回放和比较的样本。

更稳的闭环是：

1. 在线任务暴露问题。
2. 用户修正或人工审阅留下线索。
3. 这些线索进入 annotation queue（标注队列）或 case list（案例列表）。
4. 逐步整理成可复用的离线样本。
5. 后续每次修改 skill，都能拿这批样本回放。

这一步的核心意义，是把“线上踩坑”变成“以后不再重复踩同一个坑”。[E27]

## D8. candidate revision 为什么不能替代人工门槛

`candidate revision（候选修订）` 的直觉是：

- 先让系统或作者生成一个可能更好的版本
- 再拿这个版本去比较

这一步很有用，但它的边界也要讲清。

更适合自动候选修订的内容包括：

- `description`
- 示例
- 步骤重排
- 输出契约说明
- 支撑文件组织

不适合直接放权给自动化的，是：

- 最终晋级决定
- 风险边界放宽
- 生产级默认替换

因为自动候选修订最擅长的是“提出方案”，不是“替你承担后果”。  
所以真正稳的体系里，human promotion gate（人工晋级门槛）仍然必须存在。[E27][E28]

## D9. 一份最好用的最小回放表

如果你现在就想把这套方法用起来，最简单的做法是先建一张表：

| 任务 | 失败类型 | 基线版结果 | 候选版结果 | 是否变好 | 下一步动作 |
| --- | --- | --- | --- | --- | --- |
| 任务 A | trigger failure | 没触发 | 正确触发 | 是 | promote |
| 任务 B | tool-use failure | 参数错 | 参数仍错 | 否 | iterate |
| 任务 C | regression failure | 原来通过 | 新版失败 | 否 | fallback |

这张表有两个价值：

- 它逼你把感觉变成判断
- 它逼你把“修了什么”与“到底有没有更好”连起来

## D10. 最容易犯的 5 个错

### 1. 把所有失败都归因到 prompt

很多失败其实是触发、结构、工具契约或版本问题，不是正文一句话的问题。

### 2. 每次同时改很多层

一旦同时改太多，后面很难判断到底是哪一层带来了改善或退化。

### 3. 只看最终答案

workflow skill 经常是过程先坏掉，只看最后结果会漏很多问题。

### 4. 线上踩坑却不回流

没有回流，下一轮修订就只能继续靠感觉。

### 5. 把自动候选修订当自动晋级

自动化可以帮你找候选，不该直接替你做最终门槛判断。

## D11. 如果你现在就想起步，最推荐的顺序

最小起步顺序如下：

1. 选一个高频 skill。
2. 留下 `5-10` 个代表性任务。
3. 给最近失败分类型。
4. 一次只修一类问题。
5. 跑基线版和候选版比较。
6. 记录结果。
7. 决定 promote、fallback 或继续改。

只要你真做了这 7 步，你的 skill 质量就已经开始从“凭感觉维护”迈向“可比较、可回退、可进化”。

## D12. 读完这份附录后，下一步做什么

立刻给你最常用的一个 skill 新建两样东西：

- 一份 failure taxonomy 草稿
- 一张最小回放表

很多时候，你不缺新的工具，缺的是把失败组织起来的方式。
