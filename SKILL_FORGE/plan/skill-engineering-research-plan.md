# 面向 Coding Agent 的 Skill 工程化研究 Progress Plan

## 0. 计划定位

这不是一份泛泛的研究说明，而是一份可以持续推进、持续更新状态的 progress plan。

核心目标不是“多看几个 skill 仓库”，而是分阶段回答下面的问题：

- 社区里是否已经存在比较成熟的 skill engineering 方法论
- 是否有适合 coding agent 的开源工具链，能降低 skill 打磨、复用、发布成本
- 哪些项目只是 skill 内容集合，哪些项目已经具备工程化流水线能力
- 最终能否筛出 1 到 3 个最值得持续跟踪或直接采用的基座项目

## 1. 总体交付目标

最终交付物不是单篇笔记，而是四个结果：

- 一份候选对象池
- 一份横向对比表
- 一份前 3 名推荐结论
- 一份“如何落到自己的 skill workflow”建议

## 2. Progress Tracking 机制

### 2.1 状态定义

每个阶段统一使用下面四种状态：

- `not_started`：尚未开始
- `in_progress`：已启动，正在收集或整理
- `blocked`：因信息不足、链接失效或判断标准不清而暂停
- `done`：该阶段的验收条件已经满足

### 2.2 更新频率

每完成一个阶段，必须更新一次进度。

每次更新至少补四类信息：

- 当前阶段状态
- 已完成内容
- 新发现的问题或偏差
- 下一步动作

### 2.3 更新格式

后续每次推进时，都按这个格式往下追加：

```md
## Progress Update N

- 日期：
- 当前阶段：
- 状态：
- 已完成：
- 新发现：
- 风险 / 阻塞：
- 下一步：
```

## 3. 执行原则

- 不把“skill 内容仓库”和“skill 工程化工具链”混为一谈
- 不只读 GitHub README，要看社区收录、引用、采用信号
- 不追求一次找全，而是优先筛出最有可能成为基座的项目
- 不只做描述性总结，每一步都要有验收条件
- 不让进度停留在脑内，每个阶段结束必须写明状态更新

## 4. 分阶段执行步骤

### P1. 建立候选池

**目的**

把研究对象从“凭感觉盲找”收敛成结构化候选池，先建立生态地图。

**执行内容**

- 按四类收集对象：方法论 / pipeline、运行时 / 装载层、官方 skill 样板、社区目录 / 市场 / registry
- 每类至少记录 2 到 5 个候选对象
- 给每个候选对象补最基本的识别信息：
  - 名称
  - GitHub 地址
  - 类型
  - 解决的问题
  - 初步印象

**执行细节**

- 先从当前已知对象出发：
  - skill-forge
  - OpenSkills / open-skills
  - vercel-labs/agent-skills
  - supabase/agent-skills
  - awesome-agent-skills
  - skills.sh / SkillsMP 一类生态入口
- 再补充外部同类项，避免样本偏差
- 如果某对象只是内容仓库，不直接淘汰，但要标记为“样板型”而非“工具链型”

**验收手段**

- 候选对象不少于 10 个
- 每个对象都被归到明确类别
- 至少识别出 3 个“可能成为工程化基座”的候选项

**完成标志**

- 形成候选池表格
- 可以回答“现在我们看的是哪些对象，为什么看它们”

### P2. 工程能力拆解

**目的**

把候选对象从“知道名字”推进到“知道它到底解决 skill 生命周期哪一段”。

**执行内容**

对重点候选对象统一拆解以下维度：

- 它解决的问题是什么
- 它面向谁：个人作者、团队、平台、社区
- 它覆盖 skill 生命周期的哪一段：
  - 编写
  - 组织
  - 触发
  - 装载
  - 治理
  - 发布
  - 分发
  - 评估
- 它本质上是内容库、加载器、审计器、分发工具，还是 pipeline
- 它是否真的降低 skill 打磨成本

**执行细节**

- 优先拆重点对象，不要求一开始对所有候选都深拆
- 第一批重点对象建议固定为：
  - skill-forge
  - OpenSkills / open-skills
  - vercel-labs/agent-skills
  - supabase/agent-skills
  - awesome-agent-skills 或 skills.sh / SkillsMP
- 拆解结果用统一模板记录，避免后面难以比较

**验收手段**

- 每个重点对象都有同一套字段的分析记录
- 能明确区分“工程化工具”“运行时兼容层”“高质量样板”“社区导航”
- 至少能指出每个对象最强的一段能力和最弱的一段能力

**完成标志**

- 有一版可横向比较的能力拆解表
- 可以回答“这个项目到底在 skill 工程化链路里扮演什么角色”

### P3. 社区信号验证

**目的**

避免只看项目自述，补上社区评价、外部引用、采用情况，判断其真实可信度。

**执行内容**

- 查是否被官方团队或知名项目采用
- 查是否被收录到 curated list
- 查是否有外部文章、讨论、教程、二次介绍
- 查是否有跨平台兼容信号
- 查是否存在“看起来很强但社区几乎没有回声”的情况

**执行细节**

- 不要求所有对象都有大量讨论，但重点对象必须有基本外部信号
- 如果外部信号不足，要明确标记“社区验证不足”
- 对官方维护仓库和个人实验仓库分开判断，不混用标准

**验收手段**

- 每个重点对象至少有 2 条以上外部信号或可信采用线索
- 能区分“生态位置真实”和“只是看上去完整”
- 能标出哪些项目值得继续深挖，哪些只值得参考

**完成标志**

- 有一版社区信号记录
- 可以回答“为什么这个项目不只是自己说自己好”

### P4. 横向对比与排序

**目的**

把前面收集的材料收敛成选择，而不是无限制继续浏览。

**执行内容**

- 按统一维度打对比：
  - 工程化成熟度
  - 对 coding agent 的适配度
  - 降低打磨成本能力
  - 社区可信度
  - 可迁移性
- 对重点对象进行梯队划分：
  - 第一梯队：值得深入研究 / 直接试用
  - 第二梯队：值得借鉴结构
  - 第三梯队：更多提供生态信号

**执行细节**

- 不追求“绝对分数”，重点是排序逻辑是否能解释清楚
- 如果出现能力互补关系，要明确写出来
- 允许得出“不是单一项目取胜，而是组合采用”的结论

**验收手段**

- 至少形成一张横向对比表
- 至少给出前 3 名或前 3 组合
- 对每个入选对象都能说清：为什么入选、为什么不是别人

**完成标志**

- 形成排序结果
- 可以回答“当前最值得继续投入研究的是哪几个”

### P5. 推荐结论与落地建议

**目的**

把研究结果落到你的真实目标上：不是看热闹，而是找到“怎么帮 coding agent 更容易打磨 skill”的实际路径。

**执行内容**

- 给出三类推荐：
  - 研究母体
  - 实际使用工具
  - 结构样板
- 判断是否需要组合方案，例如：
  - 一个项目做工程治理
  - 一个项目做运行时装载
  - 一个官方样板做结构参照
- 输出一个最小 baseline 建议：
  - skill 怎么组织
  - 怎么评审
  - 怎么跨 agent 装载
  - 怎么更新和验证

**执行细节**

- 推荐结论必须和前面阶段证据直接对应
- 不给空泛建议，要落到“可以下一步试什么”
- 如果没有绝对赢家，也要明确最合理的组合路径

**验收手段**

- 能明确推荐前 3 名或前 3 组合
- 能明确回答“如果现在就开始做，第一步该采用什么”
- 能形成一版面向你自己项目的 baseline 建议

**完成标志**

- 有最终建议稿
- 可以直接进入试用、复刻或二次设计阶段

## 5. 研究范围与候选方向

### 5.1 方法论 / Pipeline 类

重点观察是否具备：

- 结构标准
- 审计与校验
- 安全扫描
- README / 描述一致性检查
- 发布与分发
- 多平台安装

候选方向：

- skill-forge

### 5.2 运行时 / 装载层 类

重点观察是否具备：

- Claude Code skill 兼容
- 通用 skill loader
- AGENTS.md / SKILL.md 桥接
- 按需加载
- 跨 agent 兼容

候选方向：

- OpenSkills
- open-skills

### 5.3 官方 skill 样板类

重点观察：

- skill 的结构是否稳定
- 触发词和 use when 是否清晰
- references / scripts / resources 是否组织良好
- 是否体现官方最佳实践

候选方向：

- vercel-labs/agent-skills
- supabase/agent-skills

### 5.4 社区目录 / 市场 / 榜单 类

重点观察：

- 是否有 curated 能力
- 是否有生态聚合价值
- 是否能看到 adoption 信号
- 是否能反推社区偏好的设计模式

候选方向：

- awesome-agent-skills
- skills.sh
- SkillsMP

## 6. 统一评估标准

### 6.1 工程化成熟度

- 是否有明确结构规范
- 是否有校验、审计、修复机制
- 是否考虑发布与分发
- 是否考虑版本化和复用

### 6.2 对 Coding Agent 的适配度

- 是否专门面向 coding agent
- 是否兼容多个 agent
- 是否支持按需加载而不是一次塞满上下文
- 是否能提升触发准确性和执行一致性

### 6.3 降低打磨成本的能力

- 是否减少人工整理工作
- 是否让 skill 更易复用
- 是否让 skill 更容易被 agent 正确调用
- 是否让 skill 从草稿更快走向可交付

### 6.4 社区可信度

- 是否有清晰维护者
- 是否有被引用、被收录、被复用的证据
- 是否有真实生态位置，而不是孤立 demo

### 6.5 可迁移性

- 是否能迁移到你自己的 workflow
- 是否能和现有 coding agent 协同
- 是否适合作为后续 skill engineering baseline 的底座

## 7. 当前初步判断

基于目前线索，先保留一个工作假设：

- **skill-forge** 更像工程治理与发布流水线
- **OpenSkills / open-skills** 更像运行时兼容层与装载层
- **vercel-labs/agent-skills / supabase/agent-skills** 更像高质量结构样板
- **awesome-agent-skills / skills.sh / SkillsMP** 更像社区导航与信号源

这意味着最终答案很可能不是一个单体项目，而是一个组合：

- 用方法论项目看“怎么工程化”
- 用装载层项目看“怎么跨 agent 真正跑起来”
- 用官方样板看“好 skill 应该怎么组织”
- 用社区入口看“哪些东西真的被采用”
