# Agent Skills 总览

## 这套东西到底是什么

`agent-skills` 不是单纯的提示词合集，而是一套把软件工程流程模块化的技能系统。它把 AI agent 在真实工程里应该遵守的行为，拆成可以组合的 skill。

它的核心目标不是让 agent “更会写代码”，而是让 agent：

- 少跳步骤
- 少靠猜
- 少过度实现
- 少把没验证的结果当完成
- 更像一个有流程纪律的资深工程师

## 生命周期结构

整个仓库按工程生命周期分层：

- Define：先定义问题和边界
- Plan：把工作拆成可执行单元
- Build：开始实现，但要求上下文、接口、UI、测试都有规范
- Verify：出现问题时先排障，浏览器问题要看真实运行态
- Review：在合并前做代码质量、安全、性能、简化检查
- Ship：把提交、CI/CD、迁移、文档、上线纳入发布纪律

## 7 个命令入口

- `/spec`：先写规格，再开始做
- `/plan`：把规格拆成任务
- `/build`：按切片实现
- `/test`：证明结果是真的
- `/review`：在合并前压质量
- `/code-simplify`：压复杂度
- `/ship`：准备发布

## 我对这套设计的总体理解

### 它最强的地方

- 把很多“老工程师默认知道”的隐性规则变成显性流程
- 适合 AI agent，因为 AI 最容易在流程纪律上失守
- skill 之间能串联成闭环，不是孤立技巧
- verification 被设计成必须交证据，而不是主观感受

### 它的真实代价

- 流程变多
- 文档变多
- 需要团队维护规则和基础设施
- 小任务上会显得偏重

## 我建议怎么使用

### 小任务

- `using-agent-skills`
- `incremental-implementation`
- `test-driven-development`
- `code-review-and-quality`

### 中等功能

- `spec-driven-development`
- `planning-and-task-breakdown`
- `context-engineering`
- `incremental-implementation`
- `test-driven-development`
- `code-review-and-quality`

### 高风险上线功能

- 上面所有核心 skill
- 加上 `security-and-hardening`
- 加上 `performance-optimization`
- 加上 `ci-cd-and-automation`
- 加上 `shipping-and-launch`

## 适合引入这套 skill 的团队

- 有正式代码库，不是一次性脚本
- 会做 code review
- 愿意维护测试和 CI
- 希望 AI 参与真实工程而不是只做原型

## 不适合的场景

- 单次快速实验
- 没有验证基础设施
- 不希望增加流程负担
- 主要诉求只是“赶紧产出代码”
