# superpowers 结构拆解

## 先说结构结论

superpowers 的结构不是“skills/ 一个目录就完事”，  
而是四层相互咬合：

- skill 入口层
- bootstrap 与宿主适配层
- 支撑 prompt / reference 层
- 测试与验证层

它的强项不在代码体量，  
而在这些层共同把 agent 推向固定流程。

## 1. skill 入口层

### 主要路径

- `skills/*/SKILL.md`

当前共有 14 个显式 skill。

### 结构特征

从 `brainstorming`、`writing-plans`、`systematic-debugging`、`using-superpowers` 看，  
这一层普遍包含：

- YAML front matter
- `name`
- `description`
- 强流程正文
- checklist / phase / hard gate / red flags

### 这一层的作用

- 向宿主暴露能力入口
- 用 `description` 定义触发条件
- 用正文定义执行纪律

这一层本质上是行为代码，  
不是普通文档。

## 2. bootstrap 与宿主适配层

### A. `using-superpowers`

路径：

- `skills/using-superpowers/SKILL.md`

这一层的职责是：

- 要求任何任务先检查 skill
- 规定 Skill tool 的使用方式
- 给不同宿主做工具映射
- 把“有 1% 概率相关也必须查 skill”的纪律写死

它本质上是整个系统的总开关。

### B. hook 注入层

路径：

- `hooks/hooks.json`
- `hooks/session-start`

这层负责：

- 在 SessionStart 时运行 hook
- 读取 `using-superpowers`
- 把完整内容注入到会话上下文
- 根据 Claude / Cursor / Copilot CLI 输出不同字段

这说明 superpowers 不是等 agent 自己想起来用 skill，  
而是主动把“如何用 skill”灌进 session。

### C. 多宿主分发层

路径：

- `.claude-plugin/`
- `.cursor-plugin/`
- `.codex/`
- `.opencode/`
- `gemini-extension.json`
- `docs/README.codex.md`
- `docs/README.opencode.md`

这层说明仓库真正的分发单位是：

- 同一套方法论
- 多个宿主入口

而不是“只给某个工具写的私有 prompt”。

## 3. 支撑 prompt / reference 层

### 代表路径

- `skills/brainstorming/visual-companion.md`
- `skills/brainstorming/spec-document-reviewer-prompt.md`
- `skills/requesting-code-review/code-reviewer.md`
- `skills/subagent-driven-development/implementer-prompt.md`
- `skills/subagent-driven-development/spec-reviewer-prompt.md`
- `skills/subagent-driven-development/code-quality-reviewer-prompt.md`
- `skills/systematic-debugging/root-cause-tracing.md`
- `skills/test-driven-development/testing-anti-patterns.md`

### 这层的意义

它说明 superpowers 不是“一个 skill = 一个文件”。  
复杂 skill 已经拆成：

- 主入口 skill
- 子角色 prompt
- reference
- 反模式库
- 专项说明

这是一种轻量模块化。

### 和更重型系统的区别

它没有像 gstack 那样配大量 runtime 服务，  
但通过这些支撑文件，已经把 skill 从单文件提示词推进到技能包。

## 4. 兼容命令层

路径：

- `commands/brainstorm.md`
- `commands/write-plan.md`
- `commands/execute-plan.md`

这层的信号很明确：

- 老入口还在
- 但已经被标成 deprecated
- 用户被引导转到新 skill 体系

这说明仓库在演进过程中很注意迁移成本。

## 5. 测试与验证层

### 主要路径

- `docs/testing.md`
- `tests/claude-code/`
- `tests/skill-triggering/`
- `tests/explicit-skill-requests/`
- `tests/opencode/`
- `tests/subagent-driven-dev/`
- `tests/brainstorm-server/`

### 这一层在验证什么

从命名和测试文档看，重点不是传统函数测试，  
而是：

- skill 是否被触发
- 触发顺序是否正确
- 子 agent 是否被调度
- TodoWrite 是否被使用
- 计划是否只读一次
- OpenCode 插件是否能加载

这意味着验证对象是 **agent 行为链**。

## 6. 结构上的真正重点

### 重点一：以 bootstrap 为中心

很多 skill 仓库从 skill 本身开始看。  
superpowers 更适合从：

- `using-superpowers`
- hook 注入

开始看。

因为这里决定了其他 skill 有没有机会被用起来。

### 重点二：以流程链为骨架

它的主线不是按领域分很多 skill，  
而是围绕软件开发生命周期组织：

- brainstorm
- write plan
- execute
- review
- finish

### 重点三：以测试层兜底

如果没有测试层，  
这类强纪律 skill 很容易沦为写得很好看的说明书。

superpowers 的结构专门留出测试层，  
说明它知道“行为 shaping”必须被验证。

## 结构拆解后的判断

superpowers 的结构非常像一个：

- 内容层负责规则
- hook 层负责注入
- 平台层负责分发
- 测试层负责校验

的技能插件系统。

它不重在 runtime 执行器，  
重在 **让宿主持续按一套流程纪律工作**。
