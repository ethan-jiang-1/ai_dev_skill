# superpowers 优缺点

## 总体判断

superpowers 的最大优点，是它非常明确地在解决一个真实问题：

- agent 明明有技能，却经常不按流程工作

它的最大缺点，也和这个方向直接相关：

- 一旦纪律过强，就会带来厚重感、宿主依赖和体验分化

## 主要优点

## 1. 流程骨架非常清晰

superpowers 不是把 skill 平铺成一个目录让你自己拼。  
它给出了一条很明确的主链：

- brainstorm
- worktree
- write plan
- execute
- TDD
- review
- finish

这使它非常适合需要完整工程节奏的任务。

## 2. 先解决“skill 不被使用”的问题

很多 skill 仓库默认 agent 会记得用技能，  
但现实里经常不是这样。

superpowers 通过：

- `using-superpowers`
- SessionStart hook
- 强制 skill 调用语言

优先解决“入口纪律”问题。  
这一点很有针对性。

## 3. 对 agent 常见失败模式理解很深

像这些失败模式，它都在显式对抗：

- 跳过设计
- 跳过 root cause 分析
- 跳过 failing test
- 先行动再合理化
- 任务做着做着偏题

这类失败模式覆盖，使 skill 不是泛泛建议，而是有实战针对性。

## 4. 测试意识很强

`docs/testing.md` 和 `tests/` 结构说明它不是“写完 skill 就算了”。  
它已经在验证：

- skill 是否触发
- 执行顺序是否正确
- 子 agent 是否真的被调度
- 多宿主插件是否能加载

这对 skill 仓库来说非常加分。

## 5. 多宿主适配做得成熟

当前可以看到明确适配：

- Claude Code
- Cursor
- Codex
- OpenCode
- Gemini
- Copilot CLI

这让 superpowers 的方法论传播范围很广，  
也降低了它被单一平台锁死的风险。

## 6. 复杂 skill 已经具备“技能包”形态

像 `subagent-driven-development`、`systematic-debugging`、`brainstorming`，  
都已经不止一个入口文件，而是带：

- reviewer prompt
- anti-pattern reference
- visual companion
- tracing note

这比单文件 skill 更有深度。

## 主要缺点

## 1. 强纪律天然会带来“重”

对一些用户来说，  
它的主流程会显得太完整、太严格。

比如：

- 很小的改动
- 临时实验
- 一次性脚本
- 只想快速试错的场景

这种方法可能会让人觉得流程负担过高。

## 2. 效果高度依赖宿主是否真的遵守

skill 写得再好，  
如果宿主：

- 不稳定地发现 skill
- 不稳定地执行 hook
- 对 tool mapping 支持不一致

那实际效果就会打折。

也就是说，它的一部分质量在仓库里，  
另一部分质量在宿主生态里。

## 3. 行为塑形主要靠文本纪律

虽然 superpowers 有测试和 hook，  
但它大量核心逻辑仍然来自：

- 强硬文案
- Red Flags
- Hard Gate
- 明确的 routing 叙述

这类设计很强，但也意味着：

- 改 wording 可能影响行为
- 版本演进需要持续测
- 维护者对文本细节非常敏感

## 4. runtime 执行力不如更重型系统

和带浏览器 daemon、设计 runtime 的仓库相比，  
superpowers 更偏流程编排。

这带来一个边界：

- 它很擅长约束 agent 怎么工作
- 但不靠大量底层执行器去扩展能力面

所以它是“方法论插件”更强，  
不是“执行基础设施”更强。

## 5. 跨宿主适配会增加长期维护成本

多宿主是优点，  
也是一个长期包袱。

因为后续会不断遇到：

- 工具名不一致
- 子 agent 能力不一致
- hook 机制不一致
- 插件市场与安装方式不一致

这些都会持续拉高维护成本。

## 6. 某些 skill 对初学者可能过于强势

对已经认同 TDD、系统化调试、细粒度计划的人来说，  
这些 skill 很有吸引力。

但对只想让 agent 快速出结果的人，  
`MUST`、`ALWAYS`、`DO NOT` 这类语气可能会显得压迫感很强。

## 适用边界

## 最适合

- 想把 agent 用成工程搭档的人
- 愿意接受流程纪律的开发者
- 需要跨多个宿主复用同一工作流的人
- 关注 TDD、调试、review、计划质量的人

## 一般适合

- 想研究 skill 设计和行为塑形的人
- 想学习如何测试 agent workflow 的人
- 想给自己团队做技能插件的人

## 不太适合

- 只想收集 prompt 句子的用户
- 极度偏好自由发挥、弱流程的用户
- 更在意 runtime 执行器而不是流程治理的用户

## 最后一行总结

superpowers 最大的优点，是它在认真把 agent 变成“守纪律的开发者”。  
superpowers 最大的缺点，是要做到这一点，它就必须比很多轻量 skill 仓库更强势、更重、更依赖宿主配合。
