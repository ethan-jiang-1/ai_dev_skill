# superpowers 评分总表

## 评分说明

这一份先用主体分析框架的 1-5 分做横向盘点：

- 1 = 很低
- 3 = 中等
- 5 = 很高

它的作用不是下最终结论，  
而是把 superpowers 的几类关键 skill-like 单元拆开比较。

## 评估对象切分

我把 superpowers 拆成五类对象：

- 顶层 skill 入口系统
- `using-superpowers` 与 bootstrap 系统
- 流程型核心 skill 链
- 多宿主适配系统
- 测试与验证系统

## 分项解读

## 1. 顶层 skill 入口系统

### 评分

- 结构化程度：5
- 可验证性：4
- 复用性：4
- 工具依赖：2
- 协作价值：5
- 可维护性：4
- 学习成本：3
- 落地门槛：2

### 为什么这样打

这一层最稳定、最清晰：

- `SKILL.md` 结构统一
- `description` 明确写触发条件
- skill 命名直接表达职责

对读者来说，上手难度不高；  
对系统来说，它也是最基础的能力地图。

## 2. `using-superpowers` 与 bootstrap 系统

### 评分

- 结构化程度：5
- 可验证性：4
- 复用性：5
- 工具依赖：3
- 协作价值：5
- 可维护性：4
- 学习成本：4
- 落地门槛：3

### 为什么这样打

这一层最能体现 superpowers 的独特性。  
它不是加一个 skill，而是先定义“skill 应该怎么被使用”。

之所以学习成本偏高，  
是因为它涉及 hook、宿主、工具映射和行为规范。

## 3. 流程型核心 skill 链

### 评分

- 结构化程度：5
- 可验证性：4
- 复用性：5
- 工具依赖：3
- 协作价值：5
- 可维护性：4
- 学习成本：4
- 落地门槛：3

### 为什么这样打

这层是 superpowers 的方法论主体：

- 强设计前置
- 强计划前置
- 强执行纪律
- 强 review 和收尾

它非常适合真实工程流程，  
但对轻任务用户会显得偏重。

## 4. 多宿主适配系统

### 评分

- 结构化程度：4
- 可验证性：4
- 复用性：5
- 工具依赖：4
- 协作价值：4
- 可维护性：3
- 学习成本：4
- 落地门槛：4

### 为什么这样打

它的扩散能力很强，  
但这一层的成本主要体现在：

- 各宿主工具映射不同
- hook 机制不同
- 子 agent 能力也不同

所以复用性高，但维护性不算特别高。

## 5. 测试与验证系统

### 评分

- 结构化程度：4
- 可验证性：5
- 复用性：4
- 工具依赖：4
- 协作价值：4
- 可维护性：4
- 学习成本：4
- 落地门槛：4

### 为什么这样打

这一层非常关键。  
如果没有它，superpowers 会更像“写得很强的 prompt 仓库”。

而有了行为测试之后，  
它才更像一个能被持续验证的 agent workflow system。

## 横向比较后的结论

### 最强的两层

- `using-superpowers` 与 bootstrap 系统
- 流程型核心 skill 链

原因是：

- 一个解决“技能如何被稳定启用”
- 一个解决“技能启用后如何推动完整开发流程”

### 最重要的护城河

- 测试与验证系统

因为它让 superpowers 不只是方法论，  
而是开始具备可验证性。

### 最容易变重的一层

- 多宿主适配系统

因为它最容易随着平台演化不断增加维护负担。

## 主体框架结论

如果只看主体 1-5 分框架，  
superpowers 明显高于普通 skill 仓库。

它最突出的不是某一个工具能力，  
而是：

- 行为入口治理
- 流程链设计
- 行为测试

最终仓库级判断，还要放到全量评估里用 1-10 六维结构收口。
