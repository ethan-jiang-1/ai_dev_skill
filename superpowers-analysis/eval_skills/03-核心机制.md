# superpowers 核心机制

## 先抓三条主机制

如果只挑最核心的机制，我会抓这三条：

- `using-superpowers` 的强制技能检查机制
- brainstorming -> writing-plans -> subagent-driven-development 的主流水线
- 用真实会话测试验证技能执行的行为测试机制

这三条一起，构成 superpowers 的核心。

## 1. 强制技能检查机制

### 机制载体

- `skills/using-superpowers/SKILL.md`
- `hooks/session-start`
- `hooks/hooks.json`

### 它在做什么

这套机制不只是说“你可以用技能”，  
而是在 session 一开始就把下面这些规则压进去：

- 任何任务先想 skill 是否相关
- 即使只有 1% 概率也必须调用 skill
- 不允许先读代码、先问问题、先随手行动再补 skill
- process skill 优先于 implementation skill

### 为什么这机制重要

因为很多技能仓库最大的问题不是 skill 不够多，  
而是 agent 根本不会稳定地用它们。

superpowers 先解决的就是这个问题：

- 先让 agent 记住“必须查 skill”
- 再让 skill 去塑造具体行为

### 它的工程价值

这一步把“技能库”提升成“行为入口系统”。

## 2. 主流水线机制

### 主线 skill

- `brainstorming`
- `using-git-worktrees`
- `writing-plans`
- `subagent-driven-development`
- `executing-plans`
- `test-driven-development`
- `requesting-code-review`
- `finishing-a-development-branch`

### 它如何连接

从 README 和几个关键 skill 可以看出，这些 skill 不是并列卡片，  
而是一个阶段链：

- brainstorm 明确需求和设计
- worktree 隔离开发环境
- writing-plans 把需求拆成极细步骤
- subagent-driven-development / executing-plans 执行计划
- TDD 约束每个实现步骤
- code review 做阶段检查
- finishing skill 收尾、合并或丢弃分支

### 这条链的关键特征

- 每一步都有明确前置条件
- 很多 skill 会禁止你跳步
- 很多 skill 会指定下一个 skill 是谁

这让它不是“建议型方法论”，  
而是带 routing 的流程系统。

## 3. Hard Gate 与 Red Flags 机制

### 在哪些 skill 里特别明显

- `brainstorming`
- `systematic-debugging`
- `test-driven-development`

### 这机制在做什么

它把 agent 常见的失败模式写成禁止事项，比如：

- 没设计就开始写代码
- 没找 root cause 就开始修 bug
- 没有 failing test 就直接改实现
- 觉得事情简单，所以想跳过流程

### 为什么有效

因为它直接对准的是 LLM 常见弱点：

- 过早行动
- 合理化跳步
- 为了显得高效而偷流程
- 出现局部成功就误判整体成功

superpowers skill 的很多语言之所以“强硬”，  
其实是在跟这些失败模式对抗。

## 4. 子 agent 执行与双重审查机制

### 主要载体

- `skills/subagent-driven-development/SKILL.md`
- `skills/subagent-driven-development/implementer-prompt.md`
- `skills/subagent-driven-development/spec-reviewer-prompt.md`
- `skills/subagent-driven-development/code-quality-reviewer-prompt.md`

### 核心思路

执行计划时，不让一个上下文无限膨胀，  
而是：

- 每个任务派 fresh subagent
- 先做 spec compliance review
- 再做 code quality review
- 不通过就回环修复

### 这机制的价值

- 用隔离上下文减少串味
- 用两阶段 review 降低“实现了但偏题”或“正确但代码差”的风险
- 让主会话更像协调者，而不是一个人做完所有事

## 5. 行为级测试机制

### 主要载体

- `docs/testing.md`
- `tests/claude-code/`
- `tests/skill-triggering/`
- `tests/explicit-skill-requests/`
- `tests/opencode/`

### 它在测什么

这套测试不只测文件输出，  
更测：

- Skill tool 是否被调用
- Task 是否被调用
- TodoWrite 是否被使用
- review 顺序是否正确
- 计划是否只读一次
- 不同宿主插件是否能加载

### 这一点为什么很值钱

因为对 skill 仓库来说，真正重要的是：

- agent 有没有按规则做

而不是：

- 某个辅助脚本返回了什么文本

superpowers 已经开始把“agent 行为”当一等测试对象。

## 6. 多宿主工具映射机制

### 主要载体

- `docs/README.codex.md`
- `docs/README.opencode.md`
- `skills/using-superpowers/references/*.md`
- `gemini-extension.json`

### 这机制在做什么

它把 skill 文本里以 Claude Code 为主的工具名称，  
映射到：

- Codex
- Copilot CLI
- Gemini
- OpenCode

各自的等价操作。

### 机制意义

这让 superpowers 不是“绑定单一厂商的技巧集”，  
而是更接近可迁移的 agent workflow specification。

## 7. 为什么这些机制能拼成闭环

这套闭环可以概括成：

- session start 先注入“必须查 skill”
- `using-superpowers` 统一约束 skill 使用纪律
- 流程 skill 把任务推进到设计、计划、执行、审查、收尾
- hard gates 阻止常见跳步
- 测试层验证这些规则是否真的发生
- 多宿主映射保证这套规则能跨平台传播

## 核心机制总判断

superpowers 最核心的不是某个 prompt 文案多精彩，  
而是它把：

- 行为入口
- 流程路由
- 失败模式约束
- 行为验证

做成了闭环。

这就是它和普通 skill 仓库拉开差距的地方。
