# digested_src 二轮 Deep Research Progressive Plan

## 调研的根本目的

这轮 Deep Research 不是为了产出调研报告本身。

**真正的目的是：为一份面向专业人士的系统性正式报告提供可靠的原材料地基。**

报告主题：**AI Skill 生态的基础规范、分发体系、供给来源与 SDLC 方法论框架——系统性全景评估**

这份最终报告的要求是：

- **系统性**：覆盖 AI 辅助 SDLC 领域的关键维度，不遗漏，不失衡
- **逻辑性**：每个判断都有证据支撑，推理链条可追溯，结论不悬空
- **一致性**：跨章节术语统一、口径统一、评估框架统一，不自相矛盾
- **专业可读**：读者是领域内专业人士，不需要手把手解释，但需要准确、严谨、无歧义

因此，每一个进入 `reference_src` 的文档、每一条写入 `digested_src` 的判断，都必须问自己：

> **这条内容能直接支撑最终报告的哪个论点？如果不能，它就不该存在。**

## 手段策略

本轮研究采用"先建地基，再分题深挖，再横向综合，最后检验报告可用性"的递进策略。

具体分为三个波次（Wave 0 → Wave 1 → Wave 2），加一个最终验收检查点（Report Readiness Check）。

每个波次都有明确的最低标准，不满足则不进入下一波次。

## 验收标准：如何判断调研结果质量合理

调研完成的判断不依赖"感觉搜够了"，而是对照下面 4 个维度逐项检查：

### 1. 证据可追溯性

- 每个关键判断都能回指到 `reference_src` 中的具体文档
- 没有悬空结论（结论有，但找不到来源）
- 没有孤立来源（有来源，但找不到它支撑哪个结论）

### 2. 覆盖完整性

- 4 个主题各自的 6 个固定问题都已有证据支撑
- 趋势、难度、争议三类内容各有专门证据，不是顺手一提
- 已完成至少一轮"官方说法 vs 社区实践"交叉核验

### 3. 跨主题一致性

- 同一概念在不同主题下的定义口径一致
- 不同主题之间的交叉结论已经对齐或已明确标注分歧
- 已区分"基础层事实"与"局部主题内成立的判断"

### 4. 报告可用性（最终验收）

执行完成后，必须能通过以下检查：

- 拿出任意一个重要判断，能在 30 秒内找到支撑它的 `reference_src` 文档
- 对 4 个主题，能各自写出一段"机制 + 趋势 + 难点"的连贯叙述，不需要临时补搜
- 跨主题能写出一段"这个生态的整体结构是什么"的综合判断，而不只是 4 段平行描述
- 一位没有参与调研的专业人士，只看 `digested_src + reference_src`，能理解研究结论并继续往下走

如果以上任一条不满足，调研不算完成。

## 先反省：当前版本为什么还不够

如果严格对照目标，这份计划的上一版有 5 个明显问题：

### 1. 更像研究提纲，不够像执行方案

它说明了“研究什么”，  
但没有足够明确地说明：

- 每个主题至少要抓多少一手证据
- 什么程度才算“搜够了”
- 搜完后要产出什么粒度的结果

### 2. `reference_src` 的入库标准还不够硬

上一版虽然写了优先保留什么，  
但还没有明确：

- 什么才算值得落地保存的 ground truth
- 什么只能看一眼，不该入库
- 入库后要记录哪些字段，才能支撑后续复用

如果没有这个标准，最后 `reference_src` 很容易变成“搜集癖资料堆”。

### 3. 没有给每个主题设置“证据配额”和“停止条件”

如果没有最低配额和停止条件，  
就会出现两种坏结果：

- 有的主题搜太浅，证据不够
- 有的主题无限扩张，越搜越散

### 4. “趋势”和“难度”虽然被写进目标，但还没被操作化

上一版提到了趋势、难度、争议，  
但没有明确要求研究者必须把它们拆成固定问题。

结果很可能是：

- 趋势只写成几句印象判断
- 难度只停留在“有门槛”

这不够。

### 5. 没有明确规定“先证据，后综合”的检查点

如果不加检查点，很容易在证据不足时提前开始综合判断。  
这样二轮研究会重新掉回“观点先行”的老路。

## 总体策略

这轮 Deep Research 不建议直接把 4 个主题完全割裂后并行暴力搜索。  
更稳的做法是“先共享地基，再分题深挖，再横向综合”。

建议采用 3 个波次推进：

### Wave 0：建立共同 ground truth 地基

目标：

- 为 4 个主题先建立一组共用的高可信资料底座
- 优先抓官方文档、官方仓库、规范文档、主仓 README、官方 marketplace / docs 页面

优先来源类型：

- 官方 docs
- 官方 GitHub 仓库
- 规范文档
- 官方博客 / release notes
- 高可信论文或安全研究

这一波的重点不是下结论，  
而是把后续 4 题都会反复用到的“硬事实”先固化到 `reference_src`。

Wave 0 完成的最低标准：

- 至少沉淀 8 份共享型 ground truth
- 其中至少 6 份来自官方文档、官方仓库或官方规范
- 至少 1 份来自安全研究或协议分析
- 至少 1 份来自高质量对比或实践复盘

只有 Wave 0 达标，才进入 Wave 1。

### Wave 1：按 4 个主题分别深挖

每个主题都要从同样 5 个视角继续扩展：

- 证据：关键事实有没有一手来源支撑
- 根本：背后的真正抽象或机制是什么
- 趋势：最近在往哪个方向演化
- 难度：采用、维护、迁移、学习的难点在哪
- 争议：社区意见分歧、局限、失败模式是什么

每个主题在 Wave 1 的最低交付标准：

- 至少 8 份该主题专属的 ground truth 文档
- 至少 4 份一手来源
- 至少 2 份二手高质量分析
- 至少 1 份能体现趋势变化的近期来源
- 至少 1 份能体现限制、失败或争议的来源

每个主题都必须形成一份“证据摘要”与一份“问题清单更新”。

### Wave 2：横向比对与综合判断

把 4 条线重新收回来，做交叉问题判断：

- 哪些平台和机制是基础层
- 哪些市场和聚合器只是入口，不是最终答案
- 哪些供给源值得长期追踪
- 哪些方法论框架是真正能迁移的工程抽象

Wave 2 的最低标准：

- 每个横向判断都能回指到具体 `reference_src/*.md`
- 每个主题至少有 2 个与其他主题发生交叉验证的结论
- 明确区分“硬事实”“分析判断”“趋势推测”

## 证据采集协议

`reference_src` 里的每个 `md` 建议遵循统一结构：

```md
# 标题

- source_url:
- source_type:
- accessed_at:
- related_topic:
- trust_level: (official / academic / practitioner / community)
- why_it_matters:

## 关键事实

## 与本研究的关系

## 可直接引用的术语 / 概念

## 风险与局限
```

补充建议字段：

- claims_supported:
- captured_excerpt:
- date_scope:
- related_entities:

其中：

- `claims_supported`：这份材料能支撑哪些结论
- `captured_excerpt`：必要的短摘录或关键表述
- `date_scope`：材料反映的是哪个时间点的状态
- `related_entities`：涉及的平台、框架、市场、协议

命名建议：

- `01-<topic>-<source-slug>.md`
- `02-<topic>-<source-slug>.md`
- `03-<topic>-<source-slug>.md`
- `04-<topic>-<source-slug>.md`

其中 `<topic>` 对应四个主题：

- `host`
- `dist`
- `supply`
- `framework`

例如：

- `01-host-agentskills-spec.md`
- `02-dist-vercel-skills-cli.md`
- `03-supply-awesome-agent-skills.md`
- `04-framework-superpowers.md`

## 什么值得存进 reference_src

不是所有搜到的页面都值得存。  
优先保留下面这些高信号材料：

- 官方规范页
- 官方仓库 README / docs
- 明确说明产品机制的官方文档
- 关键框架的核心设计文档
- 安全研究、协议分析、趋势论文
- 解释机制差异的高质量技术文章
- 能提供真实约束、真实失败模式、真实采用门槛的案例

弱化这类来源：

- 纯流量型媒体综述
- 只做表面排名的文章
- 没有实证和机制分析的泛博客

不建议入库的材料：

- 仅重复别人结论、没有新事实的内容农场文章
- 纯 SEO 聚合页
- 只有主观推荐、没有机制或证据的短帖
- 和当前 4 个主题关联非常弱的泛 AI 文章

入库判断的硬标准：

- 能补充新事实
- 能澄清机制差异
- 能提供一手或高可信二手证据
- 能帮助解释趋势、难点或争议

如果不能满足上面任一条，就不应该进入 `reference_src`。

## 搜够了没有：停止条件

每个主题只有同时满足下面条件，才能算第一轮搜集完成：

- 核心对象清单已经稳定，不再持续新增关键名字
- 新搜到的材料大多在重复已知事实，而不是贡献新信息
- 该主题的 6 个固定问题都已经有证据支撑
- 至少有 1 轮对“反例、限制、争议”的专门补搜
- 已经完成一次“官方说法 vs 社区实践”交叉核验

如果还没有达到这些条件，就不能草率进入总结。

## 四条研究线的具体目标

### 主题 1：基础规范与宿主平台生态

研究目标：

- 扩大对 `SKILL.md` / AgentSkills 的事实理解
- 明确各宿主平台到底如何支持 skill
- 找出“标准兼容”与“平台专有封装”的真实边界

必须回答：

- 哪些平台真兼容标准，哪些只是部分映射
- skill 的发现、加载、触发、权限、组合机制分别怎么做
- 哪个平台最适合软件开发类 skill 的长期积累

证据重点：

- AgentSkills / `SKILL.md` 官方规范
- Claude Code、Codex、Cursor、Windsurf、OpenCode、Copilot 官方 docs
- skills / plugins / workflows / rules / subagents 的官方说明

难点重点：

- 名义支持和真实运行模型之间的差异
- 术语相似但机制不同
- 跨宿主迁移时的隐性成本

趋势必须回答：

- 标准是走向统一还是进一步分化
- 宿主平台是在增强兼容，还是强化专有生态锁定

难度必须回答：

- 对个人开发者的上手门槛
- 对团队迁移和内部沉淀的门槛

### 主题 2：聚合器、注册表与分发安全

研究目标：

- 扩大对 skill 分发层的理解
- 把安装、分发、版本治理、安全扫描放到一个统一框架里看

必须回答：

- 现在的 skill 分发更像 npm、plugin market 还是文档目录
- 哪些平台最适合“发现并拿来用”
- 哪些机制最适合团队级安全分发

证据重点：

- Vercel Labs `skills`
- LobeHub marketplace
- Sundial Hub
- 相关 CLI、registry、security scan、versioning 文档

难点重点：

- 大市场里的噪音过滤
- 供应链安全和 prompt injection 风险
- 团队私有化和公共分发的分裂

趋势必须回答：

- 分发层是在向包管理器化演进，还是继续 marketplace 化
- 安全扫描和版本治理会不会成为标配

难度必须回答：

- 对普通用户的发现与安装难度
- 对团队治理的接入与维护难度

### 主题 3：企业官方来源、社区索引与 MCP 共生

研究目标：

- 把内容供给层真正拆开
- 分清楚企业第一方、Awesome 清单、MCP 生态各自解决什么问题

必须回答：

- 企业官方 skill 仓库为什么有长期价值
- Awesome 索引真正能发现哪些长尾高质量资源
- skill 和 MCP 的关系到底是协作、替代还是分层

证据重点：

- Expo、Cloudflare、Hugging Face 等官方仓库
- `awesome-claude-skills`
- `awesome-agent-skills`
- `awesome-copilot`
- `awesome-mcp-servers`
- MCP 目录和检查器

难点重点：

- 企业仓库更新节奏快，信息可能变化
- Awesome 清单质量参差不齐
- MCP 和 skill 的边界经常被说混

趋势必须回答：

- 企业第一方供给会不会越来越重要
- MCP 会不会从辅助层变成能力生态主轴

难度必须回答：

- 长尾高价值资源的发现难度
- skill 与 MCP 组合使用的理解门槛

### 主题 4：SDLC 方法论框架与工程治理

研究目标：

- 深挖这些框架到底在写入什么工程能力
- 找出哪些是真正可迁移的治理抽象

必须回答：

- 哪些框架适合个人，哪些适合团队
- 哪些在防漂移、防漏验证、防跑偏上最强
- 哪些更像“工程操作系统”，哪些只是轻量规则集

证据重点：

- Superpowers
- GSD
- Gstack
- BMAD-METHOD
- Spec Kit
- OpenSpec
- FDF
- Roo Code
- Aider

难点重点：

- 框架宣传话术与真实机制要分开
- 强风格化包装与可迁移机制要分开
- Brownfield / Greenfield / solo / team 场景适配不同

趋势必须回答：

- 方法论框架是在收敛到少数主流模式，还是继续分叉
- “轻规则集”与“重框架”之间哪一类更有扩张势头

难度必须回答：

- 个人采用难度
- 团队改造难度
- 长周期维护难度

## 节奏建议

### Step 1

先为 4 个主题各列一份“权威来源优先级清单”，只抓高可信来源。

### Step 2

先充实 `reference_src`，不要一边浏览一边直接写结论。

检查点：

- 每新增一份 `reference_src` 文档，都要能回答“为什么值得保存”
- 如果只是浏览，没有形成可复用的本地文档，这一步不算完成

### Step 3

每个主题完成一轮“扩内容”：

- 多找对象
- 多找案例
- 多找机制差异

检查点：

- 是否已经覆盖该主题里最关键的对象
- 是否还只停留在第一轮里已有的名单，没有扩展出新对象

### Step 4

每个主题完成一轮“找根本”：

- 为什么会这样设计
- 解决的底层问题是什么
- 这些机制在工程上对应什么抽象

检查点：

- 结论是否已经从“项目名罗列”上升到“机制抽象”

### Step 5

每个主题完成一轮“找趋势与难点”：

- 生态正在往哪演化
- 哪些地方开始标准化
- 哪些地方依然混乱
- 实践门槛和失败模式是什么

检查点：

- 趋势是否有时间证据支撑，而不是印象判断
- 难点是否区分了个人、团队、平台三种视角

### Step 6

最后做跨主题综合，不要只留 4 份平行研究结果。

检查点：

- 是否已经明确哪些内容是共享底层事实
- 是否已经分清哪些结论只是局部主题内成立

## 成功标准

本计划执行完成后，应该达到下面状态：

- `reference_src` 中有一批高可信、可追溯的 ground truth 文档
- 对 4 个主题都不只是”知道名称”，而是能解释机制、趋势和难点
- 可以回答”什么值得长期追踪，什么只是噪音”
- 可以支撑后续继续细分、更深的 Deep Research

更具体地说，达到下面状态才算真正满足目标：

- 每个主题至少形成一组可复用证据包，而不是临时搜索结果
- 每个重要判断都能回指到本地 `reference_src` 文档
- 对趋势、难度、争议都有专门证据，而不是顺手一提
- 后续新的 agent 接手时，只看 `digested_src + reference_src` 就能继续往下研究

**最终报告可用性检查（Report Readiness Check）：**

以下 4 条全部满足，才算本轮调研真正完成：

1. 任意抽取一个重要判断，30 秒内能找到 `reference_src` 中的支撑文档
2. 对 4 个主题各自能写出”机制 + 趋势 + 难点”的连贯段落，不需要临时补搜
3. 跨主题能写出”这个生态的整体结构是什么”的综合判断，不只是 4 段平行描述
4. 没有参与调研的专业人士，只看 `digested_src + reference_src`，能理解结论并继续深入

## Progressive Plan 元规则

每一轮 progressive plan，不论是第几轮，都必须在开头写清楚以下 4 件事：

1. **调研目的**：这一轮调研最终服务于什么产出？谁是读者？读者期待什么质量？
2. **手段策略**：用什么结构推进？分几个阶段？各阶段的最低标准是什么？
3. **验收标准**：如何客观判断调研结果质量合格？验收不依赖主观感觉，必须可检查
4. **最终结果质量要求**：调研完后能直接支撑的产出是什么？这个产出对专业读者意味着什么？

不满足这 4 条的 plan，不算一份合格的 progressive plan。

## 备注

这一轮研究应该鼓励深挖，而不是追求快搜快写。
重点不是把链接堆多，而是逐步形成一套可靠的研究地基，让后续每一轮都能踩在已经沉淀的 evidence 上继续往前走。

调研的终点不是”搜够了”，而是”能写出一份专业人士愿意看、看得懂、挑不出大漏洞的报告”。这是每一轮调研的最高约束。
