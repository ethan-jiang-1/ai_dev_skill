报告主题：**工程师如何通过 AI Skill 掌握工程方法学体系——认知路径、工具生态与组织治理的系统性评估**

这份最终报告的要求是：

- **系统性**：覆盖从个人认知路径到团队治理的完整维度，不遗漏关键环节
- **逻辑性**：每个判断都有一手或高可信二手证据支撑，推理链条可追溯，结论不悬空
- **一致性**：跨章节术语口径统一，评估维度统一，不同章节结论不自相矛盾
- **专业可读**：读者是工程领导者、技术决策者、L&D 负责人，需要准确、严谨、无歧义

因此，每一个进入 `reference_eng` 的文档、每一条写入分析结论的判断，都必须问自己：

> **这条内容能直接支撑最终报告的哪个论点？如果不能，它就不该存在。**

**权威信源优先规则（最高优先级，不可违反）：**

> 当权威一手来源已经足以支撑某个判断时，不为满足配额额外引入低质量二手来源。配额是防止搜太浅的下限，不是必须凑满的目标。宁缺毋滥。

## 先反省：第一轮消化结果为什么还不够

### 1. 核心主张"认知脚手架"缺乏一手实证

第一轮提炼出了"Skill 是认知脚手架"这个核心判断，但没有找到直接支撑它的一手证据——没有针对 Skill 使用效果的实证研究，没有工程师能力提升的真实数据，没有"逆向学习"在软件工程场景下的独立验证。

### 2. 难度分层是逻辑推演，缺乏实践验证

三层分类（简单/中等/复杂）是合理的框架，但没有引用任何真实的工程师学习案例、企业培训数据或社区反馈来验证这个分层是否符合实际。

### 3. 工具对比停留在定性描述

Cursor vs Windsurf 的对比（显式控制 vs 隐式自动化）是有价值的判断，但缺乏量化数据、独立评测或真实用户反馈支撑。"Windsurf 更容易认知卸载"这类判断需要实证。

### 4. Skill 开发生命周期缺乏真实案例

四阶段开发路径（Discovery→Build→Evals→Deploy）是合理的工程框架，但没有引用任何工程师详细记录自己开发 Skill 全过程的真实案例，也没有关于 Evals 设计的具体实践。

### 5. 团队治理策略缺乏落地证据

四阶段跃迁路径和团队采纳策略是理论构建，没有真实的企业案例、组织阻力数据或治理失败模式作为支撑。

**Claim Strength Policy（适用于 eng 全部主题，尤其是"认知脚手架"主张）：**

当直接实证不存在时，允许结论降级表述，禁止硬凑强结论：

| 证据状态 | 允许的表述强度 |
|----------|--------------|
| 有直接针对 AI Skill 的实证研究 | "已有证据表明" / "研究显示" |
| 只有邻近领域研究（教育学、认知科学）可类比 | "邻近证据支持" / "类比推断" |
| 无直接实证，只有逻辑推演 | "合理推断" / "尚缺直接验证" |
| 社区有争议或反例 | 必须并列呈现正反两方证据 |

执行时：不允许把"教育学研究显示脚手架有效"直接等价为"AI Skill 作为认知脚手架有效"。必须明确标注跨域类比的边界。

## 验收标准：如何判断调研结果质量合理

调研完成的判断不依赖"感觉搜够了"，而是对照下面 4 个维度逐项检查：

### 1. 证据可追溯性

- 每个关键判断都能回指到 `reference_eng` 中的具体文档
- 没有悬空结论（结论有，但找不到来源）
- 没有孤立来源（有来源，但找不到它支撑哪个结论）

### 2. 覆盖完整性

- 4 个主题各自的 6 个固定问题都已有证据支撑
- 趋势、难度、争议三类内容各有专门证据，不是顺手一提
- 已完成至少一轮"理论主张 vs 实践证据"交叉核验

### 3. 跨主题一致性

- 同一概念（如"认知脚手架""认知卸载"）在不同主题下的定义口径一致
- 不同主题之间的交叉结论已经对齐或已明确标注分歧
- 已区分"个人开发者场景"与"团队/企业场景"的不同结论

### 4. 报告可用性（最终验收）

以下 4 条全部满足，才算本轮调研真正完成：

1. 任意抽取"认知脚手架"主张的一个论点，30 秒内能找到 `reference_eng` 中的支撑文档
2. 对 4 个主题各自能写出"主张 + 证据 + 局限"的连贯段落，不需要临时补搜
3. 能写出一段"工程师 Skill 掌握路径整体图景"的综合判断，不只是 4 段平行描述
4. 没有参与调研的专业人士（工程领导者或 L&D 负责人），只看 `round2_eng + reference_eng`，能理解结论并继续深入

如果以上任一条不满足，调研不算完成。

## 总体策略

先共享地基，再分题深挖，再横向综合。

### Wave 0：建立共同 ground truth 地基

目标：为 4 个主题先建立一组共用的高可信资料底座。

优先来源类型：
- 认知科学与学习理论的学术研究（认知脚手架、认知卸载、逆向学习）
- SKILL.md 规范的官方定义与设计文档
- 各工具官方文档（Cursor、Windsurf、Claude Code）
- 企业工程博客中的真实 Skill 实践（非软文）

Wave 0 完成的最低标准：

- 至少沉淀 8 份共享型 ground truth
- 其中至少 3 份来自认知科学或教育学研究（直接支撑"认知脚手架"主张）
- 至少 3 份来自工具官方文档或官方仓库
- 至少 1 份来自安全/质量工程视角的独立分析
- 至少 1 份来自社区真实使用反馈（非框架作者）

只有 Wave 0 达标，才进入 Wave 1。

### Wave 1：按 4 个主题分别深挖

每个主题都要从同样 5 个视角继续扩展：

- 证据：关键事实有没有一手来源支撑
- 根本：背后的真正机制是什么
- 趋势：最近在往哪个方向演化
- 难度：采用、维护、迁移、学习的难点在哪
- 争议：社区意见分歧、局限、失败模式是什么

每个主题在 Wave 1 的最低交付标准：

- （严格版）至少 12 份该主题专属的 ground truth 文档
- （严格版）至少 6 份一手/高可信来源（official 或 academic）
- 至少 2 份独立第三方分析（与工具/框架作者无直接利益绑定）
- 至少 2 份直接软件工程场景的“过程/行为/结果”实证（可为实验/纵向/field study；但必须是可复核方法的研究，不是泛论）
- 至少 1 份能体现趋势变化的近期来源
- 至少 2 份能体现限制、失败或争议的来源（不要只写正面）

每个主题都必须形成一份"证据摘要"与一份"问题清单更新"。

### Wave 2：横向比对与综合判断

把 4 条线重新收回来，做交叉问题判断：

- "认知脚手架"主张在哪些工具和场景下有实证支撑

- 工具选择与学习路径的交叉影响
- 团队治理的通用原则与场景限制

Wave 2 的最低标准：

- 每个横向判断都能回指到具体 `reference_eng/*.md`
- 每个主题至少有 2 个与其他主题发生交叉验证的结论
- 明确区分"硬事实""分析判断""趋势推测"

## 四条研究线的具体目标

### 主题 1：Skill 作为认知脚手架与逆向学习

研究目标：

- 为"认知脚手架"主张找到直接的实证支撑（不只是类比推演）
- 验证"逆向学习"机制在软件工程场景下的有效性
- 找出哪些 Skill 设计特征能促进学习而非替代学习

必须回答：

- 有没有直接研究 Skill 对工程师能力提升影响的实证研究
- "认知卸载"在 AI 辅助编程中的发生条件和程度是什么
- 哪些 Skill 设计特征被证明能促进学习（有实证）
- TDD、Spec-Driven 这类方法为什么特别适合被 Skill 化

证据重点：

- Anthropic 的 AI 辅助编程认知研究
- 教育技术领域关于脚手架学习的研究
- 工程师社区关于"用 AI 学到了什么"的真实反馈
- 高质量 Skill 的设计原则文档（如 Block Engineering Blog）

难点重点：

- 认知科学研究和软件工程实践之间的跨域映射需要谨慎
- 需要区分"短期效率提升"和"长期能力提升"

趋势必须回答：

- Skill 的设计是否在向"更强制人参与"的方向演化
- 认知卸载风险是否已经引起工具设计者的重视

难度必须回答：

- 设计一个"既能帮人做事、又能训练人"的 Skill 的难点在哪
- 工程师识别和抵抗认知卸载的门槛是什么

### 主题 2：Skill 难度分层与能力训练矩阵

研究目标：

- 为三层分类（简单/中等/复杂）找到真实的实践验证
- 找出不同层级 Skill 对工程师能力要求和训练效果的实证
- 验证"中等简单层是能力提升主战场"这一判断

必须回答：

- 如何判断一个 Skill 属于哪一层（有没有可操作的标准）
- 哪些能力是使用 Skill 的前置条件，哪些会被反向训练出来
- 为什么中等简单这一层最适合工程师成长（有没有实证）
- 开源 Skill 在不同复杂度层上应该怎么采纳（有没有真实案例）

证据重点：

- 高 star 的开源 Skill 仓库的分类和使用说明
- 工程师社区关于"哪类 Skill 最有学习价值"的讨论
- 企业工程培训项目中的 Skill 分层实践
- 独立评测中对不同复杂度 Skill 的使用体验报告

难点重点：

- 分层标准容易主观，需要找到可操作的判断维度
- 需要区分"对个人开发者的价值"和"对团队的价值"

趋势必须回答：

- Skill 的复杂度分布是否在向高层级集中
- 工具是否在降低高复杂度 Skill 的使用门槛

难度必须回答：

- 从第二层跨越到第三层的真实障碍是什么
- 团队推广高复杂度 Skill 的主要阻力在哪里

### 主题 3：Skill 开发生命周期与工具生态支持

研究目标：

- 为四阶段 Skill 开发生命周期找到真实的实践案例
- 对 Cursor、Windsurf、CLI agents、节点式工具做有实证的横向对比
- 验证"显式控制 vs 隐式自动化"对工程师认知的不同影响

必须回答：

- 有没有工程师详细记录自己开发 Skill 全过程的案例
- Skill 的 Evals（评估测试集）在实践中是如何设计和执行的
- Cursor 的显式控制范式 vs Windsurf 的隐式级联，在实际使用中对工程师认知的影响有没有独立评测
- 哪个工具对"从零开始学习 Skill"的工程师最友好

证据重点：

- 各工具官方文档中关于 Skill/Rules 的技术说明
- 独立评测博客（非工具官方）
- 工程师社区关于"Skill 开发踩坑"的真实反馈
- 跨工具 Skill 兼容性的实际测试报告

难点重点：

- 工具迭代很快，需要注意来源的时效性
- 很多"对比"文章实际上是软文，需要识别

趋势必须回答：

- 工具是否在向"更支持 Skill 版本化和团队共享"的方向演化
- CLI agents 和 IDE 工具的 Skill 生态是否在收敛

难度必须回答：

- 从"会用 Skill"到"会开发 Skill"的真实门槛是什么
- Skill 的 Evals 设计为什么难，常见失败模式是什么

### 主题 4：工程师跃迁路径、团队采纳与样本矩阵

研究目标：

- 为四阶段跃迁路径找到真实的工程师成长案例
- 为团队级 Skill 治理策略找到真实的企业实践案例
- 验证三类采纳策略（直接用/改着用/自建）的适用边界

必须回答：

- 工程师从"不会用 AI"到"会写 Skill"中间有哪些真实阶段（有案例）
- 团队怎么把 Skill 当成数字资产而不是个人玩具（有实践）
- 直接用、改着用、自建，分别适用于哪些类型的 Skill（有实证）
- 哪些具体 Skill 最适合作为"边解决问题边学习方法学"的教学样本

证据重点：

- 企业工程博客中的真实 Skill 治理实践（Atlan、Elementor 等）
- GitHub 上企业级 Skill 仓库的组织方式和 fork 率
- 工程领导者关于 AI 工具治理的公开分享
- 开源 Skill 库的真实采纳数据（star 趋势、issue 活跃度）

难点重点：

- 企业治理案例通常不公开细节
- 需要区分"个人开发者的最佳实践"和"企业级治理方案"

趋势必须回答：

- 团队级 Skill 治理是否在向标准化方向演化
- 开源 Skill 生态的质量和可信度是否在提升

难度必须回答：

- 组织推广 Skill 体系时最常遇到的阻力是什么（技术 vs 文化）
- 从个人 Skill 到团队 Skill 的演进过程中，最常见的障碍是什么

## 什么值得存进 reference_eng

优先保留：

- 认知科学、教育技术领域的学术研究
- 工具官方文档中关于 Skill/Rules 的技术说明
- 企业工程博客中的真实 Skill 治理实践（非软文）
- 工程师社区的真实使用反馈（包括批评和失败案例）
- 独立评测报告（非工具官方）

不建议入库：

- 工具官方的营销内容（除非包含具体技术细节）
- 没有实证的"AI 改变工程师"类泛论文章
- 纯 SEO 聚合页或 AI 生成的工具对比文章
- 与 4 个主题关联非常弱的内容

入库判断的硬标准：

- 能补充新事实或澄清机制差异
- 能提供一手或高可信二手证据
- 能帮助解释学习路径、工具差异、治理难点或争议

## 搜够了没有：停止条件

每个主题只有同时满足下面条件，才能算搜集完成：

- 核心工具和 Skill 清单已经稳定，不再持续新增关键名字
- 新搜到的材料大多在重复已知事实，而不是贡献新信息
- 该主题的 6 个固定问题都已经有证据支撑
- 至少有 1 轮对"反例、限制、争议"的专门补搜
- 已经完成一次"理论主张 vs 实践证据"交叉核验

### 口径澄清（避免与“探索宽度持续增长”冲突）

- “探索宽度持续增长”指的是：围绕同一主张同时扩张多条证据线（学术/工具机制/真实案例或社区反馈），并不断下钻到可验证细节；它不等价于“无限新增工具名字”。
- “核心工具和 Skill 清单稳定”指的是：用于最终报告推理链条的核心对象集合不再发生结构性变化；在此之后仍然允许继续补更强证据、补反例、补局限，以及把证据链条变得更可追溯。
- 实操上：先允许对象清单增长，直到“对象集合稳定”；随后以“证据链加密”为主，而不是继续堆新对象。

### Step 3

检查点：

- 是否还停留在第一轮里已有的名单，没有扩展出新对象

检查点：

### 严格意义上的“完成定义”（Hard Gates，满足才允许说“跑完”）

- P0 缺口必须被关闭或被“结论降级并稳定化”：
  - “关闭”指：找到直接 SE 实证，能把关键主张从“邻近证据/类比”升级到“已有证据表明”（仍需写局限）。
  - “降级并稳定化”指：把主张写成条件性结论，并在 `round2_eng/*.md` 与 `round2_eng/_artifacts/W2-cross-topic-synthesis.md` 中同步更新，确保跨章节一致，不再以强结论口吻出现。
- 必须有“负结果/反例”支撑的诚实边界：
  - 至少 1 条来自高可信研究或官方限制清单的“无显著提升/存在明确限制/失败模式”的证据，避免报告变成只讲收益的软文。

## Progressive Plan 元规则

每一轮 progressive plan，不论是第几轮，都必须在开头写清楚以下 4 件事：

1. **调研目的**：这一轮调研最终服务于什么产出？谁是读者？读者期待什么质量？
2. **手段策略**：用什么结构推进？分几个阶段？各阶段的最低标准是什么？
3. **验收标准**：如何客观判断调研结果质量合格？验收不依赖主观感觉，必须可检查
4. **最终结果质量要求**：调研完后能直接支撑的产出是什么？这个产出对专业读者意味着什么？

不满足这 4 条的 plan，不算一份合格的 progressive plan。

## 备注

这一轮研究的核心挑战是：第一轮已经有了清晰的结论框架，但结论的证据基础不够硬。研究的重点不是重新发现已知结论，而是为已有结论找到可追溯的一手证据，同时主动寻找反例和局限性。

特别注意：对"认知脚手架"这个核心主张，研究目标不是"证明它是对的"，而是找到它在哪些条件下成立、在哪些条件下不成立——这才是专业报告应有的诚实态度。

调研的终点不是"搜够了"，而是"能写出一份专业人士愿意看、看得懂、挑不出大漏洞的报告"。这是每一轮调研的最高约束。
