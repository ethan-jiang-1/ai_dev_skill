# AI Agent在SDLC中的能力单元架构：机制、证据与企业迁移价值评估

**面向工程架构师、开发效能团队与技术决策者的实践报告**

---

## 执行摘要

AI Agent 在软件开发中的能力建设，正在从“提示词技巧”走向“能力单元架构”。真正高价值的对象，已经不是某个会说漂亮话的 skill，而是那些能够改变执行控制流、限制动作空间、生成可验证产物、并把治理要求写进默认流程的机制化单元。

本报告基于前期四条专题研究线的综合整理，对 AI Agent 在 SDLC 中的能力单元做系统性整合，回答五个问题：

1. 什么样的对象才配称为“能力单元”，而不是 prompt 包装
2. 在规划、构建、调试、审查、发布、运维等阶段，哪些机制真正改变了 agent 的工作方式
3. 这些能力单元背后的共通工程抽象是什么
4. 哪些能力单元具备较高的企业迁移价值，哪些只是原框架里的风格化外壳
5. 企业在迁移这些机制时，需要承担哪些治理成本、维护成本和安全风险

**核心发现**：

1. **能力单元的本质是“改变执行机制”，不是“换个说法下指令”**。  
   一个值得迁移的能力单元，必须至少具备以下特征中的几项：改变执行顺序、引入显式 gate、产出可审计文件、重排权限边界、或把验证前置为确定性约束。

2. **SDLC 中最可迁移的四个工程抽象已经相当清晰**。  
   它们分别是：控制流分离、验证闭环前置、确定性门禁与对抗性审查、状态持久化。这四类抽象比具体命令名、人格设定或框架包装更值得复用。

3. **规划阶段的价值，不在“多想一步”，而在“阻止模型过早写代码”**。  
   强规划框架普遍把需求澄清、范围控制、设计审查和验证前置写成硬门禁，用 design docs、validation artifacts、配置面和计划审查替代“先写出来再修”的惯性。

4. **构建与调试阶段的核心，不是“让模型更聪明”，而是把退化、盲猜和无限重试工程化地隔离出去**。  
   orchestration 与执行分离、fresh-context executor、TDD iron law、systematic debugging、repair budget 与熔断机制，都是在对抗 context rot 与假完成。

5. **Review / QA / Ship / Operate 阶段正在从“聊天式检查”走向“证据优先的运行时治理”**。  
   多 reviewer 聚合、adversarial critic、runtime QA、baseline + delta、checkpoint/handoff、health/canary/benchmark 等机制，正在把质量与稳定性转成持续可审计的资产。

6. **企业迁移价值的核心不在“功能更多”，而在“净收益是否大于治理负担”**。  
   高迁移价值能力单元往往具备 determinism、auditability 与 portability；低迁移价值对象则常常只有 prompt 包装，没有验证产物、没有宿主适配层，也没有明确的安全控制面。

**关键行动建议**：

- 对架构与效能团队，应把能力单元当作工程资产设计，而不是知识库条目维护。
- 对框架作者，应优先迁移机制层，不要把命令名、语气、人设和文化包装误当成核心价值。
- 对企业采用方，应优先落地那些能外化为 gate、artifact、policy 和 baseline 的能力单元。
- 对治理团队，应把 approval、sandbox、allowlist、签名、审计与宿主适配层视为迁移方案的一部分，而不是后补项。

**证据透明性声明**：  
本报告明确区分三类判断。

- **硬事实**：官方框架文档、公开配置契约、学术与工业实证、社区公开失败样本中能直接复核的描述
- **分析判断**：基于多个框架与研究信号提炼出的共同工程抽象
- **趋势推测**：基于近期论文、产品说明与社区变化对未来方向的推断

**贯穿全文的统一评估框架**：  
为避免四个阶段各说各话，本文统一用五个问题评价能力单元：

1. 它是什么：是 prompt、规则、工作流、适配层、控制面，还是可执行工件
2. 它改变了哪一层执行控制流：规划、执行、验证、审查、交接、运维
3. 它生成了什么确定性产物：文档、验证文件、评审报告、checkpoint、趋势数据、策略配置
4. 它带来了什么迁移成本：采用复杂度、维护负担、宿主适配面、训练成本
5. 它的治理边界是什么：approval、sandbox、allowlist、审计、供应链与外部工具风险

当前仍有三个重要证据缺口：

1. 迁移后的 correctness 与质量净收益，仍缺更多可复核企业案例
2. 多宿主 portability layer 的长期维护成本，目前更多是 snapshot 与局部 churn 切片
3. 外部 MCP / 插件能力进入企业交付流程后的安全与审批实战案例仍不充分

## 第一部分：能力单元的本质与前置规划机制

### 1.1 能力单元不是“会说话的 prompt”

如果一个对象只是告诉模型“请更认真一些”“请像高级工程师一样思考”，它最多算 prompt 包装，不算能力单元。

真正的能力单元需要改变系统的工作方式。更具体地说，它通常至少具备以下一种能力：

- 改变执行顺序
- 引入显式 gate
- 限制 agent 的动作空间
- 生成可审计产物
- 把某类隐性经验外化为协议或配置
- 让后续阶段能够读取其结果，而不是重新猜测上下文

因此，能力单元更接近“执行控制与治理部件”，而不是“高级提示词”。

（硬事实：基于多套框架官方工作流说明、配置文件与控制面定义）

### 1.2 规划阶段的核心，不是想得更全，而是更早阻断失控

AI Agent 在 SDLC 早期最常见的失败模式，不是不会写代码，而是过早写代码。

在需求尚未澄清、范围尚未锁定、验证路径尚未定义时，模型就开始生成实现。这会导致三类典型问题：

- 需求跑偏被编码进结构
- 范围悄悄扩张或缩减
- 交付标准直到编码后期才暴露

高质量规划能力单元的真正价值，在于把这种隐性风险显式拦住。

代表性的做法包括：

- 在 design 阶段设置 hard gate，明确禁止写实现代码
- 把 scope 变化做成人类显式批准的决策点
- 将 coverage、distribution、验证方式写进计划审查
- 把 requirements 映射成后续必须执行的验证命令

换句话说，好的 planning 单元不是“帮你写个计划”，而是把“何时才能进入实现”这件事定义成系统契约。

### 1.3 前置规划最有价值的三个机制

#### 1. 范围控制

规划阶段最大的隐藏故障，是 silent drift。  
模型会在没有显式告知用户的情况下扩大范围、缩减目标，或者把复杂目标偷换成更容易完成的版本。

因此，scope control 的本质不是“提醒注意范围”，而是把 scope change 升级为需要明确选择的 gate。

#### 2. 结构化计划工件

计划如果只是自然语言待办清单，就很容易在执行阶段退化成“看起来合理”的泛化指令。  
更可靠的做法，是把计划变成可执行工程协议：

- 任务足够原子
- 阶段切分清晰
- 风险点前置暴露
- 后续验证路径已绑定

这样计划不再是“思路记录”，而是执行树。

#### 3. 验证前置

这是规划阶段最值得迁移的高价值机制之一。  
它的核心不是多写测试，而是在真正执行前就回答：

- 这项需求最终如何被证明成立
- 对应的命令、脚本、观察信号或 UAT 是什么
- 若当前缺少验证基础设施，是否要先补齐 Wave 0 脚手架

当“完成标准”被提前写死成可执行验证方式时，模型就更难靠流畅的叙事伪装交付。

### 1.4 Context Files 的价值与边界

Context files，例如 `AGENTS.md`、rules、repo instructions，正在成为跨宿主的常见工件。

它们的价值很明确：

- 帮 agent 快速建立最小可运行语境
- 提供命令、目录、约束与入口信息
- 提高一致性，降低初始探索成本

但现有研究也提示了一个关键边界：**context files 不是越长越好**。  
如果把大量非必要要求、细节规则和治理逻辑全部堆进 context file，可能会带来：

- 成功率下降
- 推理成本上升
- 无关约束干扰主任务

更合理的分层通常是：

- context files：保留最小 requirements、目录线索、关键命令和入口
- hooks / gates / validation artifacts：承载质量、安全、审批与回归等强约束

这也是“入口索引”和“治理系统”必须分层的原因。

### 1.5 第一部分结论

这一部分可以收敛成四个判断：

1. [硬事实] 能力单元的价值在于改变执行机制，而不是修饰表达方式
2. [分析判断] planning 阶段最关键的作用是阻止模型过早进入实现
3. [硬事实] 结构化计划、scope gate 与验证前置，是多个框架共通的 planning 抽象
4. [分析判断] context files 最适合作为最小可运行入口，而不是承载全部治理逻辑

## 第二部分：构建执行、调试与验证闭环

### 2.1 Build 阶段真正的问题是 context rot

在 AI Agent 编程里，构建阶段最严重的失败模式往往不是不会写，而是随着会话变长逐渐写错。

常见表现包括：

- 忘记早先的约束与假设
- 把多个任务缠绕在同一上下文中
- 完成信号依赖语言叙事而不是外部证据
- 在长时间运行后出现 vagueness、partial completion、跳步

这意味着构建问题首先是一个进程模型问题，而不是“让模型更努力一点”的问题。

### 2.2 控制流分离：编排器不执行，执行器不背负长上下文

这是 `cap` 研究里最重要的高迁移价值抽象之一。

当 orchestrator 负责：

- 分解任务
- 安排依赖
- 派发 wave
- 收集结果
- 触发 repair 或 escalate

而 fresh executor 只负责在干净上下文里完成原子任务时，系统获得的收益并不仅是并发更快。

真正的收益是：

- 执行质量不再随上下文长度线性退化
- Task 1 与 Task 50 的局部实现质量更接近
- 上一个任务遗留的“认知垃圾”不再污染下一个任务
- 完成判定更容易外置到文件、提交、验证报告等产物

这类控制流分离，是对抗 context rot 最可迁移的工程抽象之一。

### 2.3 Context Budget 与 Compaction：上下文本身是一种工程资源

高质量构建体系不会把上下文当成无限资源。

这类体系通常会明确规定：

- 大文件不要整段 inline 进 prompt
- read depth 需要按上下文窗口动态调整
- 长日志需要截断，只保留关键头尾与截断提示
- 必要时要通过 compaction 保留状态而压缩冗余细节

宿主实现和框架协议在这里给出了相同方向的证据：  
不管是框架层的 context budget 规则，还是宿主层的 output cap 与 compaction，本质都在做一件事，即把上下文退化从“聊天偶然现象”升级为可治理的运行时约束。

### 2.4 强制 TDD：改变的是习惯，不只是顺序

很多人把 TDD 看成开发风格，但在 agent 语境下，它更像一种反幻觉协议。

“没有 failing test，不准写 production code”的意义不在于教条，而在于：

- 把接口定义提前到实现之前
- 强迫模型先见到失败信号，再开始修复
- 避免靠“看起来像完成了”的实现误导后续阶段

当 TDD 被工程化为：

- 一 feature 一计划
- RED / GREEN / REFACTOR 具备明确提交边界
- 缺基建时先补测试脚手架
- 违背顺序就回滚或删除提前写出的实现

它就不再是建议，而是硬门禁。

### 2.5 调试的关键，是剥夺“马上改代码”的权限

高质量调试协议之所以重要，不是因为它把 debugging 写得更详细，而是因为它重排了执行权限。

典型的阶段化调试会要求：

1. 先收集症状与观测信号
2. 再隔离问题边界
3. 再提出根因假设
4. 最后只允许通过小范围验证去证实或证伪

在根因确认前，不允许大面积改实现。

这直接抑制了 LLM 的典型失败行为：

- 盲猜
- 试错式乱改
- 把 symptom 当 root cause
- 重复失败但继续无限重试

### 2.6 Repair Budget 与熔断：自动化失败也要优雅退出

高级能力单元不只会做成功路径，也会把失败路径工程化。

这类协议通常会在失败后要求系统明确选择：

- 是 RETRY
- 是 DECOMPOSE
- 是 PRUNE
- 还是 ESCALATE

并为每种路径定义预算与后果。

这类 repair budget 的价值非常直接：  
它把资源黑洞变成了可控流程，把“越修越坏”变成“在预算内失败、在边界内升级”。

### 2.7 第二部分结论

这一部分可以收敛成五个判断：

1. [硬事实] 构建阶段最核心的工程问题是 context rot，而不是单次编码能力
2. [分析判断] 控制流分离是对抗长程退化最强的可迁移抽象之一
3. [硬事实] TDD enforcement 与 verification-before-completion 属于高价值硬门禁
4. [分析判断] system debugging 的价值在于剥夺盲改权限，而不是增加几步解释
5. [趋势推测] 交互式动态调试、runtime 查询与宿主级 compaction 会成为下一阶段调试能力的重要方向

## 第三部分：审查、发布、运维与状态持久化

### 3.1 Review 的致命问题，是“同一个模型审自己”

Builder 自审的问题，不在于“不够认真”，而在于结构性偏差。

模型会自然偏向自身输出，容易出现：

- 漏掉安全问题和边界条件
- 给出过于宽松的评价
- 用更流畅的叙事掩盖推理漏洞

相关研究和框架设计都在指向同一结论：  
如果 review 仍由 builder 在同一上下文内完成，那么它的制衡价值会非常有限。

### 3.2 对抗性审查与 multi-review aggregation

高质量 review 能力单元的共同特点是“解耦”：

- reviewer 使用 fresh context
- reviewer 关注需求、diff、证据与 blocking issues
- 审查结果被落盘，而不是停留在对话中
- 多个 reviewer 或 specialist 的结论会被汇总，而不是单点裁决

这类设计的价值已经不只是经验判断。  
从 benchmark 到框架实践，越来越多信号都说明：multi-review aggregation 能提高问题发现覆盖率，而 adversarial critic 能有效对抗 builder 的回音壁。

因此，review 的真正升级，不是“多说两句审查意见”，而是把审查做成一个独立的、可审计的控制流阶段。

### 3.3 Runtime QA：从“代码文本”走向“可观察事实”

成熟的 QA 能力单元不会止步于静态 diff 或文本审查，它会尝试穿透到真实运行时。

高杠杆的做法包括：

- 浏览器自动化验证
- 页面交互与截图证据
- console error 采样
- 性能采样
- baseline 与 regression 对照

这意味着 QA 的信号开始从“模型说没问题”转向：

- 看得见的页面变化
- 可测量的性能差异
- 可复现的运行时异常

当这些结果被写成报告、JSON、JSONL 或结构化文件时，QA 就从一次性检查变成了回归资产。

### 3.4 Ship / Operate 的核心是 baseline + delta

发布与运维最容易失败的地方，在于系统只会做“当前是否正常”的静态判断，而不记录与过去的关系。

高价值运维能力单元强调：

- 先建立 baseline
- 之后只对变化发出告警
- 用 delta 与阈值判断 regression
- 把结果写成可持续追踪的趋势资产

这种机制在 canary、benchmark、health 三类单元里尤为明显。

它们共同说明，运维治理的关键不是让 agent “更会看日志”，而是让系统拥有比较基线、变化检测和长期积累的能力。

### 3.5 文档同步与状态持久化：长期稳定性的底层支撑

代码变了、文档没变，是最常见的系统漂移之一。  
因此，document-release 这类能力单元之所以重要，不是因为它“顺手更新文档”，而是因为它把“文档与实现是否仍一致”做成了发布闭环的一部分。

更底层的，是文件型状态持久化。

无论是：

- checkpoint
- handoff
- `.continue-here`
- `STATE.md`
- `PLAN.md`
- `DECISIONS.md`
- review / health / benchmark history

这些工件都在做同一件事：  
把 agent 项目的长期状态，从聊天记忆中迁移到可审计、可恢复、可交接的文件系统上。

在目前的实践阶段，文件型状态机仍然是最可靠的项目记忆基础设施。

### 3.6 安全控制面：后半段能力单元必须纳入 policy

一旦 review、ship、operate 阶段需要连接：

- shell
- 浏览器
- MCP server
- 外部工具
- 自动化环境

治理问题就不再是附加项，而是进入主流程。

因此，approval policy、sandbox、allowlist、签名、语义 vetting、runtime guardrails 这些对象，不能只放在安全附件里。  
它们本身就是能力单元架构的一部分。

尤其是在 MCP 场景下，风险不止 prompt injection，而是会扩展到：

- confused deputy
- SSRF
- session hijacking
- tool poisoning
- shadowing
- rug pulls
- 多 server 放大效应

这也是为什么后半段能力单元要和 policy contract 一起设计，而不是事后补丁。

### 3.7 第三部分结论

这一部分可以收敛成五个判断：

1. [硬事实] review 必须与 builder 解耦，fresh-context critic 才具备真正的制衡价值
2. [分析判断] multi-review aggregation 与 adversarial review 是更接近企业级默认流程的方向
3. [硬事实] runtime QA、canary、benchmark 与 health 正在把后半段治理转成持续可审计资产
4. [分析判断] 文件型状态持久化是当前最可靠的长期项目记忆机制
5. [硬事实] 外部工具链、MCP 与宿主权限边界已进入能力单元架构的核心治理面

## 第四部分：能力地图、迁移价值与跨框架抽象

### 4.1 不要迁移名字，要迁移机制

企业引入开源框架时最常见的误判，是把“有名字的 skill”当成“有价值的能力抽象”。

但很多对象真正可迁移的部分并不是：

- 命令名
- 口吻
- 人设
- 特定社区文化
- 看起来很酷的流程包装

真正值得迁移的是它们背后的机制：

- 怎样组织控制流
- 怎样前置验证
- 怎样设置 gate
- 怎样审查 builder
- 怎样持久化状态
- 怎样把风险收敛成 policy

### 4.2 四个最核心的工程抽象

基于四条研究线的综合，最值得保留下来的不是项目名单，而是以下四类抽象。

#### 1. 控制流分离

将 orchestrator、executor、reviewer、critic、operator 分离，使不同阶段在更干净的上下文与更明确的权限边界下运行。

#### 2. 验证闭环前置

把验证标准从“写完之后再想办法证明”前置到 planning 与 execution 前，形成 validation artifacts、UAT、baseline、preflight 等确定性约束。

#### 3. 确定性门禁 + 对抗性审查

通过 hard gate、approval、TDD iron law、verification-before-completion、adversarial review、multi-review aggregation，把质量要求外移为可检查 contract。

#### 4. 状态持久化

通过 checkpoint、handoff、health history、review logs、QA reports、benchmark trends 等工件，让系统具备跨会话、跨分支、跨阶段的连续性。

这四类抽象，比任何单个框架都更稳定，也更适合作为企业迁移的最小骨架。

### 4.3 迁移价值的判断标准

迁移价值不应该理解为 popularity ranking，而应理解为一个净收益判断。  
至少需要同时考虑以下因素：

- adoption complexity：引入前提、集成改造量
- governance burden：审批、权限、审计与安全要求
- maintenance overhead：兼容债、依赖更新、回归成本
- portability cost：多宿主映射层、目录约定、能力差异
- team training cost：团队稳定使用所需的训练与心智负担

因此，高迁移价值不等于“实现最复杂”，而等于：  
**确定性收益显著，且治理成本能被组织吸收。**

### 4.4 当前能力地图：哪些值得优先迁移

#### 高迁移价值

- 仓库级最小配置工件，例如 `AGENTS.md`、rules、instructions
- 验证前置与 validation artifacts
- TDD enforcement 与 completion gate
- systematic debugging protocol
- adversarial review 与 multi-review aggregation
- 企业治理控制面，例如 approval、sandbox、prefix rules、allowlist

这些对象的共同点是：

- 能直接影响控制流
- 能形成可审计产物
- 能把质量或安全要求外移为确定性约束

#### 中到高迁移价值

- 宿主适配层，例如 installer、rewrites、adapters、conversion tests
- runtime QA、benchmark、health、canary
- 文档漂移审计与发布闭环

这类对象的价值很高，但它们往往伴随更高的维护负担、环境集成成本与组织依赖。

#### 低迁移价值

- 只有 prompt 包装、没有验证产物的 skills
- 高度依赖某个宿主或创始人风格的人设化流程
- 缺少契约、缺少 gate、缺少治理控制面的“体验型能力包”

这类对象通常适合参考，不适合直接照抄进严肃工程体系。

### 4.5 Context Files 与 Host Portability Layer

从迁移角度看，当前一个非常重要的现实判断是：

**仓库级配置工件正在成为最主流的落盘形态，而真正昂贵的是让它们跨宿主稳定生效的映射层。**

多项材料共同指向这一点：

- Context Files 仍是采用基线
- `AGENTS.md` 正在走向跨工具互操作
- 多宿主迁移不是手工复制，而是 installer + converter + tests
- 映射层是持续维护对象，而不是一次性脚本

这意味着企业在评估迁移价值时，不能只看“有没有这个能力”，还要看：

- 需要维护多少 adapter
- 需要覆盖多少目录与 frontmatter 差异
- 回归测试如何固化 contract
- 宿主升级后是否会带来 breakage

### 4.6 第四部分结论

这一部分可以收敛成五个判断：

1. [分析判断] 企业应迁移机制层，而不是命令名、语气或社区包装
2. [硬事实] 四个最稳健的高阶抽象是控制流分离、验证闭环前置、确定性门禁与对抗性审查、状态持久化
3. [分析判断] 迁移价值必须按净收益评估，而不是按“看起来先进”排序
4. [硬事实] Context Files 是当前主流落盘形态，而 portability layer 是真实维护成本中心
5. [趋势推测] 多宿主适配会越来越工程化，也会越来越成为框架作者与企业采用方的共同负担

## 第五部分：综合判断、证据边界与企业采用建议

### 5.1 当前可以稳健成立的结论

基于现有材料，以下结论可以视为高置信度判断：

1. 能力单元的本质是执行机制，而不是 prompt 包装
2. 规划阶段的核心价值是通过 gate 与验证前置阻止过早执行
3. 构建阶段最关键的结构性问题是 context rot，控制流分离是高价值解法
4. TDD enforcement、completion gate 与 systematic debugging 属于高迁移价值护栏
5. review、QA、health、canary、benchmark、checkpoint 等后半段单元，正在把质量和稳定性转成可持续资产
6. 迁移价值最高的对象，通常同时具备 determinism、auditability 与 portability

### 5.2 需要保持克制的推论

以下判断有较强依据，但仍需继续补证据：

1. **高迁移价值能力单元一定会带来更高 correctness 与质量净收益**  
   当前更能证明“机制存在”和“部分效率、覆盖率或降噪收益存在”，但端到端净收益仍缺更多企业复盘。

2. **`AGENTS.md` 等 context files 足以成为主治理平面**  
   更合理的判断是：它们适合作为入口与最小 requirements，而不是独自承载全部治理。

3. **多宿主适配层的长期维护成本可以被轻松控制**  
   当前已有 snapshot 与 churn 切片，恰恰说明它是持续维护对象，而非一次性迁移成本。

4. **MCP 工具链进入交付闭环后可以天然继承宿主安全边界**  
   现有公开材料反而说明，这一边界并不自动成立，尤其需要显式 allowlist、sandbox 与审批策略。

### 5.3 当前最值得警惕的五类失败模式

#### 1. 把能力单元误判成 prompt 包装

结果是看起来有工作流，实际上没有 gate、没有产物、没有约束，一长会话就退化。

#### 2. 把规划阶段当作“可以省掉的仪式”

结果是需求漂移、scope 变化与验证缺失被直接编码进实现。

#### 3. 把 builder 自审当作可靠 review

结果是 self-bias 被放大，真正有风险的问题更容易被遗漏。

#### 4. 把“测试跑过了”当作最终证明

弱测试、坏 oracle 与不充分 baseline 会让假完成长期隐藏。

#### 5. 把宿主适配和外部工具安全当成后补项

结果是 portability debt、审批绕过、MCP 风险与供应链问题在上线后集中爆发。

### 5.4 对企业的采用建议

#### 对工程架构与开发效能团队

- 先定义能力单元标准：什么算 gate、什么算 artifact、什么算治理控制面
- 先迁移四个抽象，再考虑具体框架外壳
- 对每个拟引入单元，显式评估 adoption complexity、governance burden、maintenance overhead、portability cost、team training cost
- 把 portability layer 和 regression tests 预算写进方案，不要假设“兼容一次就结束”

#### 对安全与平台治理团队

- 把 approval、sandbox、allowlist、签名、审计视为能力单元架构的一部分
- 对 MCP / 外部工具接入建立默认的信任边界，而不是靠框架作者自行声明安全
- 对运行时 QA、benchmark、health、canary 定义最小可接受 baseline 与存档方式

#### 对框架与内部平台作者

- 优先输出机制层资产，而不是命令名与叙事风格
- 让每个高价值能力单元都能回答：它改变了什么控制流、生成了什么产物、如何被验证、何时触发熔断
- 把 context files 控制在最小 requirements 层，把强治理逻辑下沉到可执行 gate

### 5.5 一个更成熟的总体判断

如果把整套研究压缩成一句话，可以表述为：

**AI Agent 在 SDLC 中的进步，核心不在“越来越会写代码”，而在“越来越能把执行、验证、审查、记忆与治理写成稳定的能力单元架构”。**

这也是为什么最值得长期追踪的对象，不是具体某个框架，而是这些可复用机制：

- 规划硬门禁
- 验证前置
- fresh-context execution
- TDD 与 completion gate
- adversarial review
- runtime QA 与 baseline 对照
- 文件型状态机
- portability layer
- 企业治理控制面

谁能把这些机制组合成既可审计、又可迁移、且治理成本可控的工程系统，谁就更可能在企业级 AI Agent SDLC 中形成真正稳定的优势。
