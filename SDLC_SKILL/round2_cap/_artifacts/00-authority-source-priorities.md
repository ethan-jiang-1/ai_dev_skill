# cap Wave Step 1: 权威来源优先级清单（可复用）

目标：按维度给出“先抓什么、后抓什么”的来源优先级，避免后续探索靠感觉扩张而失控；并记录当前已经落库的权威来源覆盖情况。

> 说明：这不是“来源清单越长越好”，而是一个**停止无意义扩张**的控制面。后续发现更高质量来源时可增补，但必须说明“为什么它替代/增强了现有来源”。

## 01-planning（能力单元本质与前置规划机制）

优先级（从高到低）：

- 框架官方仓库/官方 docs：定义能力单元、规划/门禁/验证前置的确定性机制（schema、脚本、模板、CI 规则、命令接口）。
- 框架作者的设计说明/发布说明：解释“为什么这样设计”与关键取舍。
- Empirical SE / HCI 的实证研究：规划、需求澄清、设计评审、早期验证在 agentic 开发中的效果与失败模式。
- 安全/质量工程视角：威胁建模、注入风险、权限最小化、门禁与审计。
- 社区真实使用反馈：落地摩擦、失败案例、误用模式（证据强度低，但用于捕捉反例与争议）。

## 02-build-debug（构建执行与系统化调试闭环）

优先级（从高到低）：

- 框架官方仓库/官方 docs：orchestrator/worker 分离、并发/波次执行、上下文净化、重试/熔断、TDD enforcement、debug protocol 的实现细节。
- 评测与回归工程：tests-as-harness、sandbox、reproducibility、trace/logging（用于把“能不能稳态”落到工程事实）。
- 实证研究/基准：对 agentic coding、debugging、multi-file changes 的成功率与失败模式评估。
- 安全/质量工程视角：工具调用权限、代码执行隔离、供应链与回滚。
- 社区反馈：真实项目中“越修越坏/无限重试/上下文腐化”的案例。

## 03-review-ship-ops（审查发布运维与状态持久化）

优先级（从高到低）：

- 框架官方仓库/官方 docs：review 解耦（critic/consensus）、质量门禁、release/ops 自动化、headless QA、canary/benchmark、文档同步与状态持久化文件的确定性机制。
- 安全研究与工具限制清单：代码审查/漏洞发现的能力边界、模型自审的系统性偏差、以及实际限制。
- 企业/组织实践：在 CI/CD、发布治理、变更管理里引入 agent 的可复核案例。
- 社区批评与失败复盘：误报/漏报、review 失效、监控缺失导致事故等。

## 04-map-migration（能力地图与迁移价值判断）

优先级（从高到低）：

- 能力地图/迁移评估的“硬证据”：真实采用案例、公开基准对比、或可复核的工程数据（对每条评级给出回指）。
- 跨框架对比的一手材料：各框架对同一能力单元的实现差异（代码/文档/接口），用于区分“通用抽象”与“专有实现”。
- 独立第三方分析：对迁移成本、风险面、组织约束的系统性评估（必须可复核，不采纳纯营销）。
- 失败模式与反例：迁移失败、成本爆炸、治理失效的证据或公开复盘。
