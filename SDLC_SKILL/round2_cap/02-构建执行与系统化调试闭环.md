# 构建执行与系统化调试闭环

## 第一轮摘要（保留，不修改）

## 这份片段在讲什么

这份上下文聚焦 SDLC 中间最容易失控的部分：

- Build 阶段怎么防止上下文腐化
- Test / Debug 阶段怎么避免模型进入盲猜和无限重试

## 从原始研究提炼出的核心结论

### 1. Build 阶段的核心问题不是“能不能写代码”，而是“能不能在长周期里持续写对”

原始研究指出，编码阶段真正的灾难来源是：

- 上下文腐化
- 变量和约束遗忘
- 长会话导致的智力基线下降

为了解这个问题，GSD 采用了一个非常强的结构：

- `Orchestrator`
- `Fresh Subagents`
- `Wave-Based Execution`

核心思想是：

- 编排器只负责调度，不亲自写代码
- 写代码的 worker 总是新的、纯净的上下文
- 任务按依赖拆成 wave 并行执行

这不是小优化，而是对进程模型的重构。

### 2. 编排与执行分离，是对抗 context rot 的最强抽象之一

原始研究把 GSD 的这套机制评价得很高，原因在于它直接解决了一个物理限制：

- 同一个超长上下文既决策又执行，质量一定会下滑

而 “编排器 + 新鲜子智能体” 的模式带来的好处是：

- Task 1 和 Task 50 的执行质量更接近
- 长会话积累的“思维垃圾”不会污染后续任务
- 可以自然支持并发波次执行

这类机制的迁移价值很高，  
因为它不依赖具体语言或技术栈。

### 3. 强制 TDD 的真正价值在于“改变模型习惯”

原始研究里的另一个关键对象是 `superpowers` 的 `RED-GREEN-REFACTOR` 强制闭环。

它最值得研究的不是“TDD 很重要”这种常识，  
而是它如何通过流程和回滚机制强迫模型遵守：

- 先写失败测试
- 再写最小实现
- 最后再重构

如果模型试图跳过测试、先写业务逻辑，  
系统会直接删除或回滚这部分代码。

这说明能力单元真正强的时候，  
不是“提醒你应该 TDD”，而是“不允许你不 TDD”。

### 4. 调试阶段最重要的是剥夺模型“马上改代码”的权限

原始研究对 `superpowers` 的 `4-Phase Systematic Debugging` 评价很高。

它的关键不是多了四步，而是执行权限被重排：

- 先收集信息
- 再隔离问题
- 再提出根因假设
- 最后只允许通过小范围验证来证伪或证实

在确认根因之前，模型不能碰生产代码。

这个抽象非常重要，  
因为它直接克制了 LLM 的急躁和瞎猜天性。

### 5. RARV Cycle 代表了“反思 + 验证 + 熔断”的另一条路线

原始研究还提到 GABBE / Loki Mode 一类的 `RARV` 循环：

- Reason
- Act
- Reflect
- Verify

这套机制值得继续研究的重点有两个：

- `Verify` 交给确定性脚本，不交给 LLM 自己判断
- 重试次数有上限，超限就熔断并要求人工介入

这说明高级能力单元不仅会验证，  
还会在自动化失败时优雅退出，而不是无限消耗资源。

## 二轮新增证据

- GSD 以“Orchestrator coordinates, not executes”为核心原则：execute-phase 负责依赖分 wave、spawn executors、处理 checkpoints、收集结果；并明确不同宿主的 runtime 兼容策略（例如 Copilot completion signal 不可靠时默认顺序执行，且必须用 git/文件 spot-check 判定完成，禁止无限等待）。`./reference_cap/02-gsd-execute-phase-workflow-wave-parallel-runtime-fallback.md`
- GSD 的编排层有明确的 context budget 规则：禁止把大文件 inline 进 prompts；read depth 随 context window 扩大；并列出 context rot 的早期行为信号（silent partial completion、vagueness、skipped steps）。`./reference_cap/02-gsd-reference-context-budget-rules.md`
- GSD 将“失败后修复”协议化为 node-repair operator：必须在 RETRY/DECOMPOSE/PRUNE/ESCALATE 中选一种，并有 repair budget（默认 2）与 SUMMARY 偏离日志协议。`./reference_cap/02-gsd-node-repair-autonomous-operator.md`
- GSD 将“先诊断 root cause，再计划修复”落为 diagnose-issues workflow：从 UAT gaps 结构化提取症状，按 gap 并行 spawn debug agents，回填 root cause 到 UAT.md，再交给 gaps-only planning。`./reference_cap/02-gsd-diagnose-issues-parallel-debug-agents.md`
- superpowers 明确提出 TDD 的硬门禁（Iron Law）：没有 failing test 不准写 production code；若提前写了必须删除重来。`./reference_cap/02-superpowers-test-driven-development-iron-law.md`
- GSD 将 TDD 协议化为 plan type：一 feature 一 TDD plan，RED/GREEN/REFACTOR 产出 2-3 个原子 commits，且在缺基建时把“测试框架搭建”纳入 RED。`./reference_cap/02-gsd-reference-tdd-plan-structure.md`
- superpowers 定义“完成宣称 gate”：没有新鲜验证证据（本消息内实际执行并阅读输出）就禁止宣称完成/通过。`./reference_cap/02-superpowers-verification-before-completion.md`
- 学术证据指出“验证信号可能是坏的”：SWE-Bench 的手写 tests 有时不足，导致错误 patch 也被标记为通过；UTBoost 用 test augmentation 揭示并修正大量 false positives。`./reference_cap/02-arxiv-2506.09289-utboost-swebench-test-augmentation.md`
- 近期趋势证据：调试能力正在从静态分析/日志走向“交互式动态分析”：
  - InspectCoder：interactive debugger control + dual-agent（inspector/coder）。`./reference_cap/02-arxiv-2510.18327-inspectcoder-interactive-debugger.md`
  - DebugHarness：主动查询 live runtime、watchpoints、闭环验证修复复杂漏洞（2026-04）。`./reference_cap/02-arxiv-2604.03610-debugharness-interactive-debugging-for-apr.md`
- 官方机制证据（agent loop 工程实现）：OpenAI 在官方 agent loop 描述中明确并发 sessions（concurrent shell sessions）、tool output cap（保留头尾并标注截断）、compaction（压缩上下文以延长任务续航/降低 context rot）、以及 skills 的分阶段加载管道（metadata→bundle→append instructions）。`./reference_cap/02-openai-responses-api-computer-environment-compaction-concurrency.md`

## 二轮新增机制理解

- “编排器 + fresh executor”真正解决的不是“并行更快”，而是**把长会话不可避免的认知退化从执行面隔离出去**；同时把“完成判定”外置为可复核证据（git 提交、SUMMARY 工件、文件存在与后续 verifier），从而降低 completion-signal/口头汇报带来的不确定性。`./reference_cap/02-gsd-execute-phase-workflow-wave-parallel-runtime-fallback.md`
- context rot 不仅发生在 worker 上，也会发生在 orchestrator 上；因此必须有编排层的 context budget 规则（不 inline 大文件、动态 read depth、识别早期退化信号），否则“编排器变成更大上下文垃圾桶”，最终同样失控。`./reference_cap/02-gsd-reference-context-budget-rules.md`
- “node-repair”把“修不好怎么办”变成确定性策略选择与预算熔断：RETRY 针对环境/执行问题，DECOMPOSE 纠正任务过粗，PRUNE 显式承认不可行并写清理由，ESCALATE 将架构决策交还人类。这个机制的本质是**对抗无限重试与盲改的资源黑洞**。`./reference_cap/02-gsd-node-repair-autonomous-operator.md`
- “先诊断再修复”的关键是把 WHAT 与 WHY 分离：UAT gaps 只记录症状（WHAT），debug agents 负责找证据与根因（WHY），修复计划由 gaps-only planning 基于根因生成，从流程上减少“凭感觉改代码”。`./reference_cap/02-gsd-diagnose-issues-parallel-debug-agents.md`
- “强制 TDD”的核心不是覆盖率，而是把接口与行为定义从实现之后提前到实现之前（test 先行），并且要求见过 fail 才能确定测试在测对东西；GSD 的 TDD plan 协议将其进一步工程化为提交历史可审计的 2-3 commits。`./reference_cap/02-superpowers-test-driven-development-iron-law.md` `./reference_cap/02-gsd-reference-tdd-plan-structure.md`
- “Verification Before Completion”是一种反幻觉机制：它把“完成”从叙事变成证据，直接针对 agent 的乐观偏差与跳步倾向。`./reference_cap/02-superpowers-verification-before-completion.md`
- UTBoost 揭示了更底层的失败模式：即便你有 verify commands，如果 tests 本身不足，也会把错误 patch 视为成功；因此 build/debug 闭环的上限取决于**验证信号（tests/oracles）质量**，而不是仅取决于执行策略。`./reference_cap/02-arxiv-2506.09289-utboost-swebench-test-augmentation.md`
- InspectCoder/DebugHarness 等工作表明：复杂 bug 的根因常在 symptom 之外，静态 trace/日志不足以定位；通过交互式动态分析获取“语义相关 runtime evidence”正在成为提升修复质量的主要方向。`./reference_cap/02-arxiv-2510.18327-inspectcoder-interactive-debugger.md` `./reference_cap/02-arxiv-2604.03610-debugharness-interactive-debugging-for-apr.md`
- OpenAI 官方实现侧的 output cap + compaction，等价于把“上下文是一种受控资源”写进运行时：通过截断/压缩避免长日志淹没推理，并把多步任务的关键状态显式保留下来；这与 GSD 的 context budget/rot signals 在方向上是一致的，只是一个是宿主实现、一个是框架协议。`./reference_cap/02-openai-responses-api-computer-environment-compaction-concurrency.md` `./reference_cap/02-gsd-reference-context-budget-rules.md`
- 并发 sessions 的工程意义不仅是更快，而是把“执行/探测/验证”拆成独立会话减少相互污染；它为 wave-based 并行提供了宿主级的可行性证据（并提示需要 scheduler/资源预算与冲突治理）。`./reference_cap/02-openai-responses-api-computer-environment-compaction-concurrency.md` `./reference_cap/02-gsd-execute-phase-workflow-wave-parallel-runtime-fallback.md`

## 二轮新增趋势与难点

- 趋势：并行与多 agent 编排开始显式考虑“宿主差异”（completion signal、工具 API 可用性），并提供顺序执行 fallback + spot-check 的确定性推进策略。`./reference_cap/02-gsd-execute-phase-workflow-wave-parallel-runtime-fallback.md`
- 趋势：调试从“看日志/看堆栈”向“交互式动态分析/可查询运行时事实”迁移，强调 stateful session 与过程奖励（process reward）。`./reference_cap/02-arxiv-2510.18327-inspectcoder-interactive-debugger.md` `./reference_cap/02-arxiv-2604.03610-debugharness-interactive-debugging-for-apr.md`
- 趋势：宿主层开始把“上下文治理”工程化（output cap、compaction、skills progressive loading），意味着“长程 build/debug 闭环”不再完全依赖 prompt 自律。`./reference_cap/02-openai-responses-api-computer-environment-compaction-concurrency.md`
- 难点：并行执行的收益受制于冲突（文件/锁/工具链）与治理（hooks/CI/权限）；必须在“速度”与“确定性门禁”之间做工程折中（例如 `--no-verify` 与集中跑 hooks）。`./reference_cap/01-gsd-architecture-orchestrator-agent-pattern.md`
- 难点：严格 TDD 与系统化调试会在缺测试基建、不可复现环境、或强时间压力场景下产生阻力；需要 Wave 0 脚手架与明确的降级策略（何时可 skip，何时必须 gate）。`./reference_cap/02-gsd-reference-tdd-plan-structure.md`
- 争议/失败模式：弱测试导致 false completion；即使“跑过测试”也可能没修好问题，因此必须持续强化 test/oracle（test augmentation、回归策略）。`./reference_cap/02-arxiv-2506.09289-utboost-swebench-test-augmentation.md`

## 当前判断（二轮综合后）

- **硬事实是什么：** 多个框架已把 build/debug 的关键步骤协议化为可执行/可审计的工作流（wave execution、node repair、TDD plan、debug workflow、完成验证 gate）。`./reference_cap/02-gsd-execute-phase-workflow-wave-parallel-runtime-fallback.md` `./reference_cap/02-gsd-node-repair-autonomous-operator.md` `./reference_cap/02-gsd-reference-tdd-plan-structure.md` `./reference_cap/02-superpowers-verification-before-completion.md`
- **背后的根本机制是什么：**
  - 用 fresh context 与 orchestrator/worker 分离对抗 context rot
  - 用 repair budget 与策略集合对抗无限重试
  - 用“证据优先的验证 gate”对抗假完成
  - 用更强 tests/oracles 对抗弱信号收敛。`./reference_cap/02-gsd-reference-context-budget-rules.md` `./reference_cap/02-gsd-node-repair-autonomous-operator.md` `./reference_cap/02-superpowers-verification-before-completion.md` `./reference_cap/02-arxiv-2506.09289-utboost-swebench-test-augmentation.md`
- **生态最近在往哪里演化：** 从静态调试→动态调试，从 outcome-only reward→过程奖励与交互式 runtime probing。`./reference_cap/02-arxiv-2510.18327-inspectcoder-interactive-debugger.md` `./reference_cap/02-arxiv-2604.03610-debugharness-interactive-debugging-for-apr.md`
- **采用/落地难点在哪里：** 并行执行的工程冲突、权限与隔离；测试基建缺失；以及把 gate 自动化到 CI 的成本。`./reference_cap/02-gsd-execute-phase-workflow-wave-parallel-runtime-fallback.md`
- **社区争议和失败模式在哪里：** “跑过现有测试 ≠ 修复正确”，弱测试会系统性高估成功；此外 completion signal 不可靠会导致编排挂死，需要 spot-check/工件化证据推进。`./reference_cap/02-arxiv-2506.09289-utboost-swebench-test-augmentation.md` `./reference_cap/02-gsd-execute-phase-workflow-wave-parallel-runtime-fallback.md`
- **哪些对象最值得继续追踪：**
  - 如何在企业环境实现“可查询 runtime facts”的调试工具链（debugger session、可回放、隔离）。`./reference_cap/02-arxiv-2604.03610-debugharness-interactive-debugging-for-apr.md`
  - test augmentation / oracle strengthening 的可操作方案与成本模型（从 UTBoost 到企业 CI）。`./reference_cap/02-arxiv-2506.09289-utboost-swebench-test-augmentation.md`
  - node-repair/diagnose-issues 这类“失败治理协议”与组织 escalation 对齐的最佳实践。`./reference_cap/02-gsd-node-repair-autonomous-operator.md`
