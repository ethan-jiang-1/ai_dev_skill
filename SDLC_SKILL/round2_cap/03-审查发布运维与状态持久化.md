# 审查、发布、运维与状态持久化

## 第一轮摘要（保留，不修改）

## 这份片段在讲什么

这份上下文聚焦 SDLC 后半段和底层支撑能力：

- Review 阶段怎么避免模型自我审查失真
- Ship / Operate 阶段怎么把 QA、发布、监控和文档同步自动化
- 全生命周期怎么通过文件状态持久化避免“空白石板”问题

## 从原始研究提炼出的核心结论

### 1. Review 阶段最致命的问题是“同一个模型审自己”

原始研究把这个问题讲得很透：

- Builder 再去 review 自己生成的代码，会产生回音壁
- LLM 会倾向于认可自己的逻辑
- 结果是安全漏洞和边界问题被系统性漏掉

因此最值得研究的机制不是“多做 review”，  
而是“把 review 从 builder 身上解耦”。

### 2. 对抗性审查是高价值能力单元

原始研究重点讨论了：

- `Adversarial Critic`
- `CONSENSAGENT`
- 盲审 + 魔鬼代言人机制

这些机制的共同点是：

- builder 结束后，critic 用全新上下文接手
- critic 只看需求和 diff，不继承 builder 的思路
- 审查不是泛泛点评，而是绑定打分卡和 blocking 问题

这类能力单元的价值非常高，  
因为它为自动化开发链条引入了必要的制衡机制。

### 3. Ship / Operate 阶段的关键是“从代码文本走向真实运行时”

原始研究在 gstack 的 `/qa`、`/ship`、`/canary`、`/benchmark` 这些单元上给出了很强的例子。

最值得提炼的点是：

- 用真实 Playwright 无头浏览器做交互验证
- 用视觉差异、控制台报错、性能指标做运行时 QA
- 发布后继续做 canary 和 benchmark 监控

这说明一个成熟能力单元不该只停留在 repo 内部，  
而要穿透到真实运行环境。

### 4. 文档同步本身也是重要能力单元

原始研究提到 `document-release` 这一类对象时，给了一个很实用的提醒：

- 代码变了，文档不更新，是很典型的漂移问题

因此，文档同步不只是收尾动作，  
而是发布闭环的一部分。

### 5. 文件型持久化记忆是目前最可靠的项目记忆机制

原始研究对 `PROJECT.md`、`STATE.md`、`PLAN.md`、`DECISIONS.md` 这一套模式评价非常高。

核心观点是：

- 不要指望 LLM 聊天记忆来保存项目状态
- 结构化 Markdown 文件才是高保真的状态机载体

这套机制的价值在于：

- 新智能体启动时可以快速重建上下文
- 项目决策能够被持续沉淀
- 支持回滚、溯源和阶段切换

这不是附属优化，  
而是全生命周期稳定性的底层支撑。

## 二轮新增证据

- gstack 将“预落地审查”写成 `/review` 工作流：scope drift 检测、checklist 驱动、specialist 分派、fix-first 流水线、文档漂移检查，以及默认开启的 adversarial review（fresh-context subagent + 可选 Codex）。并把 review 结果落盘成 JSONL，成为后续 `/ship` 的可追溯 gate 输入。`./reference_cap/03-gstack-review-skill-adversarial-and-specialists.md`
- gstack 将“运行时 QA”协议化为 `/qa`：用 browse daemon 做页面访问/交互验证/console error 抽取/perf 采样/截图证据，并通过模板化报告、issue taxonomy、health score rubric 与 baseline/regression 模式把 QA 结果资产化。`./reference_cap/03-gstack-qa-skill-runtime-verification.md` `./reference_cap/03-gstack-skill-browse-qa-routing.md`
- gstack 将“文档同步”提升为发布闭环能力单元 `/document-release`：after `/ship`, before merge，基于 diff 做逐文件 audit 与 auto-update/ask-user 风险分级，并硬约束 CHANGELOG 的安全编辑规则以防历史被覆盖。`./reference_cap/03-gstack-document-release-skill-doc-drift.md`
- gstack 将“状态持久化/会话交接”落为 `/checkpoint`：明确 HARD GATE 禁止实现代码，只采集 git 状态并写结构化 checkpoint 文件，支持跨 session/跨 branch 恢复。`./reference_cap/03-gstack-checkpoint-skill-state-persistence.md`
- gstack 将“质量门禁与趋势”落为 `/health`：显式化 Health Stack（可写回 CLAUDE.md），运行工具并按 rubric 打分，结果写入 health-history.jsonl 并做 trend analysis。`./reference_cap/03-gstack-health-skill-quality-gates.md`
- gstack 将“发布后早期监控”落为 `/canary`：baseline 捕获 + 60s 周期巡检 + 只对“变化”报警 + transient tolerance，并把 report/JSONL 落盘。`./reference_cap/03-gstack-canary-skill-post-deploy-monitor.md`
- gstack 将“性能回归检测”落为 `/benchmark`：采集浏览器侧性能与资源分解，写 baseline、对比 delta、阈值判定 regression/warning，并支持 trend 与 budget 检查，输出 md/json 报告资产。`./reference_cap/03-gstack-benchmark-skill-perf-regression.md`
- gstack 的 `/ship` 把从“本地完成”推进到“可交付 PR”的发布链条（preflight/test/review/versioning/changelog/PR），并将治理选项（telemetry/proactive/routing）显式化。`./reference_cap/03-gstack-ship-skill-preflight-telemetry.md`
- GSD 把“代码审查”做成 phase-scoped 能力单元 `gsd:code-review`：可配置 review depth、内置 config gate、输出 REVIEW.md 产物，并用独立 reviewer agent 执行审查。`./reference_cap/03-gsd-code-review-command-and-artifact.md`
- GSD 把“验证是否真的工作”做成 `gsd:verify-work`：对话式 UAT 一条一条验证可观察 truths，结果写入 `{phase}-UAT.md`（含 gaps YAML，供后续 planning/execution 消费），并配套 VALIDATION 合约与 verification-report 模板。`./reference_cap/03-gsd-verify-work-uat-and-verification-artifacts.md`
- GSD 把“长程不中断”做成 `pause-work` + `context monitor` hook：hook 在 context 低时注入 WARNING/CRITICAL 并提示自然停点运行 `/gsd-pause-work`；pause-work 生成 `.continue-here.md` handoff 文件并做 WIP commit。`./reference_cap/03-gsd-pause-work-and-context-monitor.md`
- 学术证据（PR-centric code review）：SWR-Bench 提出 1000 PR + full context 的 benchmark，并验证 multi-review aggregation 可显著提升 issue detection F1（摘要给出 up to +43.67%）。`./reference_cap/03-arxiv-2509.01494-swr-bench-llm-code-review-benchmark.md`
- 学术证据（review comment 理解/定位）：CodeReviewQA 将 ACR 拆成 CTR/CL/SI probes，指出 review comment 的隐含性/口语性要求模型同时理解代码与人类意图，且 line-level localisation 仍是弱项。`./reference_cap/03-arxiv-2503.16167-codereviewqa-code-review-comprehension.md`
- 学术证据（自审偏差机制）：self-refine 可能放大 self-bias（模型倾向偏好自身输出），外部反馈能缓解偏差并带来更真实的改进。`./reference_cap/03-arxiv-2402.11436-llm-self-bias-self-refinement.md`
- 工业实证（质量门禁降噪）：在腾讯的真实静态分析流水线中，false positives 会造成单 alarm 10-20 分钟级人工审查成本；LLM+静态分析 hybrid 在“高 recall”约束下可消除 94%-98% 假阳性，且 per-alarm 时间/金钱成本为秒级与低美元量级（摘要口径）。`./reference_cap/03-arxiv-2601.18844-reducing-false-positives-static-bug-detection-industry.md`
- 工业现场研究（LLM-assisted code review 工作流）：WirelessCar 的 field study + field experiment 指出传统 review 的摩擦点（context switching、缺上下文）与 LLM 辅助的关键顾虑（false positives、trust issues），并对比 AI-led upfront review 与 on-demand interactive 两种模式，强调偏好取决于代码库熟悉度与 PR 严重程度；并采用 RAG/semantic search 拼装上下文。`./reference_cap/03-arxiv-2505.16339-rethinking-code-review-workflows-llm-assistance.md`
- 官方安全机制证据（宿主级门禁）：Codex 文档将 approval policy 与 sandbox_mode（read_only/workspace_write/dangerous）作为可配置 contract，并披露本地与云端隔离实现（macOS Seatbelt、Linux bubblewrap+seccomp、cloud gVisor）。`./reference_cap/03-openai-codex-agent-approvals-security-sandbox.md`
- 官方组织级实践（review loop + 文档治理）：OpenAI 的 Harness 工程复盘强调 docs/ 作为系统事实源、AGENTS.md 作为知识库 TOC、以及 PR/review loop 与持续 doc gardening（文中提到 1,500+ PRs、20+ engineers 的量级）。`./reference_cap/03-openai-harness-engineering-agent-first-world.md`
- 官方规范（MCP 安全最佳实践）：MCP spec 明确列出 confused deputy、token passthrough、SSRF、session hijacking/prompt injection、local server compromise 等威胁，并强调 least privilege scopes 与 progressive authorization、以及把 server/tool 输出视为不可信输入。`./reference_cap/03-mcp-security-best-practices.md`
- 学术证据（MCP/A2A 对抗风险）：Breaking the Protocol 分析 MCP 与 agent-to-agent 通信的安全风险，并强调多恶意 server 组合可能显著放大攻击成功率（摘要量级 10%→97%），并提出协议扩展与评估。`./reference_cap/03-arxiv-2601.17549-breaking-the-protocol-mcp-security.md`
- 学术证据（tool poisoning 语义攻击）：Securing MCP 提出 tool poisoning / shadowing / rug pulls 三类攻击，并给出分层防御（manifest signing、semantic vetting、runtime guardrails）的评估框架与量化结果。`./reference_cap/03-arxiv-2512.06556-securing-mcp-tool-poisoning.md`

## 二轮新增机制理解

- “审查解耦”的关键不是换一个口吻做评论，而是**引入独立上下文与独立 gate 输入**：gstack 的 adversarial subagent 明确强调 fresh context 与 attack/chaos 视角，并把跨模型发现做 synthesis；这相当于把“review 质量”从 builder 叙事中剥离出来。`./reference_cap/03-gstack-review-skill-adversarial-and-specialists.md` `./reference_cap/03-arxiv-2402.11436-llm-self-bias-self-refinement.md`
- “review 资产化”的关键是持久化：gstack 把 review 结果写入 review log（JSONL，含 findings/severity/quality_score/specialists stats/commit），使 `/ship` 等后续能力单元能“读状态”而不是靠聊天记忆。`./reference_cap/03-gstack-review-skill-adversarial-and-specialists.md`
- “运行时 QA”之所以高杠杆，是因为它把验证信号从“文本”迁移到“可观察事实”：截图、console errors、perf、交互前后差异（`snapshot -D`），并要求 issue 立即落盘为 report；baseline/regression 进一步把一次性 QA 变成回归资产。`./reference_cap/03-gstack-qa-skill-runtime-verification.md`
- “文档同步”能力单元的本质是防漂移：通过 diff-aware per-file audit 把“文档是否仍与系统一致”变成可执行任务；对 CHANGELOG 的硬规则则是降低 agent 覆写历史的高风险动作空间。`./reference_cap/03-gstack-document-release-skill-doc-drift.md`
- “发布后监控/基准”能力单元的共同机制是 baseline 对照：只对“变化”报警（canary），用 delta+阈值判定回归（benchmark），并把结果落盘成 report/JSON/JSONL，形成可审计的时间序列。`./reference_cap/03-gstack-canary-skill-post-deploy-monitor.md` `./reference_cap/03-gstack-benchmark-skill-perf-regression.md`
- “长程稳定性”依赖文件型状态机：gstack `/checkpoint` 的 no-code hard gate 与结构化 checkpoint 格式，GSD 的 `.continue-here.md` handoff 与 context monitor hook，都是把“上下文耗尽/会话中断”从偶发事故变成可治理的流程节点。`./reference_cap/03-gstack-checkpoint-skill-state-persistence.md` `./reference_cap/03-gsd-pause-work-and-context-monitor.md`
- 学术与框架的一致信号：multi-review aggregation 在 benchmark 上可显著提升 issue detection；对应到工程实践就是“specialists + adversarial passes + synthesis”的组合，而不仅仅是“多说几句”。`./reference_cap/03-arxiv-2509.01494-swr-bench-llm-code-review-benchmark.md` `./reference_cap/03-gstack-review-skill-adversarial-and-specialists.md`
- “审批策略 + 沙箱模式”是把 ops 风险外移为确定性门禁的关键控制面：它把“能不能执行/能执行到什么程度”从 prompt 约束升级为宿主级 policy（并且与 OS/云隔离实现绑定），因此属于企业迁移时必须纳入的能力单元（否则 review/ship/ops 无法合规落地）。`./reference_cap/03-openai-codex-agent-approvals-security-sandbox.md`
- “docs/ 作为系统事实源 + AGENTS.md 作为 TOC”的价值在于：把 review/ship/ops 的关键口径从聊天记忆迁移到可审计知识库，并使“更新知识库”成为 PR/review loop 的常规动作（doc gardening），从而降低长期漂移。`./reference_cap/03-openai-harness-engineering-agent-first-world.md`
- MCP 的安全治理不能停留在“不要 prompt injection”：spec 与研究共同指向更系统的威胁模型（confused deputy、SSRF、tool poisoning、rug pulls、多 server 放大），因此必须把 scopes/allowlist/signing/vetting/guardrails 纳入 ops 的 gate 设计（否则外部工具链会绕开 review/verify）。`./reference_cap/03-mcp-security-best-practices.md` `./reference_cap/03-arxiv-2512.06556-securing-mcp-tool-poisoning.md` `./reference_cap/03-arxiv-2601.17549-breaking-the-protocol-mcp-security.md`

## 二轮新增趋势与难点

- 趋势：code review 的评估与能力建设正在从“snippet/单 diff hunk + text-match 指标”走向“PR-centric + full project context + 更接近人评的覆盖式评价”；这会推动企业更重视“review artifacts 与可追溯门禁”。`./reference_cap/03-arxiv-2509.01494-swr-bench-llm-code-review-benchmark.md`
- 趋势：LLM 的 review 价值正在从“生成建议”扩展到“补上下文与降噪”：工业现场显示 AI-led upfront vs on-demand interactive 的交互形态在探索中，且 RAG/semantic search 被用于组装项目语境；但 trust/false positives 仍是主要 adoption 约束。`./reference_cap/03-arxiv-2505.16339-rethinking-code-review-workflows-llm-assistance.md`
- 趋势：review comment 理解能力被拆成可测的中间环节（CTR/CL/SI），并显式讨论 data contamination；意味着企业在引入自动 review 时，不能只看最终 patch，还要看“定位/理解/方案选择”的失败模式。`./reference_cap/03-arxiv-2503.16167-codereviewqa-code-review-comprehension.md`
- 趋势：多 reviewer 聚合（multi-review aggregation）与 cross-model synthesis 被证明能显著提升覆盖；工程上对应“adversarial pass + specialist army + 汇总”。`./reference_cap/03-arxiv-2509.01494-swr-bench-llm-code-review-benchmark.md` `./reference_cap/03-gstack-review-skill-adversarial-and-specialists.md`
- 趋势：宿主与企业产品开始把“执行安全”写成可配置 contract（approval policy / sandbox_mode / 隔离实现），推动 review/ship/ops 从“靠人盯”走向“policy 驱动的门禁”。`./reference_cap/03-openai-codex-agent-approvals-security-sandbox.md`
- 趋势：LLM+静态分析 hybrid 被用于 false positive elimination，把 SAT 告警 triage 从“人力分钟级”压到“秒级 API 成本”，可作为企业质量门禁体系的一部分（但仍需对 recall/漏报做治理）。`./reference_cap/03-arxiv-2601.18844-reducing-false-positives-static-bug-detection-industry.md`
- 难点：运行时 QA/监控/基准强依赖环境可达性与认证（SSO/2FA/CAPTCHA），并要求 baseline 维护；一旦环境漂移或 baseline 不代表真实场景，就会产生误报/漏报。`./reference_cap/03-gstack-qa-skill-runtime-verification.md` `./reference_cap/03-gstack-canary-skill-post-deploy-monitor.md`
- 难点：adversarial review 与多 pass 带来成本与时延，需要风险分级与阈值策略（diff size、变更类型、组件敏感度）来治理。`./reference_cap/03-gstack-review-skill-adversarial-and-specialists.md`
- 难点：当能力单元通过 MCP 外移到外部 server，攻击面从“repo 内 prompt”扩大到“授权 scopes + token 流转 + tool descriptors 语义”，且存在多 server 放大效应；企业必须引入 allowlist、签名、语义 vetting 与 runtime guardrails，否则 review/ship 的门禁可能被绕开。`./reference_cap/03-mcp-security-best-practices.md` `./reference_cap/03-arxiv-2512.06556-securing-mcp-tool-poisoning.md` `./reference_cap/03-arxiv-2601.17549-breaking-the-protocol-mcp-security.md`
- 争议/失败模式：self-refine 可能放大 self-bias，导致“看起来更顺”但并未更正确；因此必须引入外部评估信号或独立 reviewer，不能把 builder 自审当作可靠 gate。`./reference_cap/03-arxiv-2402.11436-llm-self-bias-self-refinement.md`
- 争议/失败模式：模型对 review comment 的定位与理解可能系统性失败（尤其 localisation），导致审查覆盖缺口；这要求把验证落到“可观察 truths + 回归资产”，而不是只依赖自然语言评价。`./reference_cap/03-arxiv-2503.16167-codereviewqa-code-review-comprehension.md` `./reference_cap/03-gsd-verify-work-uat-and-verification-artifacts.md`

## 当前判断（二轮综合后）

- **硬事实是什么：** 多个框架已把 Review/QA/Docs/Health/Canary/Benchmark/Verify/Pause-Resume 写成可执行工作流，并将关键结果落盘为 artifacts（JSONL/报告/模板化文件），不再依赖聊天记忆。`./reference_cap/03-gstack-review-skill-adversarial-and-specialists.md` `./reference_cap/03-gstack-qa-skill-runtime-verification.md` `./reference_cap/03-gstack-document-release-skill-doc-drift.md` `./reference_cap/03-gsd-verify-work-uat-and-verification-artifacts.md` `./reference_cap/03-gsd-pause-work-and-context-monitor.md`
- **硬事实是什么（补充）：** 工业实践中“质量门禁的最大噪音源”之一是 SAT 的高假阳性告警；LLM+静态分析 hybrid 在企业数据上显示出显著降噪潜力与可控成本模型，但必须以 recall/漏报治理为前提。`./reference_cap/03-arxiv-2601.18844-reducing-false-positives-static-bug-detection-industry.md`
- **背后的根本机制是什么：**
