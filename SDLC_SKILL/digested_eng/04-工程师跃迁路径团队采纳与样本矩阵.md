# 工程师跃迁路径、团队采纳与样本矩阵

## 第一轮摘要（保留，不修改）

## 这份片段在讲什么

这份上下文聚焦从个人成长到团队治理的最后一层：

- 工程师如何从 skill 新手一路成长到方法学创造者
- 团队如何共享、版本化和采纳 skill
- 哪些代表性样本最适合“先解决问题，再反向学习方法学”

## 从原始研究提炼出的核心结论

### 1. 工程师掌握 skill 可以被描述成四阶段跃迁

原始研究把成长路径划成四步：

- 破冰与引导执行
- 上下文协作与约束适应
- 规范驱动的系统设计
- 经验封装与组织赋能

这四步对应的变化是：

- 从害怕 AI 到敢用 AI
- 从写 prompt 到管理上下文和约束
- 从写代码到定义 spec 和 plan
- 从 skill 使用者到 skill 作者和治理者

### 2. 高阶段成长的关键不是写更多代码，而是定义更好的规范

原始研究的一个核心判断是：

- 在现代 AI 工程里，代码本身越来越便宜
- 真正稀缺的是需求抽象、系统边界、验证协议和组织知识显性化

所以工程师往后走，  
本质上是在从“执行者”转向“规范制定者”。

### 3. 团队级 skill 必须像代码一样治理

原始研究在团队层面强调了几件事：

- skill 要版本化
- 要共享
- 要避免多项目复制粘贴带来的碎片化

它给出的几种治理思路包括：

- 项目内直接管理规则
- 集中式 skill 仓库
- submodule / monorepo 等共享机制

这说明 skill 一旦进入团队，就不再只是个人习惯，  
而是组织级资产。

### 4. 开源 skill 的采纳策略要分三类

原始研究给出的三分法非常适合继续研究：

- 直接用
- 改着用
- 作为灵感重写

并且这个判断依赖于 skill 的性质：

- 通用工具链类更适合直接用
- 代码审查、架构约束类更适合改着用
- 高复杂度、企业私有流程类更适合重写

### 5. 样本矩阵是“认知脚手架”最直接的落点

原始研究最后给了一组很有代表性的样本：

- GitHub Spec Kit
- 代码审查类 skill
- TDD guide / builder
- Git 语义化提交 skill
- Cursor 深度防幻觉规则

这些样本的价值不只是能用，  
而是很适合拿来观察：

- skill 怎么训练人
- 哪些适合新手
- 哪些适合进阶者
- 哪些适合团队沉淀

## 二轮新增证据

- 团队级“指令资产治理”存在官方可执行机制：repo-wide 与 path-specific 的 custom instructions，并存在明确的优先级与承载文件（含 `AGENTS.md`）。（ref: `../reference_eng/04-path-github-copilot-repo-custom-instructions.md`）
- Copilot CLI 的 tool permissions 模型提供了可落地的“允许/拒绝/需批准/YOLO”控制面，可作为团队治理原语。（ref: `../reference_eng/04-path-github-copilot-cli-tool-permissions.md`）
- Copilot CLI 的 trusted folders/config 机制提供了一个“信任边界”的工程入口。（ref: `../reference_eng/04-path-github-copilot-cli-trusted-folders-config.md`）
- Copilot 的组织/企业策略概念（feature/privacy/models 等）说明治理可从个人偏好上升到组织级政策。（ref: `../reference_eng/04-path-github-copilot-policies-concepts.md`）
- Copilot cloud agent 的官方说明展示了企业级 agentic workflow 的形态：在 GitHub 上自主研究、计划、改代码并可开 PR；运行于 GitHub Actions 的 ephemeral 环境，同时列出限制（含 content exclusions 不被 account 等）。（ref: `../reference_eng/04-path-github-copilot-cloud-agent-about.md`）
- Copilot cloud agent 的 pilot guide 给出了试点 rollout playbook（权限/配置/流程），可作为“团队采纳路径”的硬证据。（ref: `../reference_eng/04-path-github-copilot-cloud-agent-pilot-guide.md`）
- OWASP LLM Top 10 提供了风险分类语言（prompt injection、supply chain、excessive agency 等），可作为团队治理章节的风险地基。（ref: `../reference_eng/04-path-owasp-top-10-llm-apps-v1-1.md`）
- Cursor 的 team marketplace/插件治理官方信号提示：生态可能在向“私有分发 + 团队集中治理”演化（趋势信号，需更多对照验证）。（ref: `../reference_eng/04-path-cursor-plugins-marketplace-team-governance-2026-02-17.md`）

## 二轮新增机制理解

### 1) 团队采纳的关键不是“选哪个模型”，而是“是否具备可执行控制面”

- 工具层的治理原语（指令资产、权限模型、trusted folders、组织策略）意味着团队可以把 Skill/agent 的使用从“个人习惯”升级到“可审计的工程制度”。（ref: `../reference_eng/04-path-github-copilot-repo-custom-instructions.md`, `../reference_eng/04-path-github-copilot-cli-tool-permissions.md`）
- cloud agent 把 agent 工作过程落在 commits/PR/logs 里，并在 Actions 环境运行，这天然适配审计与 review，但也引入平台限制与新风险面（例如 content exclusions）。（ref: `../reference_eng/04-path-github-copilot-cloud-agent-about.md`）

### 2) 风险治理需要共同语言：从“担心”到“可枚举风险面”

- OWASP LLM Top 10 的价值在于把风险拆成可枚举类别，便于把治理对策映射到具体控制点（权限、隔离、审计、供应链等），避免讨论停留在“AI 不安全”的泛论。（ref: `../reference_eng/04-path-owasp-top-10-llm-apps-v1-1.md`）

### 3) “样本矩阵”的机制要求：样本必须能让人复盘规则与决策链

- 第一轮的样本矩阵想服务“逆向学习”，二轮的机制约束是：样本不仅要能跑，还要能让人看懂其规则/边界/验证协议，并能被团队治理与回归（与 03-devlife 的 eval/稳态交叉）。
- 目前我们在 eng 的 evidence 中已经补齐治理控制面，但“高质量样本仓库的长期复盘”仍不足（需在报告中标注缺口）。

## 二轮新增趋势与难点

- 趋势：agentic workflow 正在从 IDE 内走向 GitHub 平台化（cloud agent + pilot playbook），把工作过程变成可审计资产。（ref: `../reference_eng/04-path-github-copilot-cloud-agent-about.md`, `../reference_eng/04-path-github-copilot-cloud-agent-pilot-guide.md`）
- 趋势（信号）：工具生态可能在向“私有 marketplace + 团队集中治理”演化，以解决插件/技能供给的安全与一致性问题。（ref: `../reference_eng/04-path-cursor-plugins-marketplace-team-governance-2026-02-17.md`）
- 难点：平台限制会成为治理漏洞来源（例如 content exclusions 不生效）；团队必须在试点阶段显式识别并补救，而不是默认“政策能自动生效”。（ref: `../reference_eng/04-path-github-copilot-cloud-agent-about.md`）
- 难点：企业落地案例通常不公开细节；很多“成功经验”很难被第三方复核，需要用控制面事实与风险分类来替代软叙事，且在报告中标注证据强度。

### 1) 六个固定问题回答（二轮）

1. 这个主题当前的硬事实是什么
   - 存在可执行治理控制面（指令资产、权限模型、trusted folders、组织策略）与平台级 agent 工作流（cloud agent + pilot playbook）。（ref: `../reference_eng/04-path-github-copilot-repo-custom-instructions.md`, `../reference_eng/04-path-github-copilot-cloud-agent-about.md`, `../reference_eng/04-path-github-copilot-cloud-agent-pilot-guide.md`）
2. 背后的根本机制是什么
   - 团队采纳要可持续，必须把 Skill/agent 变成“可审计、可回滚、可控权限”的工程资产；风险治理需要可枚举分类语言与对应控制点。（ref: `../reference_eng/04-path-owasp-top-10-llm-apps-v1-1.md`）
3. 生态最近在往哪里演化
   - 向平台化与可审计的异步 agentic SDLC 演化（GitHub 上的 cloud agents）；并出现私有分发与集中治理的趋势信号。（ref: `../reference_eng/04-path-github-copilot-cloud-agent-about.md`, `../reference_eng/04-path-cursor-plugins-marketplace-team-governance-2026-02-17.md`）
4. 采用或落地的难点在哪里
   - 关键难点是把控制面组合成可执行 SOP（谁能改指令、如何 review、如何灰度、如何回滚），并处理平台限制带来的治理漏洞（content exclusions 等）。（ref: `../reference_eng/04-path-github-copilot-cloud-agent-about.md`）
5. 社区争议和失败模式在哪里
   - 失败模式集中在：权限过大导致过度代理、提示注入/供应链风险、插件生态污染、以及“政策/排除规则未生效”的错觉。（ref: `../reference_eng/04-path-owasp-top-10-llm-apps-v1-1.md`, `../reference_eng/04-path-github-copilot-cloud-agent-about.md`）
6. 哪些对象最值得继续追踪
   - 可复核的企业试点复盘（含指标与失败模式），以及把 OWASP 风险分类落到治理 checklist 的公开实践。
   - “高质量样本矩阵”仓库：同时具备规则可读性、回归评测、版本治理与团队审计链条。
