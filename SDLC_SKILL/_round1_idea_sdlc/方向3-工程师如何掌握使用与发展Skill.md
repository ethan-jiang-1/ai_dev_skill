# 方向3：工程师如何掌握、使用与发展 Skill

## 任务

这份 Deep Research 只研究一件事：

**对不熟悉 AI 和方法学的工程师来说，skill 怎样帮助他先解决问题，再逐步学会方法、提高工程能力？**

这里的重点不是安装和迁移，  
而是“人怎么适应 skill，skill 又怎么反过来训练人”。

## 一个关键认识

`skill` 可以先理解成高级提示词或高级上下文，  
但好的 skill 不只是 prompt。

它通常还封装了：

- 对问题的拆解方式
- 对执行顺序的约束
- 对验证和 review 的要求
- 对常见失败模式的预防

所以 skill 既是可借用的 AI 能力，  
也是被写进文本和流程里的工程经验。

## 研究边界

优先看：

- 工程师如何发现、理解、调用、组合、修改 skill
- skill 如何作为学习脚手架，帮助人从“不会用 AI”走向“会设计方法”
- 不同工具怎样支持 skill 的学习、调试、分享和开发
- 开源 skill 更适合直接用、改着用，还是作为灵感重写

默认弱化：

- 纯安装说明
- 只谈理念、不谈学习路径的文章
- 与软件开发关系很弱的泛 AI 技巧

## 要看什么

研究时重点判断：

- 新手第一次接触 skill 的门槛是什么
- 哪些 skill 适合先拿来解决问题，再回头学习其写法
- 工程师需要哪些能力才能稳定使用 skill
- 哪些能力本身可以通过长期阅读和使用 skill 被训练出来
- 不同工具在 skill 的发现、调用、调试、改写、版本化方面支持如何
- 工程师应该优先直接用开源、局部改造，还是自建 skill

## 难度分层

请尽量把 skill 分成三层来看：

- `特别简单`：拿来即用，适合新手快速见效
- `中等简单`：需要理解场景和边界，适合日常稳定复用
- `复杂`：需要组合、改写或自己开发，适合进阶用户和团队

同时判断每一层分别要求什么能力，  
又能反过来训练出什么能力。

## 交付

最终输出尽量包含：

- 一条工程师掌握 skill 的路径图
- 哪些 skill 适合新手，哪些适合进阶者，哪些适合团队
- 不同工具对 skill 学习和发展的支持比较
- 对采用路径的建议：直接用、改着用、自建、仅作启发
- 哪些 skill 最适合作为“先解决问题，再反向学习方法学”的样本

如果需要方便后续合并，可选记录这些最小字段：

- `name`
- `url`
- `difficulty_level`
- `adoption_path`
- `inspiration_value`
- `notes`

## 搜索方向

优先沿这些方向找：

- 学习路径：`quickstart`、`tutorial`、`walkthrough`、`how to use skills`
- 用户证据：issues、discussions、博客、视频、真实案例、踩坑总结
- 工具支持：skills discovery、commands palette、hooks、subagents、workflow builder、sharing
- 作者化：`custom skill`、`create a skill`、`extend workflow`、`agent customization`
- 学习型线索：`learn by reading skills`、`skill as scaffolding`、`engineering practices encoded in prompts`

研究时要始终记住：  
skill 不只是帮人完成任务，  
也可能是帮助工程师学会更好做事的方法。
