## 关键事实

- 研究对象：WirelessCar Sweden AB（工业公司）代码审查工作流。
- 方法：两阶段经验研究（论文摘要口径）：
  - Phase 1：半结构化访谈的 field study，用于识别现有 code review 的挑战与 AI 介入机会。
  - Phase 2：field experiment，评估两种 LLM-assisted review 工具变体在真实 review 任务中的开发者交互与偏好。
- 识别的主要挑战（摘要口径）：频繁 context switching、缺少足够 contextual information。
- 机会与风险（摘要口径）：机会包括对复杂 PR 的自动 summarization；风险/担忧包括 false positives 与 trust issues。
- 工具形态（摘要口径）：两种模式
  - AI-led：LLM upfront 给出 review/summary，reviewer 再跟进。
  - on-demand：reviewer 主动提问，LLM 作为交互式助手。
- 关键工程机制：使用 retrieval-augmented generation（RAG）/语义检索管道来组装相关上下文，缓解“缺上下文导致建议空泛”。
- 结果趋势（摘要口径）：AI-led reviews 总体更受偏好，但偏好受 codebase 熟悉度与 PR 严重程度影响（context-dependent）。

## 与本研究的关系

- 补 `digested_cap/03` 的“争议与失败模式”：
  - LLM 评审的核心风险不是“能力不足”一句话，而是 false positives 与信任边界，且与 reviewer 对代码熟悉度高度相关。
- 补 `digested_cap/04` 的“企业迁移价值评估维度”：
  - team training cost 与 governance burden 在 review 侧体现为：何时信任 AI、如何把 AI 输出变成可审计证据、以及如何降低 reviewer 的注意力分散。
