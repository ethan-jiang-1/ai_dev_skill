## 关键事实

- 提出 CodeReviewQA benchmark：900 条人工校验的真实 code reviews，覆盖 9 种语言、199 个 GitHub 仓库；评估 72 个近期模型。
- 将 ACR（自动根据 review comment 改代码）的 generative task 拆成 3 个 reasoning probes：
  - CTR：change type recognition
  - CL：change localisation
  - SI：solution identification
- 通过 MCQA reformulation 试图降低 data contamination 风险，并提供更细粒度的失败诊断信号。
- 在 limitation 中明确：dataset 规模受限于人工验证成本；并指出很多模型在 line-level localisation 上仍很吃力。

## 与本研究的关系

- 为 `digested_cap/03` 的“review 能力单元需要把评论理解/定位/解决方案分离，并做独立验证”提供学术侧证据。
- 为“为什么要做 adversarial review + persisted review artifacts”提供间接支撑：传统单一指标与单一 pass 容易掩盖失败模式。

## 风险与局限

- benchmark 的 MCQA reformulation 主要缓解 surface-level contamination；语义层面的污染仍可能存在（论文 limitation 也明确提到）。
- 结论面向“理解 review comment 的能力评估”，不直接等价于“某个具体 review workflow 在真实团队里更好”；仍需结合工程门禁与运行时验证闭环。
