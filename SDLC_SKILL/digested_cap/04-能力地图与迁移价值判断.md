# 能力地图与迁移价值判断

## 第一轮摘要（保留，不修改）

## 这份片段在讲什么

这份上下文聚焦原始研究最后的抽象总结：

- 哪些能力单元最值得形成“能力地图”
- 哪些是高迁移价值抽象
- 哪些更偏包装层，不值得照抄

## 从原始研究提炼出的核心结论

### 1. 不要把“有名字的 skill”误认为“有价值的能力抽象”

原始研究最后给出一个非常关键的判断：

- 开源社区里很多所谓 agent skills，本质上只是 prompt 包装
- 它们对系统执行流程没有实质约束力
- 上下文一长、模型一漂移，就会失效

因此，真正值得迁移的不是名字和人设，  
而是底层工程抽象。

### 2. 高迁移价值的能力单元有明显共性

原始研究总结出的高价值对象包括：

- `Nyquist Validation Layer`
- `Orchestrator & Wave Parallelism`
- `RED-GREEN-REFACTOR Enforcement`
- `4-Phase Systematic Debugging`
- `Adversarial Critic / CONSENSAGENT`
- `File-Based Knowledge Persistence`
- `Playwright QA & Canary Watcher`

这些对象的共同点是：

- 直接干预控制流
- 限制 agent 的动作空间
- 引入确定性验证
- 打破单模型自我封闭
- 支持长周期和跨会话稳定运行

### 3. 原始研究最后沉淀出了四个最核心的工程抽象

这是整篇文档最值得继续研究的部分：

1. `控制流分离`
2. `验证闭环前置`
3. `确定性门禁 + 对抗性审查`
4. `状态持久化`

这四个抽象，比任何单个 skill 名字都重要。

因为它们回答的是：

- 系统怎么组织
- 质量怎么保证
- 漂移怎么约束
- 记忆怎么保存

### 4. 低迁移价值对象往往带有过强风格化包装

原始研究以 gstack 的 YC 风格需求澄清为例，做了一个很有用的区分：

- 某些对象在原框架里很有表现力
- 但迁移到企业环境时可能适配性差

这提醒我们：

- 不要把“好玩”误判成“适合复用”
- 不要把带强烈创始人风格的流程直接抄进严肃工程体系

### 5. 真正应该迁移的是“机制”，不是“语气”

整篇研究最后其实落到了一个很朴素但很重要的判断：

- 借鉴能力单元时，应优先迁移机制层
- 少迁移表演性包装、人设、口吻和特定文化风格

也就是：

- 借鉴“先验证再执行”
- 借鉴“用全新 critic 审查”
- 借鉴“用文件持久化状态”

而不是简单复制某个框架里的花哨命令名。

## 二轮新增证据

- OSS 大样本采用基线：仓库级配置机制以 Context Files 为主，且 `AGENTS.md` 正在成为跨工具互操作标准；同时 Skills/Subagents/Hooks/MCP 等高级机制整体采用仍浅。`reference_cap/04-arxiv-2602.14690-configuring-agentic-ai-coding-tools.md`
- 可复核的效率实证：在 paired within-task 设计下，`AGENTS.md` 的存在与更低的中位数 runtime（约 -28.64%）和 output tokens（约 -16.58%）相关（研究明确不把 correctness 作为结论）。`reference_cap/04-arxiv-2601.20404-impact-of-agents-md-efficiency.md`
- 迁移需要“安装器 + 目录契约 + 转换器 + 测试”而非手工复制：GSD 支持安装到 `.github/` `.codex/` `.cursor/` `.windsurf/` `.clinerules` 等多运行时，并通过 conversion tests 固化 rewrites。`reference_cap/04-gsd-multi-runtime-installer-and-format-conversion.md`
- “多宿主可移植”在代码层需要强类型 HostConfig 与 host adapter：gstack 用 allowlist/denylist、path/tool rewrites、suppressed resolvers 与语义级 adapter 治理宿主差异。`reference_cap/04-gstack-host-config-system-multi-host-portability.md`
- “模板 → 生成物 → 分发目录 → 调用约定”是可迁移形态的一部分：gstack 明确 `SKILL.md` 是 generated artifacts，并支持 `--host codex` 等 host-specific regeneration。`reference_cap/04-gstack-agents-md-workflow-and-host-portability.md`
- rules/context files 可承载 workflow-only guardrail 与工程/安全标准，但天然是宿主特定工件，需要映射与降级策略：`.clinerules` 示例。`reference_cap/04-gsd-clinerules-workflow-guardrails.md`
- 迁移映射的细节会产生真实故障点，因此需要回归测试固化 contract：Windsurf conversion regression tests 覆盖 frontmatter 标量、命令命名、路径与 tool 名称映射等。`reference_cap/04-gsd-windsurf-conversion-regression-tests.md`
- 社区迁移摩擦与信任顾虑：GSD port 到 GitHub Copilot 的讨论暴露目录差异、语言依赖、以及对外部 MCP 供应链的犹豫。`reference_cap/04-community-gsd-copilot-integration-reddit.md`
- Codex 对 `AGENTS.md` 的契约级定义：global/project scopes、`AGENTS.override.md`、fallback（CODEX/CONTEXT/README）与单文件 32KiB 限制，说明“仓库级指令工件”本身有明确 precedence 与容量边界。`reference_cap/04-openai-codex-agents-md-scopes-override-fallback.md`
- Codex agent loop 细节：指令聚合会沿 root→cwd 逐级拼接（含 `$CODEX_HOME` 与 repo root 的 override），并明确“shell tool sandboxed，但 MCP server tools 不自动 sandbox”，以及 MCP `list_changed` 通知会导致缓存 miss。`reference_cap/04-openai-unrolling-codex-agent-loop-instruction-aggregation-mcp-sandbox.md`
- Codex 的企业治理控制面：`requirements.toml`（管理员强制约束）与 `managed_config.toml`（启动默认值）可约束 approval policy/sandbox/web search，并可通过 MCP allowlist（按 name+identity 匹配）限制可启用 server；同时可强制 prefix_rules。`reference_cap/04-openai-codex-managed-configuration-requirements-toml.md`
- Codex rules 提供可回归的“命令级治理”：`prefix_rule`（allow/prompt/forbidden）+ tests，并通过 smart approvals 用 tree-sitter 解析 `bash -lc` 防命令走私，体现“门禁应可执行、可测试”。`reference_cap/04-openai-codex-rules-prefix-rule-smart-approvals.md`
- Codex 官方产品说明强调“可追溯证据链”：引用 terminal logs 与 test outputs；并明确可由 `AGENTS.md` 指导；同时披露 secure execution 边界（默认禁用互联网，且曾在 2025-06-03 更新允许联网）。`reference_cap/04-openai-introducing-codex-cloud-agent-secure-execution.md`
- Claude Code 的 memory/rules 支持 Imports（可导入 `AGENTS.md`），为跨工具互操作提供官方路径，说明“AGENTS.md 趋势”不仅存在于单一宿主。`reference_cap/04-anthropic-claude-code-memory-claude-md-imports.md`
- MCP 的供应链与语义攻击面（spec+研究）：confused deputy/SSRF/session hijacking/tool poisoning/shadowing/rug pulls，以及多恶意 server 放大效应，意味着 MCP 进入迁移评估必须纳入 scopes/allowlist/signing/vetting/guardrails。`reference_cap/03-mcp-security-best-practices.md` `reference_cap/03-arxiv-2512.06556-securing-mcp-tool-poisoning.md` `reference_cap/03-arxiv-2601.17549-breaking-the-protocol-mcp-security.md`

## 二轮新增机制理解

- “迁移价值”不是 popularity ranking，而是一个需要可复核证据支撑的工程判断。最小口径应至少能回答：
  - adoption complexity：引入前置条件与集成改造量
  - governance burden：权限/审查/审批链条与审计需求
  - maintenance overhead：兼容债、漂移、依赖更新与回归成本
  - portability cost：跨宿主目录/格式/tool 能力差异带来的映射层成本
  - team training cost：团队达到稳定有效使用的训练成本
- “可迁移”的核心不是命令名或话术，而是可执行 contract：
  - **仓库级配置工件**（AGENTS.md / rules / instructions）提供最小可运行语境，能影响效率与行为，但需要内容最小化与漂移治理。`reference_cap/04-arxiv-2601.20404-impact-of-agents-md-efficiency.md` `reference_cap/04-arxiv-2602.14690-configuring-agentic-ai-coding-tools.md`
  - **宿主适配层**（installer/rewrites/adapters/tests）把宿主差异前置为生成与回归问题，减少“看起来可用但实际失效”的隐性故障。`reference_cap/04-gsd-multi-runtime-installer-and-format-conversion.md` `reference_cap/04-gstack-host-config-system-multi-host-portability.md` `reference_cap/04-gsd-windsurf-conversion-regression-tests.md`
  - **确定性门禁与验证产物**（VALIDATION/UAT/REVIEW/health-history 等）把质量从“模型自觉”外移到可审计、可复用的工程资产（这部分证据主要来自 01-03 维度的二轮回填）。`reference_cap/03-gsd-verify-work-uat-and-verification-artifacts.md` `reference_cap/03-gstack-health-skill-quality-gates.md`
- “仓库级指令工件”的可迁移性并不等于“随便写一个文件”：Codex 已把 scope/override/fallback/32KiB 限制写成契约；Claude Code 则提供 imports 机制用于跨工具复用。更合理的做法是把 AGENTS.md 当作“入口索引/TOC + 最小 requirements”，把细节下沉到 docs/与可执行 gates。`reference_cap/04-openai-codex-agents-md-scopes-override-fallback.md` `reference_cap/04-anthropic-claude-code-memory-claude-md-imports.md`
- “安全与治理控制面”本身也是能力单元：prefix_rules（可测试）、approval policy/sandbox modes（可强制）、MCP allowlist/scopes（可配置）决定了外部工具链能否合规进入 SDLC pipeline。`reference_cap/04-openai-codex-rules-prefix-rule-smart-approvals.md` `reference_cap/04-openai-codex-managed-configuration-requirements-toml.md`
- “Context Files 占主导”并不等价于“它们足以承载全部治理”。更合理的分层是：
  - context files：最小 requirements 与操作入口
  - hooks/gates/verification：把安全/质量/回归变成确定性门禁与证据链（可治理、可审计）

## 二轮新增趋势与难点

- 趋势：仓库级配置机制仍以 Context Files 为主，但 `AGENTS.md` 这种工件正在走向跨工具互操作；与此同时，Skills/Subagents 等高级机制在 OSS 中整体采用仍浅。`reference_cap/04-arxiv-2602.14690-configuring-agentic-ai-coding-tools.md`
- 趋势：跨宿主迁移正在被工程化为“生成/转换/测试”链路（而不是 README 里的一句‘兼容’）。`reference_cap/04-gsd-multi-runtime-installer-and-format-conversion.md` `reference_cap/04-gsd-windsurf-conversion-regression-tests.md`
- 难点：多宿主可移植性依赖映射层（paths/frontmatter/tools/capabilities），它本身是长期维护负担，且很难一次性做对。`reference_cap/04-gstack-host-config-system-multi-host-portability.md`
- 难点：迁移会引入“信任与供应链治理”问题（外部 MCP/插件的安装顾虑、扫描误报、弃置仓库劫持等），需要把安全与 provenance 拉入迁移价值评估。`reference_cap/04-community-gsd-copilot-integration-reddit.md` `reference_cap/03-arxiv-2603.16572-repo-context-skill-security.md`
- 难点：宿主安全边界并不均匀，例如 Codex 明确“shell tool sandboxed，但 MCP server tools 不自动 sandbox”；这意味着把能力单元外移到 MCP 不能默认继承宿主沙箱，需要显式做 allowlist/scopes/approval/sandbox 组合治理。`reference_cap/04-openai-unrolling-codex-agent-loop-instruction-aggregation-mcp-sandbox.md` `reference_cap/04-openai-codex-managed-configuration-requirements-toml.md`
- 难点：目前公开材料更多能证明“机制存在/可迁移形态存在”，但关于“迁移后 correctness/质量的净收益”仍需要更多可复核案例与基准补齐（否则只能把评级降级为条件性结论）。

### 六个固定问题（本主题）

- **硬事实是什么：** 迁移与治理最常见的落盘形态仍是仓库级配置工件（Context Files），并出现 `AGENTS.md` 互操作趋势；且 `AGENTS.md` 在可控实验中与效率提升相关。`reference_cap/04-arxiv-2602.14690-configuring-agentic-ai-coding-tools.md` `reference_cap/04-arxiv-2601.20404-impact-of-agents-md-efficiency.md`
- **根本机制是什么：** 跨宿主迁移的核心是“显式映射层 + 回归测试 + 生成链路”，把宿主差异从隐性运行时故障变成可审计的转换 contract；同时把权限/外联/MCP 外部工具链纳入可执行 policy（approval/sandbox/allowlist/prefix_rules），否则迁移无法通过企业合规门禁。`reference_cap/04-gsd-windsurf-conversion-regression-tests.md` `reference_cap/04-gstack-host-config-system-multi-host-portability.md` `reference_cap/04-openai-codex-managed-configuration-requirements-toml.md`
- **生态最近在往哪里演化：** 配置机制仍以 Context Files 为主，但互操作标准与多宿主适配工程化正在增强；高级机制在 OSS 采用上仍偏浅。`reference_cap/04-arxiv-2602.14690-configuring-agentic-ai-coding-tools.md`
- **采用或落地的难点在哪里：** 映射层维护负担（compatibility debt）与安全/供应链信任（MCP/插件）是迁移的主要摩擦点；并且“shell sandboxed 但 MCP tools 不自动 sandbox”这类边界差异会强制要求更重的治理控制面。`reference_cap/04-gstack-host-config-system-multi-host-portability.md` `reference_cap/04-community-gsd-copilot-integration-reddit.md` `reference_cap/04-openai-unrolling-codex-agent-loop-instruction-aggregation-mcp-sandbox.md`
- **社区争议和失败模式在哪里：** 社区对外部组件信任不足、宿主差异导致“看起来能用但实际失效”；技能生态还存在扫描误报与弃置仓库劫持攻击面。`reference_cap/04-community-gsd-copilot-integration-reddit.md` `reference_cap/03-arxiv-2603.16572-repo-context-skill-security.md`
- **哪些对象最值得继续追踪：**
  - `AGENTS.md` / context files 的互操作标准化进展与内容最小化实践（并补 correctness/quality 的因果证据）。`reference_cap/04-arxiv-2601.20404-impact-of-agents-md-efficiency.md`
  - “宿主适配层”的可维护性工程：adapter 的语义差异覆盖范围、以及 conversion tests 的长期演化成本。`reference_cap/04-gsd-windsurf-conversion-regression-tests.md` `reference_cap/04-gstack-host-config-system-multi-host-portability.md`
  - 迁移价值的硬证据库：企业试点/公开基准/失败复盘，把“高/中/低”评级从合理推断升级为可复核结论（当前仍偏缺）。
