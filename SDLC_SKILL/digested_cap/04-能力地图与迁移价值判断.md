# 能力地图与迁移价值判断

## 这份片段在讲什么

这份上下文聚焦原始研究最后的抽象总结：

- 哪些能力单元最值得形成“能力地图”
- 哪些是高迁移价值抽象
- 哪些更偏包装层，不值得照抄

## 从原始研究提炼出的核心结论

### 1. 不要把“有名字的 skill”误认为“有价值的能力抽象”

原始研究最后给出一个非常关键的判断：

- 开源社区里很多所谓 agent skills，本质上只是 prompt 包装
- 它们对系统执行流程没有实质约束力
- 上下文一长、模型一漂移，就会失效

因此，真正值得迁移的不是名字和人设，  
而是底层工程抽象。

### 2. 高迁移价值的能力单元有明显共性

原始研究总结出的高价值对象包括：

- `Nyquist Validation Layer`
- `Orchestrator & Wave Parallelism`
- `RED-GREEN-REFACTOR Enforcement`
- `4-Phase Systematic Debugging`
- `Adversarial Critic / CONSENSAGENT`
- `File-Based Knowledge Persistence`
- `Playwright QA & Canary Watcher`

这些对象的共同点是：

- 直接干预控制流
- 限制 agent 的动作空间
- 引入确定性验证
- 打破单模型自我封闭
- 支持长周期和跨会话稳定运行

### 3. 原始研究最后沉淀出了四个最核心的工程抽象

这是整篇文档最值得继续研究的部分：

1. `控制流分离`
2. `验证闭环前置`
3. `确定性门禁 + 对抗性审查`
4. `状态持久化`

这四个抽象，比任何单个 skill 名字都重要。

因为它们回答的是：

- 系统怎么组织
- 质量怎么保证
- 漂移怎么约束
- 记忆怎么保存

### 4. 低迁移价值对象往往带有过强风格化包装

原始研究以 gstack 的 YC 风格需求澄清为例，做了一个很有用的区分：

- 某些对象在原框架里很有表现力
- 但迁移到企业环境时可能适配性差

这提醒我们：

- 不要把“好玩”误判成“适合复用”
- 不要把带强烈创始人风格的流程直接抄进严肃工程体系

### 5. 真正应该迁移的是“机制”，不是“语气”

整篇研究最后其实落到了一个很朴素但很重要的判断：

- 借鉴能力单元时，应优先迁移机制层
- 少迁移表演性包装、人设、口吻和特定文化风格

也就是：

- 借鉴“先验证再执行”
- 借鉴“用全新 critic 审查”
- 借鉴“用文件持久化状态”

而不是简单复制某个框架里的花哨命令名。

- 高迁移价值能力单元列表
- 四大工程抽象
- 包装层 vs 机制层的区分

## 适合继续 Deep Research 的问题

- 如何系统评估一个能力单元的迁移价值
- 哪些能力单元最值得优先落地到企业软件开发流程
- 宿主生态迁移时，哪些东西应该保留，哪些应该丢掉
- 怎么区分“值得借鉴的工程抽象”和“只是在原项目里看起来很酷的包装”
