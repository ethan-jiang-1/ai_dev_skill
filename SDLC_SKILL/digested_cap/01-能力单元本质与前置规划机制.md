# 能力单元本质与前置规划机制

## 第一轮摘要（保留，不修改）

## 这份片段在讲什么

这份上下文聚焦 AI Agent 在 SDLC 前半段最关键的两个问题：

- 能力单元到底是什么，为什么它不是普通 prompt 包装
- 在真正写代码之前，系统如何通过需求澄清、架构约束、计划拆解和验证前置来防止失控

## 从原始研究提炼出的核心结论

### 1. 能力单元的本质是“改变执行机制”，不是“换个说法下指令”

原始研究给出的核心判断非常明确：

- 真正有价值的能力单元，不只是提示词
- 它必须有输入、输出、执行顺序和约束条件
- 它通过机制层改变 agent 的工作方式

文中总结了四种基础架构模式：

- `Skill-like Units`
- `Subagents & Personas`
- `Workflows & Orchestrators`
- `Hooks & Quality Gates`

这四类对象分别对应：

- 专业知识的按需注入
- 纯净上下文中的任务隔离
- 宏观调度和依赖分发
- 用确定性检查阻断错误流转

### 2. Define / Design 阶段的关键不是生成代码，而是阻止模型过早写代码

原始研究特别强调：

- 需求定义阶段最大的失控问题是“任务理解跑偏”
- 架构设计阶段最大的失控问题是“盲目执行”

代表性机制有两类：

- `gstack` 的 `/office-hours` 和 `/plan-ceo-review`
- `GABBE / Loki Mode` 的 Gate 1 / Gate 2 认知工程单元

这类能力单元的共同特点是：

- 先追问需求和范围
- 强制产出结构化设计文档
- 用打分矩阵、模板或脚本去限制偷懒式设计
- 没过关就不允许进入后续实现阶段

### 3. Plan 阶段最核心的抽象是“细粒度任务分解 + 验证前置”

原始研究把计划阶段的两个关键对象讲得很清楚：

- GSD 的 XML 结构化计划
- Nyquist Validation Layer

这里最值得继续研究的，不是 XML 形式本身，  
而是背后的两个工程抽象：

- 任务要细到足够原子，避免模型写出占位符代码
- 每个任务在开始前就必须绑定一个可执行的验证命令

也就是说，计划不能只是“待办清单”，  
而必须是“可验证的执行树”。

### 4. Nyquist Layer 是最值得迁移的前置验证机制

原始研究认为，Nyquist Layer 的价值极高，因为它把验证从“写完代码再补”提前到了计划阶段：

- 先找出现有测试基础设施
- 把需求映射成具体 verify 命令
- 没有 verify 的任务不允许进入执行
- 必要时先自动补齐 Wave 0 脚手架

这本质上是在压缩 agent 幻觉空间。  
因为一旦交付标准被锁成终端命令，模型就更难靠“看起来像完成了”蒙混过关。

## 二轮新增证据

- gstack `/office-hours` 明确设定 **HARD GATE**：禁止写代码/脚手架/实现动作，唯一产出是 design document。`reference_cap/01-gstack-office-hours-hard-gate-design-doc.md`
- gstack `/plan-ceo-review` 将“计划审查”建模为 4 种 scope mode（EXPANSION/SELECTIVE/HOLD/REDUCTION），并规定每个 scope 变化都必须 AskUserQuestion 显式 opt-in，禁止 silent add/remove scope。`reference_cap/01-gstack-plan-ceo-review-scope-modes-and-directives.md`
- gstack `/plan-eng-review` 把计划审查前置为 Step 0（scope challenge、complexity smell、search-before-building、distribution check、completeness check），并把测试审查写成 coverage diagram + 回归测试铁律。`reference_cap/01-gstack-plan-eng-review-completeness-and-coverage.md`
- GSD 用 `.planning/config.json` 作为治理控制面：workflow toggles、gates、safety、parallelization、hooks；并声明 prompt injection guard hook 永远开启且不可关闭。`reference_cap/01-gsd-configuration-schema-and-gates.md`
- 实证研究显示 agent context files（AGENTS.md/CLAUDE.md）在 OSS 中像“活的配置工件”演化，内容偏功能性指令，非功能约束（安全/性能）显著稀缺。`reference_cap/01-arxiv-2511.12884-agent-readmes-context-files.md`
- 反例证据：在多 agents/LLMs 上，context files 可能降低任务成功率并提高推理成本（20%+）；并指出“不必要要求会让任务更难”，建议 human-written context files 只写最小 requirements。`reference_cap/01-arxiv-2602.11988-evaluating-agents-md-helpfulness.md`
- “计划=可验证执行树/验证前置”的一手规范化表达：Nyquist Validation（requirements→test commands 映射，产出 `{phase}-VALIDATION.md`）与 Plan Checker 8 维校验（含 Nyquist compliance）。`reference_cap/01-gsd-features-nyquist-plan-checker-wave-exec.md`

## 二轮新增机制理解

- “禁止过早写代码”的价值不在于“更谨慎”，而在于**阻断模型把不确定需求直接投射成代码结构**的路径；用 HARD GATE 把输出从“代码”换成“可审计的设计工件”，相当于把不确定性集中到更便宜的阶段处理。`reference_cap/01-gstack-office-hours-hard-gate-design-doc.md`
- “scope 控制”如果靠模型自觉，很容易发生 silent drift（暗自扩张或缩减）。把 scope 变化做成 AskUserQuestion 的显式 opt-in，本质上是把“范围变化”升级为一个需要人类签字的 gate。`reference_cap/01-gstack-plan-ceo-review-scope-modes-and-directives.md`
- “completeness is cheap/boil the lake”本质是 **AI 将边际实现成本压缩后，最优策略从‘先跑通再补齐’转向‘一次做全避免返工’**；但这会与企业的风险分级、合规审计、发布窗口产生张力，需要治理层（gates/config）介入裁剪。`reference_cap/01-gstack-plan-ceo-review-scope-modes-and-directives.md` `reference_cap/01-gsd-configuration-schema-and-gates.md`
- planning 阶段能否变成“可执行工程协议”，关键在于把“验证/覆盖/错误路径/可观测性/分发”这些传统后置问题前置，并把它们写成计划审查的强制章节与产物（coverage diagram、distribution check）。`reference_cap/01-gstack-plan-eng-review-completeness-and-coverage.md`
- context files 的作用更像“最低限度让 agent 可运行的 repo 配置”，而不是质量保证系统。实证与描述性研究共同提示：把过多要求塞进 context files 可能降低成功率并增加成本，因此更合理的分层是：
  - context files：最小可运行信息（命令、结构、关键约束）
  - hooks/gates/verification：把安全/质量/合规转成可执行、可审计的确定性门禁。`reference_cap/01-arxiv-2511.12884-agent-readmes-context-files.md` `reference_cap/01-arxiv-2602.11988-evaluating-agents-md-helpfulness.md` `reference_cap/01-gsd-configuration-schema-and-gates.md`

## 二轮新增趋势与难点

- 趋势：AGENTS.md/CLAUDE.md 这类“agent context files”正在成为跨工具常见工件，但其内容结构更偏功能性“让它能跑起来”，而非非功能护栏。`reference_cap/01-arxiv-2511.12884-agent-readmes-context-files.md`
- 趋势：对于“验证前置”的需求在增强，Nyquist-like 机制开始被写成明确产物与约束（VALIDATION.md、plan-check 维度）。`reference_cap/01-gsd-features-nyquist-plan-checker-wave-exec.md`
- 难点：context files 的“越写越长”可能带来负效应（成功率下降、成本上升），且不必要要求会让任务更难；需要定义“最小 requirements”与外移策略。`reference_cap/01-arxiv-2602.11988-evaluating-agents-md-helpfulness.md`
- 难点：gstack 这类 heavily-interactive 的 AskUserQuestion gate 在企业批处理/自动化环境中难以直接复用，需要支持 text mode、默认策略、以及“谁有权批准”的组织设计（对应 GSD 的 gates/safety 配置面）。`reference_cap/01-gsd-configuration-schema-and-gates.md`
- 争议：completeness 偏好与交付节奏/风险控制的冲突。企业并非所有变更都值得 boil the lake；必须引入变更类型与风险分级来决定 coverage/observability/diagram 的强度。`reference_cap/01-gstack-plan-eng-review-completeness-and-coverage.md`

## 当前判断（二轮综合后）

- **硬事实是什么：** planning 能力单元在多个框架中都被实现为“硬门禁 + 结构化工件 + 显式决策点”，而不仅是 prompt 文本。`reference_cap/01-gstack-office-hours-hard-gate-design-doc.md` `reference_cap/01-gsd-features-nyquist-plan-checker-wave-exec.md` `reference_cap/01-gabbe-loki-mode-sdlc-phases-rarv.md`
- **根本机制是什么：** 用 gate 把“不可逆/高成本决策”锁在更便宜阶段，用确定性产物（VALIDATION.md、config.json、design docs）把隐式假设外显化并可审计。`reference_cap/01-gsd-configuration-schema-and-gates.md` `reference_cap/01-gstack-plan-eng-review-completeness-and-coverage.md`
- **生态往哪里演化：** context files 普及，但“最小化 + 外移到 hooks/gates”会成为更合理的演化方向（因为存在负效应证据）。`reference_cap/01-arxiv-2602.11988-evaluating-agents-md-helpfulness.md`
- **落地难点：** 交互式 gate 的审批链、计划工件粒度、以及 completeness 偏好与企业风险分级的对齐，是迁移的主要阻力。`reference_cap/01-gsd-configuration-schema-and-gates.md`
- **争议与失败模式：** 冗余/不必要的 context file 要求可能降低成功率并提高成本；“越写越全”不一定更好。`reference_cap/01-arxiv-2602.11988-evaluating-agents-md-helpfulness.md`
- **最值得继续追踪的对象：**
  - context files 的“最小 requirements”写法与自动 lint/缩减工具（围绕 AGENT BENCH 等基准）。`reference_cap/01-arxiv-2602.11988-evaluating-agents-md-helpfulness.md`
  - Nyquist-like 验证前置与 plan-check 维度如何在不同宿主落地（产物与命令对齐）。`reference_cap/01-gsd-features-nyquist-plan-checker-wave-exec.md`
  - gstack 的 plan review/coverage diagram 机制与 GSD 的 gates/config 机制如何组合成“可裁剪、可治理”的企业版 planning pipeline。`reference_cap/01-gstack-plan-eng-review-completeness-and-coverage.md` `reference_cap/01-gsd-configuration-schema-and-gates.md`
