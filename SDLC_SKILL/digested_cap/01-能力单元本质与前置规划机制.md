# 能力单元本质与前置规划机制

## 这份片段在讲什么

这份上下文聚焦 AI Agent 在 SDLC 前半段最关键的两个问题：

- 能力单元到底是什么，为什么它不是普通 prompt 包装
- 在真正写代码之前，系统如何通过需求澄清、架构约束、计划拆解和验证前置来防止失控

## 从原始研究提炼出的核心结论

### 1. 能力单元的本质是“改变执行机制”，不是“换个说法下指令”

原始研究给出的核心判断非常明确：

- 真正有价值的能力单元，不只是提示词
- 它必须有输入、输出、执行顺序和约束条件
- 它通过机制层改变 agent 的工作方式

文中总结了四种基础架构模式：

- `Skill-like Units`
- `Subagents & Personas`
- `Workflows & Orchestrators`
- `Hooks & Quality Gates`

这四类对象分别对应：

- 专业知识的按需注入
- 纯净上下文中的任务隔离
- 宏观调度和依赖分发
- 用确定性检查阻断错误流转

### 2. Define / Design 阶段的关键不是生成代码，而是阻止模型过早写代码

原始研究特别强调：

- 需求定义阶段最大的失控问题是“任务理解跑偏”
- 架构设计阶段最大的失控问题是“盲目执行”

代表性机制有两类：

- `gstack` 的 `/office-hours` 和 `/plan-ceo-review`
- `GABBE / Loki Mode` 的 Gate 1 / Gate 2 认知工程单元

这类能力单元的共同特点是：

- 先追问需求和范围
- 强制产出结构化设计文档
- 用打分矩阵、模板或脚本去限制偷懒式设计
- 没过关就不允许进入后续实现阶段

### 3. Plan 阶段最核心的抽象是“细粒度任务分解 + 验证前置”

原始研究把计划阶段的两个关键对象讲得很清楚：

- GSD 的 XML 结构化计划
- Nyquist Validation Layer

这里最值得继续研究的，不是 XML 形式本身，  
而是背后的两个工程抽象：

- 任务要细到足够原子，避免模型写出占位符代码
- 每个任务在开始前就必须绑定一个可执行的验证命令

也就是说，计划不能只是“待办清单”，  
而必须是“可验证的执行树”。

### 4. Nyquist Layer 是最值得迁移的前置验证机制

原始研究认为，Nyquist Layer 的价值极高，因为它把验证从“写完代码再补”提前到了计划阶段：

- 先找出现有测试基础设施
- 把需求映射成具体 verify 命令
- 没有 verify 的任务不允许进入执行
- 必要时先自动补齐 Wave 0 脚手架

这本质上是在压缩 agent 幻觉空间。  
因为一旦交付标准被锁成终端命令，模型就更难靠“看起来像完成了”蒙混过关。

- 什么样的能力单元才算真正改变了 agent 的执行方式

- 计划阶段的“验证前置”到底该怎么实现
- Nyquist Layer 这种机制能否迁移到别的宿主或团队工作流里
