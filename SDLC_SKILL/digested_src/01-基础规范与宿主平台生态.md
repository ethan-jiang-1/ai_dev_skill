### 这份片段在讲什么

这份上下文聚焦在软件开发智能体能力包的“底层协议层”和“宿主平台层”。

研究对象不是普通 prompt 仓库，  
而是会改变 coding agent 规划、执行、验证方式的结构化能力包，常见形态包括：

- skill
- workflow
- plugin
- command
- hook
- subagent

这一层最重要的判断是：

- AgentSkills / `SKILL.md` 正在变成跨平台流通的事实标准
- 真正的差异不只在 skill 内容本身，而在宿主平台怎么发现、挂载、调用、授权和组合这些能力

### 从原始研究提炼出的核心结论

#### 1. `SKILL.md` 是整个生态的基础协议

原始研究认为，AgentSkills 规范已经成为智能体能力包的基础设施骨架。

它的关键不是“把提示词放进一个文件”，  
而是把能力包做成一个可版本控制的目录：

- `SKILL.md`
- `scripts/`
- `references/`
- `assets/`

更关键的是“渐进式披露”机制：

- 会话启动时只加载轻量目录清单
- 只有命中某个 skill 描述时才动态读取完整上下文

这意味着 skill 不是一次性塞进 system prompt 的长提示词，  
而是一种按需挂载的上下文资产。

#### 2. 官方宿主平台决定了能力包的真实运行方式

原始研究里最值得单独深挖的一条线，是不同宿主对 skill 的支持模型并不相同。

代表性的几类：

- Anthropic / Claude Code：官方 `anthropics/skills` 是最权威参考实现，另有 `skill-creator` 这种“帮助开发 skill 的 skill”
- Cursor：更偏插件市场模型，把 Rules、Skills、Commands、Hooks、MCP 打包进统一插件体系
- Windsurf：同时支持 skills 和手动触发的 workflows，更强调自动上下文检索
- OpenCode：采用开放规范，但在权限、目录边界和工具调用拦截上做得极细
- Google Antigravity：把多子智能体并行管理和工程制品生成放到系统层
- OpenAI Codex：用分层注册表管理能力，强调 curated / experimental 的目录治理
- GitHub Copilot：skill 能力和 VS Code 扩展体系深度绑定

#### 3. 真正值得比较的是“宿主抽象”，不是只看谁支持 skill

原始研究里已经隐含了一个更强的问题：

- 谁在用统一标准
- 谁在做专有包装
- 谁支持按需挂载
- 谁支持更细粒度的权限控制
- 谁更适合复杂 SDLC 工作流
- 谁更适合团队级共享和治理

也就是说，后续 Deep Research 不应该停留在“有没有 marketplace”，  
而要继续比较不同宿主的能力抽象模型。

### 这一片段里最值得继续研究的对象

- AgentSkills.io / `SKILL.md`
- `anthropics/skills`
- Cursor marketplace / plugin model
- Windsurf skills + workflows
- OpenCode skills + permission model
- Google Antigravity
- OpenAI Codex skills catalog
- GitHub Copilot agent skills

### 适合继续 Deep Research 的问题

- `SKILL.md` 作为标准，在哪些平台是真兼容，在哪些平台只是名义支持
- 不同宿主在 skill 的发现、加载、触发、权限、组合方面差异多大
- 哪个宿主最适合软件开发类 skill，而不是泛助手类能力
- 哪个平台更适合团队沉淀内部 skill 库
- 哪个平台对开发者自定义 skill 最友好

## 二轮新增证据

<!-- 每条新增事实都带 ../reference_src/*.md 回指 -->

- Agent Skills 规范明确：skill 最小是一个目录，至少包含 `SKILL.md`，可选包含 `scripts/`、`references/`、`assets/` 等子目录。（Ref: ../reference_src/00-shared-agentskills-specification.md）
- Agent Skills 规范明确：`SKILL.md` 必须由 YAML frontmatter + Markdown body 组成，frontmatter 至少包含 `name` 与 `description`，并对 `name` 的字符集/长度/父目录一致性等做了硬约束。（Ref: ../reference_src/00-shared-agentskills-specification.md）
- `anthropics/skills` 仓库自述为 Anthropic 为 Claude 实现的 skills；每个 skill 在独立文件夹内用 `SKILL.md` 提供指令与元数据；仓库同时包含开源（Apache-2.0）与 source-available 的文档处理类生产技能示例。（Ref: ../reference_src/00-shared-anthropic-skills-readme.md）
- Windsurf 的 Workflows 是 Markdown 文件，通过 `/<workflow-name>` 手动触发（manual-only，Cascade 不会自动调用）；并给出 `.windsurf/workflows/*.md`（repo 可提交）与 `~/.codeium/windsurf/global_workflows/*.md`（全局）等存储与发现规则。（Ref: ../reference_src/00-shared-windsurf-workflows-docs.md）
- Claude 官方说明把 skills 的工作方式描述为 progressive disclosure：Claude 判断相关 skills 并按需加载，以避免 context window overload。（Ref: ../reference_src/01-host-claude-what-are-skills.md）
- Claude 官方说明区分了 Anthropic Skills（官方维护、相关时自动调用）、Custom Skills（用户/组织自建）、Organization provisioned skills（Team/Enterprise 组织预置）与 Partner skills（与 MCP connectors 集成的伙伴技能）。（Ref: ../reference_src/01-host-claude-what-are-skills.md）
- Claude 的自定义技能文档对 Skill.md 元数据给出约束与字段示例（如 `name` 为 human-friendly name；`description` 200 chars maximum；可选 `dependencies`），并把 metadata 描述为 progressive disclosure 的第一层。（Ref: ../reference_src/01-host-claude-creating-custom-skills.md）
- Claude 的“使用 skills”文档给出启用与治理路径：需要启用 Code execution and file creation；通过 Customize > Skills 管理、上传 ZIP；Team/Enterprise 支持组织预置与分享（shared skills 为 view-only）。（Ref: ../reference_src/01-host-claude-using-skills-in-claude.md）
- Claude 自定义技能文档对 `name/description` 的约束与 agentskills.io 规范的 `name` 字符集/父目录一致性、`description` 上限等存在差异，意味着“标准兼容”在落地时需要做字段/约束映射与互操作核验。（Ref: ../reference_src/01-host-claude-creating-custom-skills.md；Ref: ../reference_src/00-shared-agentskills-specification.md）
- Anthropic 的工程博客进一步把 progressive disclosure 拆成明确分层：startup 预加载所有 skills 的 `name/description` 到 system prompt（第一层），仅在相关时加载完整 `SKILL.md`（第二层），并可按需加载 skill 目录内被引用的额外文件（第三层及更深）。（Ref: ../reference_src/00-shared-anthropic-engineering-agent-skills-2025.md）
