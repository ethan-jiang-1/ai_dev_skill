### 这份片段在讲什么

这份上下文聚焦在软件开发智能体能力包的“底层协议层”和“宿主平台层”。

研究对象不是普通 prompt 仓库，  
而是会改变 coding agent 规划、执行、验证方式的结构化能力包，常见形态包括：

- skill
- workflow
- plugin
- command
- hook
- subagent

这一层最重要的判断是：

- AgentSkills / `SKILL.md` 正在变成跨平台流通的事实标准
- 真正的差异不只在 skill 内容本身，而在宿主平台怎么发现、挂载、调用、授权和组合这些能力

### 从原始研究提炼出的核心结论

#### 1. `SKILL.md` 是整个生态的基础协议

原始研究认为，AgentSkills 规范已经成为智能体能力包的基础设施骨架。

它的关键不是“把提示词放进一个文件”，  
而是把能力包做成一个可版本控制的目录：

- `SKILL.md`
- `scripts/`
- `references/`
- `assets/`

更关键的是“渐进式披露”机制：

- 会话启动时只加载轻量目录清单
- 只有命中某个 skill 描述时才动态读取完整上下文

这意味着 skill 不是一次性塞进 system prompt 的长提示词，  
而是一种按需挂载的上下文资产。

#### 2. 官方宿主平台决定了能力包的真实运行方式

原始研究里最值得单独深挖的一条线，是不同宿主对 skill 的支持模型并不相同。

代表性的几类：

- Anthropic / Claude Code：官方 `anthropics/skills` 是最权威参考实现，另有 `skill-creator` 这种“帮助开发 skill 的 skill”
- Cursor：更偏插件市场模型，把 Rules、Skills、Commands、Hooks、MCP 打包进统一插件体系
- Windsurf：同时支持 skills 和手动触发的 workflows，更强调自动上下文检索
- OpenCode：采用开放规范，但在权限、目录边界和工具调用拦截上做得极细
- Google Antigravity：把多子智能体并行管理和工程制品生成放到系统层
- OpenAI Codex：用分层注册表管理能力，强调 curated / experimental 的目录治理
- GitHub Copilot：skill 能力和 VS Code 扩展体系深度绑定

#### 3. 真正值得比较的是“宿主抽象”，不是只看谁支持 skill

原始研究里已经隐含了一个更强的问题：

- 谁在用统一标准
- 谁在做专有包装
- 谁支持按需挂载
- 谁支持更细粒度的权限控制
- 谁更适合复杂 SDLC 工作流
- 谁更适合团队级共享和治理

也就是说，后续 Deep Research 不应该停留在“有没有 marketplace”，  
而要继续比较不同宿主的能力抽象模型。

### 这一片段里最值得继续研究的对象

- AgentSkills.io / `SKILL.md`
- `anthropics/skills`
- Cursor marketplace / plugin model
- Windsurf skills + workflows
- OpenCode skills + permission model
- Google Antigravity
- OpenAI Codex skills catalog
- GitHub Copilot agent skills

### 适合继续 Deep Research 的问题

- `SKILL.md` 作为标准，在哪些平台是真兼容，在哪些平台只是名义支持
- 不同宿主在 skill 的发现、加载、触发、权限、组合方面差异多大
- 哪个宿主最适合软件开发类 skill，而不是泛助手类能力
- 哪个平台更适合团队沉淀内部 skill 库
- 哪个平台对开发者自定义 skill 最友好

## 二轮新增证据

<!-- 每条新增事实都带 ../reference_src/*.md 回指 -->

- Agent Skills 规范明确：skill 最小是一个目录，至少包含 `SKILL.md`，可选包含 `scripts/`、`references/`、`assets/` 等子目录。（Ref: ../reference_src/00-shared-agentskills-specification.md）
- Agent Skills 规范明确：`SKILL.md` 必须由 YAML frontmatter + Markdown body 组成，frontmatter 至少包含 `name` 与 `description`，并对 `name` 的字符集/长度/父目录一致性等做了硬约束。（Ref: ../reference_src/00-shared-agentskills-specification.md）
- `anthropics/skills` 仓库自述为 Anthropic 为 Claude 实现的 skills；每个 skill 在独立文件夹内用 `SKILL.md` 提供指令与元数据；仓库同时包含开源（Apache-2.0）与 source-available 的文档处理类生产技能示例。（Ref: ../reference_src/00-shared-anthropic-skills-readme.md）
- Windsurf 的 Workflows 是 Markdown 文件，通过 `/<workflow-name>` 手动触发（manual-only，Cascade 不会自动调用）；并给出 `.windsurf/workflows/*.md`（repo 可提交）与 `~/.codeium/windsurf/global_workflows/*.md`（全局）等存储与发现规则。（Ref: ../reference_src/00-shared-windsurf-workflows-docs.md）
- Claude 官方说明把 skills 的工作方式描述为 progressive disclosure：Claude 判断相关 skills 并按需加载，以避免 context window overload。（Ref: ../reference_src/01-host-claude-what-are-skills.md）
- Claude 官方说明区分了 Anthropic Skills（官方维护、相关时自动调用）、Custom Skills（用户/组织自建）、Organization provisioned skills（Team/Enterprise 组织预置）与 Partner skills（与 MCP connectors 集成的伙伴技能）。（Ref: ../reference_src/01-host-claude-what-are-skills.md）
- Claude 的自定义技能文档对 Skill.md 元数据给出约束与字段示例（如 `name` 为 human-friendly name；`description` 200 chars maximum；可选 `dependencies`），并把 metadata 描述为 progressive disclosure 的第一层。（Ref: ../reference_src/01-host-claude-creating-custom-skills.md）
- Claude 的“使用 skills”文档给出启用与治理路径：需要启用 Code execution and file creation；通过 Customize > Skills 管理、上传 ZIP；Team/Enterprise 支持组织预置与分享（shared skills 为 view-only）。（Ref: ../reference_src/01-host-claude-using-skills-in-claude.md）
- Claude 自定义技能文档对 `name/description` 的约束与 agentskills.io 规范的 `name` 字符集/父目录一致性、`description` 上限等存在差异，意味着“标准兼容”在落地时需要做字段/约束映射与互操作核验。（Ref: ../reference_src/01-host-claude-creating-custom-skills.md；Ref: ../reference_src/00-shared-agentskills-specification.md）
- Anthropic 的工程博客进一步把 progressive disclosure 拆成明确分层：startup 预加载所有 skills 的 `name/description` 到 system prompt（第一层），仅在相关时加载完整 `SKILL.md`（第二层），并可按需加载 skill 目录内被引用的额外文件（第三层及更深）。（Ref: ../reference_src/00-shared-anthropic-engineering-agent-skills-2025.md）
- OpenCode 的官方文档明确其会从多种目录发现 skills（含 `.opencode/skills`、`.claude/skills`、`.agents/skills` 与对应的全局路径），并通过原生 `skill` tool 按需加载；同时给出 `opencode.json` 中基于模式的 skill 权限控制（allow/ask/deny）与 tool 禁用方式。（Ref: ../reference_src/01-host-opencode-agent-skills-docs.md）
- OpenAI Codex 官方文档明确：skills 使用 progressive disclosure（先读 metadata，按需加载 `SKILL.md`）；并给出 `.agents/skills` 的发现与作用域规则（从 CWD 向上扫描到 repo root 的各级 `.agents/skills`，并支持 `$HOME/.agents/skills` 等）；同时区分 “skills（authoring format）vs plugins（installable distribution unit）”，并给出 `~/.codex/config.toml` 禁用 skill 与 `agents/openai.yaml` 可选元数据的机制。（Ref: ../reference_src/01-host-openai-codex-agent-skills-docs.md）
- Gemini CLI 官方文档把“extensions”作为可安装的能力打包单位：扩展默认安装为拷贝目录，加载自 `~/.gemini/extensions`；manifest `gemini-extension.json` 支持捆绑 MCP servers 配置（`mcpServers`）、排除工具（`excludeTools`）、custom commands/hooks，以及在 `skills/<name>/SKILL.md` 目录内捆绑 agent skills；并支持按 user/workspace scope 启停扩展。（Ref: ../reference_src/01-host-google-gemini-cli-extensions-reference.md）
- Windsurf 的官方文档明确：Cascade 的 skills 使用 progressive disclosure（默认仅给模型 `name/description`，invoke 或 `@mention` 时才加载完整 `SKILL.md` 与支持资源）；skills 目录与作用域分为 `.windsurf/skills/`（workspace）、`~/.codeium/windsurf/skills/`（global）与企业级 system 目录；并支持跨 agent 扫描 `.agents/skills`，以及在开启特定设置时扫描 `.claude/skills`。（Ref: ../reference_src/01-host-windsurf-skills-docs.md）
- Cursor 的官方博客把 plugins 定义为能力打包单位，并明确其可包含 MCP servers、skills、subagents、rules、hooks；且声明正在推进 private team marketplaces 以支持组织内分发并提供 central governance 与 security controls。（Ref: ../reference_src/01-host-cursor-plugins-blog-2026-02-17.md）
- Cursor 官方论坛发布说明补充了 plugin 的产品内安装入口：可从 Cursor Marketplace 浏览，或在编辑器内使用 `/add-plugin` 直接安装；并再次确认 plugin 打包 skills/subagents/MCP/hooks/rules。（Ref: ../reference_src/01-host-cursor-forum-plugins-2-5.md）
- Cursor 社区 bug report 给出一个重要失败模式：同一 plugin 在 Cursor IDE 可用，但在 Cursor CLI mode 不会加载（报告者期望缓存路径 `~/.cursor/plugins/cache/` 的插件在 CLI/IDE 均可用）。这类差异会直接影响 headless/终端/自动化场景的可用性与团队治理一致性。（Ref: ../reference_src/01-host-cursor-forum-cli-mode-plugins-bug.md）
- Cursor 官方仓库 `cursor/plugins`（multi-plugin marketplace repo）在 README 明确：每个 plugin 是仓库根目录下的独立目录，拥有自己的 `.cursor-plugin/plugin.json` manifest；仓库根目录 `.cursor-plugin/marketplace.json` 作为 marketplace manifest 列出所有 plugins。（Ref: ../reference_src/01-host-cursor-plugins-github-readme.md）
- Cursor 官方仓库同时提供 plugin/marketplace 的 JSON Schema（draft-07）：`plugin.json` schema 明确 plugins 可声明 `skills/rules/hooks/agents/commands/mcpServers` 等组件字段，并对 `name` 模式、`additionalProperties=false` 等做硬约束；`marketplace.json` schema 的 `plugins[].source` 允许“相对路径或 remote URL”。（Ref: ../reference_src/01-host-cursor-plugins-json-schemas.md）
- Cursor 官方仓库的 create-plugin scaffold skill 明确给出本地插件默认落盘路径：`~/.cursor/plugins/local/<plugin-name>/`，并声明该路径可使 plugin “无需安装步骤即可立即被 Cursor 使用”。（Ref: ../reference_src/01-host-cursor-plugins-create-plugin-scaffold-skill.md）
- Cursor 官方仓库的示例 plugins 给出 hooks 的运行时契约实证：hooks 通过 `hooks.json` 声明 `stop/afterAgentResponse` 等事件并执行命令；`stop` hook 可通过输出 `followup_message` 驱动 agent 继续下一轮；示例中也出现 `CURSOR_PROJECT_DIR/CURSOR_PLUGIN_ROOT` 等环境变量约定，用于定位项目目录与插件根目录。（Ref: ../reference_src/01-host-cursor-plugins-hooks-runtime-contract.md）
- GitHub Copilot 的 repository custom instructions 官方文档明确三类可版本控制指令资产：repo-wide `.github/copilot-instructions.md`、path-specific `.github/instructions/*.instructions.md`、以及 agent instructions `AGENTS.md`（目录树就近优先）；并给出 `CLAUDE.md`/`GEMINI.md` 作为兼容替代。（Ref: ../reference_src/01-host-github-copilot-repo-custom-instructions.md）
- Copilot CLI 的命令参考页明确其支持 skills（`SKILL.md`）并给出 discovery locations/优先级：包括 `.github/skills/`、`.agents/skills/`、`.claude/skills/`、`~/.agents/skills/`、plugin directories、`COPILOT_SKILLS_DIRS` 等；并定义 `allowed-tools`、`user-invocable`、`disable-model-invocation` 等 frontmatter 字段。（Ref: ../reference_src/01-host-github-copilot-cli-command-reference.md）
- Copilot CLI 的治理与权限模型被显式产品化：trusted folders（`~/.copilot/config.json` 的 `trusted_folders`）、工具可见性控制（`--available-tools/--excluded-tools`）与工具执行权限控制（`--allow-tool/--deny-tool`），以及 `--yolo`/`/reset-allowed-tools` 等交互治理入口。（Ref: ../reference_src/01-host-github-copilot-cli-configure.md；../reference_src/01-host-github-copilot-cli-allowing-tools.md；../reference_src/01-host-github-copilot-cli-command-reference.md）

## 二轮新增机制理解

<!-- 从“描述”上升到“为什么这样设计”的解释；每条带 ../reference_src/*.md 回指 -->

- “progressive disclosure（渐进式披露）”之所以成为多宿主共同选择，本质是为了解决 context window 资源约束：启动时只加载 `name/description` 等轻量元数据，在模型判断相关时才按需加载完整 `SKILL.md`，必要时再加载目录内的引用文件。（Ref: ../reference_src/01-host-claude-what-are-skills.md；../reference_src/00-shared-anthropic-engineering-agent-skills-2025.md；../reference_src/01-host-windsurf-skills-docs.md；../reference_src/01-host-openai-codex-agent-skills-docs.md）
- “开放标准”主要解决的是 skill 的最小可交换单位（目录 + `SKILL.md`）与基础 frontmatter 约束，但真实体验差异来自宿主的发现范围、优先级与冲突处理（例如 Codex/OpenCode 都会在 repo 中向上扫描 `.agents/skills`；Codex 还明确 user/admin/system 层与同名不合并策略）。（Ref: ../reference_src/00-shared-agentskills-specification.md；../reference_src/01-host-openai-codex-agent-skills-docs.md；../reference_src/01-host-opencode-agent-skills-docs.md）
- 宿主的“能力打包单位”正在分化：Codex 明确区分 skills（authoring format）与 plugins（installable distribution unit）；Cursor 把 skills/subagents/MCP/rules/hooks 等都封装进 plugins，并走 marketplace 安装与治理；这决定了生态里“分发/升级/审计”的落点。（Ref: ../reference_src/01-host-openai-codex-agent-skills-docs.md；../reference_src/01-host-cursor-plugins-blog-2026-02-17.md；../reference_src/01-host-cursor-forum-plugins-2-5.md）
- Cursor plugins 把“可安装单元”落在 manifest（`marketplace.json` + `plugin.json`）并提供 JSON Schema 约束，使 plugin 的可索引性、可校验性与可治理性变成文件级契约；这类 schema-first 设计是 curated marketplace 与后续私有 team marketplace 进行自动化验证/审计/索引的基础。（Ref: ../reference_src/01-host-cursor-plugins-github-readme.md；../reference_src/01-host-cursor-plugins-json-schemas.md；../reference_src/01-host-cursor-plugins-blog-2026-02-17.md）
- Gemini CLI 的 “extensions” 机制展示了另一种宿主抽象：扩展作为安装单位可以同时承载 MCP servers 配置、agent skills、custom commands 与 hooks，并提供 `excludeTools` 这类治理入口，因此在语义上更接近“plugin”而不是单纯的 skills 文件夹。（Ref: ../reference_src/01-host-google-gemini-cli-extensions-reference.md）
- 权限与治理机制是宿主差异的另一主轴：OpenCode 提供基于模式的 allow/ask/deny 与 tool 禁用；Codex 提供 `~/.codex/config.toml` 的按路径禁用；Claude 则通过产品设置与组织管理入口（Team/Enterprise provisioned & shared skills）把治理放在 UI/组织层。（Ref: ../reference_src/01-host-opencode-agent-skills-docs.md；../reference_src/01-host-openai-codex-agent-skills-docs.md；../reference_src/01-host-claude-using-skills-in-claude.md）
- Copilot CLI 展示了另一类“治理叠加”的宿主形态：既支持 `SKILL.md` 并兼容 `.agents/skills`/`.claude/skills`/`.github/skills` 等目录约定；又以 `.github/copilot-instructions.md`、`.github/instructions/*.instructions.md` 与 `AGENTS.md` 承载 repo instructions；并通过 `~/.copilot/config.json`（trusted folders）与 allow/deny tool flags 把“权限与可操作范围”显式化。（Ref: ../reference_src/01-host-github-copilot-cli-command-reference.md；../reference_src/01-host-github-copilot-repo-custom-instructions.md；../reference_src/01-host-github-copilot-cli-configure.md；../reference_src/01-host-github-copilot-cli-allowing-tools.md）
- “触发语义”决定了用户是否会误用：Windsurf Workflows 明确是 manual-only（slash command 触发），而 skills 可以自动触发或 `@mention`；Cursor plugins 则在 IDE 内通过 marketplace 入口安装，但 CLI/IDE 的加载一致性可能成为落地风险点。（Ref: ../reference_src/00-shared-windsurf-workflows-docs.md；../reference_src/01-host-windsurf-skills-docs.md；../reference_src/01-host-cursor-forum-plugins-2-5.md；../reference_src/01-host-cursor-forum-cli-mode-plugins-bug.md）
- 互操作的关键风险在“字段与约束不一致”：Claude 自定义 skills 的 `name/description` 约束与 agentskills.io/open spec 的硬约束存在差异，这意味着跨宿主迁移时需要做 metadata 映射与校验，而不能只看“都叫 SKILL.md”。（Ref: ../reference_src/01-host-claude-creating-custom-skills.md；../reference_src/00-shared-agentskills-specification.md）

## 二轮新增趋势与难点

<!-- 趋势：有时间维度的证据；难点：区分个人/团队/平台；每条带 ../reference_src/*.md 回指 -->

- 趋势：宿主开始把“skills/agent 扩展”纳入企业治理产品线，典型信号是 Cursor 明确提出 private team marketplaces，并强调 central governance 与 security controls。（Ref: ../reference_src/01-host-cursor-plugins-blog-2026-02-17.md）
- 趋势：`.agents/skills` 作为跨宿主中间层正在被更多宿主识别与扫描（Codex/OpenCode/Windsurf/Copilot CLI 均在其规则中出现），使“同一个 skill folder 被多个宿主消费”更现实。（Ref: ../reference_src/01-host-openai-codex-agent-skills-docs.md；../reference_src/01-host-opencode-agent-skills-docs.md；../reference_src/01-host-windsurf-skills-docs.md；../reference_src/01-host-github-copilot-cli-command-reference.md）
- 趋势：progressive disclosure 从“实现细节”上升为宿主公开文档中的一等概念（Claude/Windsurf/Codex 均显式提及），意味着宿主在产品层面把“上下文成本”当作必须治理的资源。（Ref: ../reference_src/01-host-claude-what-are-skills.md；../reference_src/01-host-windsurf-skills-docs.md；../reference_src/01-host-openai-codex-agent-skills-docs.md）
- 难点：生态术语高度混乱（skills/plugins/extensions/workflows/commands/hooks），且不同宿主把相似概念映射到不同载体上；如果没有一套“宿主抽象对照表”，团队很容易把能力包当成错的触发入口来使用。（Ref: ../reference_src/01-host-openai-codex-agent-skills-docs.md；../reference_src/01-host-google-gemini-cli-extensions-reference.md；../reference_src/01-host-cursor-plugins-blog-2026-02-17.md；../reference_src/00-shared-windsurf-workflows-docs.md）
- 难点：跨宿主迁移不仅是文件拷贝，还包括目录作用域、冲突优先级、以及禁用/权限策略的对齐；否则同一个 skill 在不同宿主上的可用性与安全边界会出现漂移。（Ref: ../reference_src/01-host-openai-codex-agent-skills-docs.md；../reference_src/01-host-opencode-agent-skills-docs.md）
- 难点：宿主实现不一致会带来“同一插件在不同运行形态不可用”的失败模式（例如 Cursor IDE vs CLI mode 的 plugins 加载差异），这会直接影响 headless 自动化与团队一致性。（Ref: ../reference_src/01-host-cursor-forum-cli-mode-plugins-bug.md）

- 对最终报告最重要的判断是：Agent Skills/`SKILL.md` 更像“跨宿主的最低层协议”，而宿主生态的真实差异来自 discovery/load/invoke/permission/governance 这套宿主抽象模型；因此评估宿主不能只看“是否支持 skills”。（Ref: ../reference_src/00-shared-agentskills-specification.md；../reference_src/01-host-openai-codex-agent-skills-docs.md；../reference_src/01-host-opencode-agent-skills-docs.md；../reference_src/01-host-github-copilot-cli-command-reference.md）
- progressive disclosure 已经是事实标准级的宿主共识（Claude/Windsurf/Codex 均写入官方文档），这会直接影响 skill 的写法：description 必须可触发且低歧义，详细内容应放入 `SKILL.md` 与 references，而不是强行塞进元数据。（Ref: ../reference_src/01-host-claude-what-are-skills.md；../reference_src/01-host-windsurf-skills-docs.md；../reference_src/01-host-openai-codex-agent-skills-docs.md）
- “团队治理能力”是宿主分层的分水岭：Cursor 把分发与治理推向 team marketplaces；Copilot CLI 通过 trusted folders + allow/deny tools + repo/local settings scopes 把治理入口文件化；Codex 提供 plugins/skill-installer 与 config 禁用；Gemini extensions 支持按 user/workspace scope 启停并提供 `excludeTools` 等治理入口；OpenCode 提供 allow/ask/deny 与 tool 禁用；Claude 提供组织预置与共享 view-only。最终影响的是企业能否把 skills 当成可控依赖项。（Ref: ../reference_src/01-host-cursor-plugins-blog-2026-02-17.md；../reference_src/01-host-github-copilot-cli-configure.md；../reference_src/01-host-github-copilot-cli-allowing-tools.md；../reference_src/01-host-github-copilot-cli-command-reference.md；../reference_src/01-host-openai-codex-agent-skills-docs.md；../reference_src/01-host-google-gemini-cli-extensions-reference.md；../reference_src/01-host-opencode-agent-skills-docs.md；../reference_src/01-host-claude-using-skills-in-claude.md）
- 当前阶段最需要警惕的失败模式不是“skill 不够强”，而是“宿主语义差异导致不可用/不可控”：包括字段约束不一致导致互操作失败、workflow/skill/command 混淆导致找不到入口，以及 CLI/IDE 行为分裂导致自动化断裂。（Ref: ../reference_src/01-host-claude-creating-custom-skills.md；../reference_src/00-shared-windsurf-workflows-docs.md；../reference_src/01-host-cursor-forum-cli-mode-plugins-bug.md）
- 最值得继续追踪的对象应优先选择“官方文档里把机制讲清楚的宿主”：Copilot CLI（skills/MCP/plugins + settings scopes + tool permissions + trusted folders）、Codex（skills+plugins+扫描规则+配置）、Gemini CLI（extensions+MCP/skills bundle+excludeTools）、OpenCode（权限与禁用）、Windsurf（scope+PD+跨扫描）、Cursor（plugins+team governance 方向）、Claude（技能分类与组织治理）。（Ref: ../reference_src/01-host-github-copilot-cli-command-reference.md；../reference_src/01-host-github-copilot-cli-allowing-tools.md；../reference_src/01-host-github-copilot-cli-configure.md；../reference_src/01-host-openai-codex-agent-skills-docs.md；../reference_src/01-host-google-gemini-cli-extensions-reference.md；../reference_src/01-host-opencode-agent-skills-docs.md；../reference_src/01-host-windsurf-skills-docs.md；../reference_src/01-host-cursor-plugins-blog-2026-02-17.md；../reference_src/01-host-claude-what-are-skills.md）
