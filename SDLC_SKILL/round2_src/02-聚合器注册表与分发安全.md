### 这份片段在讲什么

这份上下文聚焦在软件开发智能体能力包的“分发层”。

原始研究的一个重要发现是：  
真正高密度的 skill 供给，不只来自官方仓库，还来自正在成型的第三方分发基础设施：

- 包管理器
- CLI 安装器
- marketplace
- registry
- 安全扫描与版本治理平台

这意味着 skill 已经开始像软件包一样流通，  
而不是零散地躺在 GitHub 仓库里。

### 从原始研究提炼出的核心结论

#### 1. 分发层正在形成独立价值

原始研究里最明确的三个代表：

- Vercel Labs / `npx skills`
- LobeHub Skills Marketplace
- Sundial Hub

它们分别代表三种完全不同的价值方向：

- 跨平台安装与兼容适配
- 海量供给与社区广度
- 安全、版本和企业级治理

#### 2. Vercel Labs 代表“跨平台安装层”

这一类平台的价值，不只是托管 skill，  
而是自动探测不同宿主的 skill 目录并完成兼容安装。

原始研究把它看成类似 npm 的安装基础设施：

- 支持跨 Git 仓库拉取
- 识别多种隐藏挂载路径
- 更适合把 skill 当作标准化依赖项来消费

#### 3. LobeHub 代表“超大规模社区供给层”

原始研究认为 LobeHub 的优势是规模和多样性：

- 量很大
- 技术栈覆盖广
- 有社区实验和宏观编排器

但问题也很明显：

- 噪音高
- 质量分布不均
- 必须依赖关键词、标签和人工筛选

所以它更像一个高密度矿区，  
不是天然高质量精选库。

#### 4. Sundial 代表“安全与版本治理层”

这是原始研究里非常关键的一条线。

因为 skill 不只是文本，  
还可能携带脚本、执行逻辑和行为约束，  
所以它天然带来新的供应链风险：

- prompt injection
- 行为劫持
- 静默更新
- 版本漂移

Sundial 这种平台的意义在于把 skill 当成正式制品来治理：

- 版本化
- 安全扫描
- 验证标记
- 企业级可控分发

## 二轮新增证据

<!-- 每条新增事实都带 ../reference_src/*.md 回指 -->

- `vercel-labs/skills` README 给出 `npx skills add <source>` 的安装模型，并支持多种 source formats（GitHub shorthand、repo URL、repo 子路径、GitLab、任意 git URL、local path）。（Ref: ../reference_src/00-shared-vercel-skills-cli-readme.md）
- `vercel-labs/skills` README 给出分发治理相关能力：可按 agent（`--agent`）与按 skill（`--skill`）过滤安装；区分 project vs global 安装范围；并提供 symlink（推荐）与 copy 两种落盘方式。（Ref: ../reference_src/00-shared-vercel-skills-cli-readme.md）
- Vercel Skills 的 Lock Files 文档给出“工程化可追溯状态”的硬机制：global lock file 位于 `~/.agents/.skill-lock.json`，local lock file 位于 `./skills-lock.json` 且明确 intended to be checked into version control；并以 `skillFolderHash/computedHash` 支撑确定性与协作友好（最小化 merge conflicts）。（Ref: ../reference_src/02-dist-vercel-skills-docs-lock-files.md）
- Vercel Skills 的 Update System 文档给出更新检查机制：`skills check/update` 读取 lock file 中的 `skillFolderHash`，并通过远端 update API（`https://add-skill.vercel.sh/check-updates`）计算最新 folder hash 与本地 hash 对比，返回更新列表。（Ref: ../reference_src/02-dist-vercel-skills-docs-update-system.md）
- Vercel Skills CLI 的实现补充了安全治理线索：安装流程会通过 audit API（`https://add-skill.vercel.sh/audit`）拉取并展示 `Security Risk Assessments`（advisory），且 audit 失败/超时不会阻塞安装。（Ref: ../reference_src/02-dist-vercel-skills-cli-security-audit-api.md）
- OWASP LLM Top 10 v1.1 明确列出 Prompt Injection、Insecure Output Handling、Supply Chain Vulnerabilities、Insecure Plugin Design、Excessive Agency 等风险分类，可作为 skill 分发与执行链路的风险基线。（Ref: ../reference_src/00-shared-owasp-llm-top10-v1-1.md）
- MCP Registry 的官方文档将其定位为 MCP servers 的“app store 式清单”，并明确 registry 只托管 metadata、不托管 artifacts；发布通过官方 `mcp-publisher` CLI 完成，且发布后 metadata 不可变、需要唯一 version。（Ref: ../reference_src/03-supply-mcp-registry-github-readme.md；../reference_src/03-supply-mcp-registry-quickstart-publish.md；../reference_src/03-supply-mcp-registry-versioning.md）
- MCP Registry 进一步把“可信发布”机制化：认证方法决定 server name namespace（GitHub 或 reverse-DNS 域名），并针对 npm/PyPI/NuGet/OCI/MCPB 等不同 artifact types 定义 ownership verification；同时官方说明 registry 本身不承诺 uptime/durability 且 moderation policy 相当 permissive，增值治理预期下放到 aggregators/subregistries（可通过 `_meta` 注入安全扫描等元数据）。这为“registry/hub 分发模式”的治理边界提供了可复用参照。（Ref: ../reference_src/03-supply-mcp-registry-authentication.md；../reference_src/03-supply-mcp-registry-package-types.md；../reference_src/03-supply-mcp-registry-aggregators.md；../reference_src/03-supply-mcp-registry-moderation-policy.md）
- Sundial 首页宣称其 Skills 库具备 “70,000+ community-built skills ... verified and ready to use”，并把 “verified” 作为分发层价值主张之一。（Ref: ../reference_src/02-dist-sundial-home.md）
- Sundial 的 Security 文档明确其在展示结果前进行多重验证检查，并点名使用 Cisco AI Skill Scanner、Semgrep 与 model-based review；对模糊案例加入 manual review；并在 UI 展示 severity 与触发原因摘要。（Ref: ../reference_src/02-dist-sundial-docs-security.md）
- Sundial CLI 文档明确其会 auto-detect agent 并安装到对应目录（示例：`.claude/skills/`、`.cursor/skills/`、`.codex/skills/`、`.gemini/skills/`），并区分 global vs local；鉴权不仅用于 publish，也会解锁 private skill discovery。（Ref: ../reference_src/02-dist-sundial-docs-cli.md）
- Sundial Push & publish 文档明确每次 push 会创建 immutable version snapshot；版本来自 `SKILL.md` frontmatter 的 `version`，且当版本不更新时会 auto-bump（支持整数与 semver），并提供 public/private 可见性与 team 发布等选项。（Ref: ../reference_src/02-dist-sundial-docs-push-publish.md）
- Sundial Specification 文档给出其 `SKILL.md` 字段与实践约束（如 `name` 需 lowercase 且匹配父目录，`description` max 1024；建议主 `SKILL.md` 控制在 500 行内，将详细资料移至 `references/`；并支持 `allowed-tools`/`compatibility` 等字段）。（Ref: ../reference_src/02-dist-sundial-docs-specification.md）
- Vercel Skills 的官方文档给出 skills CLI 的 source detection logic（解析顺序）与支持格式，并明确支持 HTTP(S) 的 well-known endpoint：`/.well-known/skills/index.json`，这为“分发层向 registry/目录标准化演进”提供了机制证据。（Ref: ../reference_src/02-dist-vercel-skills-docs-source-formats.md）
- Vercel Skills CLI 的官方代码实现补齐了 well-known endpoint 的“真实 schema 与安全校验”：优先尝试 `/.well-known/agent-skills/index.json`（preferred），并回退到 `/.well-known/skills/index.json`（legacy）；`index.json` 结构为 `{ skills: [{ name, description, files }] }`，且校验 `files` 必须包含 `SKILL.md` 并禁止路径穿越（`..`、绝对路径等）。（Ref: ../reference_src/02-dist-vercel-skills-well-known-index-schema.md）
- Cloudflare 的 Agent Skills Discovery RFC（v0.2.0）给出 well-known 发现机制的一手规范：将路径从 `/.well-known/skills/` 重命名为 `/.well-known/agent-skills/`，要求提供 `/.well-known/agent-skills/index.json`；并用 `$schema` + `type/url/digest` 取代 legacy `files` 格式，要求 clients MUST 校验 SHA-256 digest、默认不执行 scripts，并对 archive 分发与解包安全（路径穿越、symlink/hardlink 逃逸、解压炸弹）给出强约束。（Ref: ../reference_src/00-shared-cloudflare-agent-skills-discovery-rfc-0-2-0.md）
- X 官方文档给出 skills CLI 的安装入口（`npx skills add https://docs.x.com`），并同时公开两套 well-known endpoints：`/.well-known/agent-skills/index.json`（含 `$schema/type/url/digest` 的 v0.2.0）与 legacy `/.well-known/skills/index.json`（`files` 格式），提供了“新旧口径并存以兼容生态”的实证样本。（Ref: ../reference_src/02-dist-x-docs-skill-md.md；../reference_src/02-dist-x-well-known-agent-skills-index-json.md；../reference_src/02-dist-x-well-known-skills-index-json.md）
- Backstage 官方文档同样把 `npx skills add` 与其 well-known index 绑定（读取 `https://backstage.io/.well-known/skills/index.json`），并说明后续更新会提供合并 upstream changes 的选项；其 index.json 实例仍为 legacy `files` 格式且未提供 `agent-skills` 端点，体现生态迁移存在滞后与口径分裂。（Ref: ../reference_src/02-dist-backstage-docs-ai-skills.md；../reference_src/02-dist-backstage-well-known-skills-index-json.md）
- LobeHub 官方仓库 README 在 Features 中提出 “10,000+ Skills” 的规模主张，并将其描述为“10,000+ tools + MCP-compatible plugins”的能力库。（Ref: ../reference_src/02-dist-lobehub-github-readme.md）
- npm 上存在 `@lobehub/market-cli`（LobeHub Marketplace CLI）包元数据（版本/主页/issue 入口），为“marketplace 可能配套 CLI 分发入口”提供存在性证据。（Ref: ../reference_src/02-dist-lobehub-market-cli-npm.md）
- `@lobehub/market-cli` 的 npm tarball 中包含可复现的 CLI 实现：可执行入口为 `lhm`，并提供 `skills search/view/install`（支持 `--version/--agent/--global` 等）以及 `mcp` marketplace 浏览与交互命令；同时 CLI 内硬编码了多宿主 skills 落盘路径映射（如 `./.claude/skills`、`./.agents/skills`、`./.cursor/skills` 等），说明 LobeHub 不仅是“网页目录”，而是具备实际安装语义的分发入口。（Ref: ../reference_src/02-dist-lobehub-market-cli-tarball-0-0-28.md）

## 二轮新增机制理解

<!-- 从“描述”上升到“为什么这样设计”的解释；每条带 ../reference_src/*.md 回指 -->

- 分发层的核心不是“再做一个列表”，而是把 skills 变成可被工程化消费的制品：提供统一安装入口（CLI）、跨宿主目录探测/落盘、可追溯安装状态（lock files）、以及更新策略（symlink vs copy、hash-based update checking、snapshot/versioning）。（Ref: ../reference_src/00-shared-vercel-skills-cli-readme.md；../reference_src/02-dist-vercel-skills-docs-lock-files.md；../reference_src/02-dist-vercel-skills-docs-update-system.md；../reference_src/02-dist-sundial-docs-cli.md；../reference_src/02-dist-sundial-docs-push-publish.md）
- “registry/目录标准化”的关键机制是可被机器发现的索引端点：Vercel Skills 官方文档暴露了 `/.well-known/skills/index.json` 入口，而其 CLI 实现进一步把该机制具体化为优先 `/.well-known/agent-skills/index.json` + legacy fallback，并定义 `index.json` schema 与校验规则（必须包含 `SKILL.md`、禁止路径穿越等）。这使 registry 不再是“页面目录”，而是可被工具链可靠消费的协议化入口。（Ref: ../reference_src/02-dist-vercel-skills-docs-source-formats.md；../reference_src/02-dist-vercel-skills-well-known-index-schema.md）
- “可信分发”需要把安全与版本治理嵌入默认链路：Sundial 在机制上同时提供 immutable snapshots（每次 push 创建不可变版本）、版本 auto-bump（当发布者未更新版本）、以及展示前的多重扫描与 review（Cisco AI Skill Scanner、Semgrep、model-based review、manual review）与 UI 风险摘要。（Ref: ../reference_src/02-dist-sundial-docs-push-publish.md；../reference_src/02-dist-sundial-docs-security.md）
- “规模市场”与“可信治理”往往是两条不同路径：尽管 LobeHub 已出现可安装的 CLI 入口（`lhm skills install` + 版本参数 + 多宿主落盘映射），但其 verification/安全扫描/不可变版本等治理链路仍缺更强一手证据，不能仅凭“规模口径”或“安装命令存在”推断其可信度。（Ref: ../reference_src/02-dist-lobehub-github-readme.md；../reference_src/02-dist-lobehub-market-cli-tarball-0-0-28.md）
- 分发层的风险面与 LLM 安全分类直接耦合：当 skills 可携带脚本、工具调用与插件封装，风险会覆盖 supply chain、insecure plugin design、excessive agency 等，需要把“安装前/安装后”的验证与权限控制纳入平台设计。（Ref: ../reference_src/00-shared-owasp-llm-top10-v1-1.md；../reference_src/02-dist-sundial-docs-security.md）

## 二轮新增趋势与难点

<!-- 趋势：有时间维度的证据；难点：区分普通用户/团队治理；每条带 ../reference_src/*.md 回指 -->

- 趋势：分发平台开始把“安全扫描 + 分级呈现 + 人工复核”做成默认链路，而不是可选项，Sundial 明确展示了自动化扫描（Cisco AI Skill Scanner、Semgrep、model review）与 manual review 的组合。（Ref: ../reference_src/02-dist-sundial-docs-security.md）
- 趋势：版本治理从“可选”走向“强制可追溯”：Sundial 的 immutable version snapshots + auto-bump 机制，将技能发布从“覆盖式更新”转为“可追溯版本序列”。（Ref: ../reference_src/02-dist-sundial-docs-push-publish.md）
- 趋势：跨宿主安装与私有发现成为分发层的核心价值点之一：Sundial CLI 会 auto-detect agent 并安装到对应目录，且鉴权既用于 publish 也用于 private discovery。（Ref: ../reference_src/02-dist-sundial-docs-cli.md）
- 趋势：分发层开始出现更标准化的“可发现入口”（well-known endpoint），并从“实现口径”走向“规范口径 + 兼容迁移”：Cloudflare RFC v0.2.0 将路径迁移到 `/.well-known/agent-skills/` 并引入 `$schema/type/url/digest` 与强制 digest 校验；Vercel CLI 实现则体现为“agent-skills preferred + skills legacy fallback”；而发布者侧已出现真实采用但口径分裂（X、Cognite 同时提供新旧 endpoints；Backstage 仅提供 legacy `/.well-known/skills/index.json`）。这意味着短期内安装器需要更强的向后兼容与安全治理策略。（Ref: ../reference_src/00-shared-cloudflare-agent-skills-discovery-rfc-0-2-0.md；../reference_src/02-dist-vercel-skills-well-known-index-schema.md；../reference_src/02-dist-x-well-known-agent-skills-index-json.md；../reference_src/02-dist-x-well-known-skills-index-json.md；../reference_src/02-dist-cognite-well-known-agent-skills-index-json.md；../reference_src/02-dist-cognite-well-known-skills-index-json.md；../reference_src/02-dist-backstage-well-known-skills-index-json.md）
- 难点：安装方式本身就会影响升级治理与安全策略（symlink vs copy）。Vercel Skills CLI 虽推荐 symlink，但在企业环境、隔离环境与审计需求下可能不得不选择 copy，进而引入版本漂移与更新成本。（Ref: ../reference_src/00-shared-vercel-skills-cli-readme.md）
- 难点：大规模 marketplace 的主要问题是噪音与质量分布，需要额外的筛选/验证/治理层；仅有“数量级”口径不足以支持团队级采纳决策。（Ref: ../reference_src/02-dist-lobehub-github-readme.md；../reference_src/02-dist-sundial-docs-security.md）
- 难点：安全扫描本身存在覆盖面与误报/漏报风险，且平台文档更多是机制描述，仍需补充可复现样例与规则覆盖面作为实证层支撑。（Ref: ../reference_src/02-dist-sundial-docs-security.md）

## 当前判断（二轮综合后）

<!-- 综合第一轮和第二轮的判断；每条判断带 ../reference_src/*.md 回指；如推翻/修正需注明 -->

- 分发层已经出现清晰分工：Vercel Skills 更像“跨宿主安装层/来源解析层”（多 source formats + 落盘策略）；Sundial 更像“带验证与版本治理的 registry/hub”；LobeHub 更像“规模市场 + 安装入口”（CLI 可安装 skills 并做多宿主落盘映射），但其 verification/安全扫描/治理机制仍需进一步证据核验。（Ref: ../reference_src/00-shared-vercel-skills-cli-readme.md；../reference_src/02-dist-sundial-home.md；../reference_src/02-dist-lobehub-github-readme.md；../reference_src/02-dist-lobehub-market-cli-tarball-0-0-28.md）
- 对团队与企业来说，分发层的关键价值不在“找到更多 skills”，而在“把 skills 变成可控依赖项”：可提交的 lock files（团队共享）、hash-based 更新检查、可追溯版本（snapshots/versioning）、可理解的安全报告、以及对私有技能的发现与访问控制。（Ref: ../reference_src/02-dist-vercel-skills-docs-lock-files.md；../reference_src/02-dist-vercel-skills-docs-update-system.md；../reference_src/02-dist-sundial-docs-push-publish.md；../reference_src/02-dist-sundial-docs-security.md；../reference_src/02-dist-sundial-docs-cli.md）
- well-known endpoint 机制提示分发层正在向“协议化 registry”演进：规范侧（Cloudflare RFC v0.2.0）已引入 `agent-skills` 路径与 digest 校验语义；实现侧（Vercel CLI）已落地 schema/校验与优先路径；发布者侧（X/Cognite/Backstage）也已出现真实采用但新旧 schema 并存（部分双栈、部分 legacy-only）。下一阶段的关键不再是“有没有入口”，而是“多安装器/多宿主能否在同一套 schema/versioning/security 语义上收敛”。（Ref: ../reference_src/00-shared-cloudflare-agent-skills-discovery-rfc-0-2-0.md；../reference_src/02-dist-vercel-skills-well-known-index-schema.md；../reference_src/02-dist-x-well-known-agent-skills-index-json.md；../reference_src/02-dist-backstage-docs-ai-skills.md；../reference_src/02-dist-cognite-well-known-agent-skills-index-json.md；../reference_src/02-dist-backstage-well-known-skills-index-json.md）
- 分发安全的底线应以 OWASP LLM Top 10 的供应链/插件/过度代理风险分类为参考，再结合平台扫描与人工复核链路来落地；否则“可安装”会快速演化成“不可控”。（Ref: ../reference_src/00-shared-owasp-llm-top10-v1-1.md；../reference_src/02-dist-sundial-docs-security.md）
