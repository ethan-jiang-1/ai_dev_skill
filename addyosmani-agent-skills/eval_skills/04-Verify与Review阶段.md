# Verify 与 Review 阶段

## 这一阶段在解决什么

很多团队的问题不是不会做，而是：

- 出错后不会系统排障
- 看起来像修好了，其实没复现过
- merge 前 review 太浅
- 安全和性能只靠感觉
- 代码越堆越复杂却没人主动减法

Verify 与 Review 阶段正是在堵这些口子。

## 1. browser-testing-with-devtools

### 职责

- 让 agent 在浏览器运行态里看 DOM、console、network、performance
- 让 UI 验证不再只靠源码猜测

### 我的理解

这相当于给 agent 补了“眼睛”。  
对于前端任务，这个 skill 的存在会显著改变调试质量。

### 优点

- 对视觉问题、交互问题、运行时错误非常有效
- 能直连真实浏览器状态
- 非常适合和 TDD 配合使用

### 缺点

- 强依赖外部工具
- 只适合浏览器场景
- 运行态数据也需要正确解读

### 判断

- 验证价值：极高
- 通用性：中等
- 工具依赖：很高

## 2. debugging-and-error-recovery

### 职责

- 遇到错误时要求停线
- 先复现，再定位，再缩小，再修复，再加防护

### 我的理解

这是一个非常成熟的 debug workflow。  
它最宝贵的地方是：阻止“边错边继续做新功能”。

### 优点

- 通用性极强
- 适合测试失败、构建失败、运行时 bug
- Guard 阶段能防止重复踩坑

### 缺点

- 对非常简单的报错会显得重
- 需要日志、测试、监控等辅助设施时效果更强

### 判断

- 验证价值：极高
- 通用性：极高
- 推荐优先级：极高

## 3. code-review-and-quality

### 职责

- 从正确性、可读性、架构、安全、性能五个维度做 review
- 控制变更规模
- 先看测试再看实现

### 我的理解

它其实是在把“资深 reviewer 的脑内审查框架”格式化。

### 优点

- 维度完整
- 适合做最终质量闸门
- 和安全、性能 skill 联动很好

### 缺点

- 如果每次都 full review，会比较重
- 审查质量仍依赖 reviewer 的理解能力

### 判断

- 审查价值：极高
- 通用性：极高
- 推荐优先级：极高

## 4. code-simplification

### 职责

- 在不改变行为的前提下，降低代码复杂度
- 防止 AI 生成过度抽象或过度聪明的代码

### 我的理解

它是对 agent 一个很必要的反制。  
AI 很擅长把简单事情写复杂，所以这个 skill 不是可选装饰，而是重要纠偏器。

### 优点

- 适合后期整理
- 很适合 review 之后配套使用
- Chesterton's Fence 的约束很成熟

### 缺点

- 没测试保护就很危险
- “更简单”有时带主观性

### 判断

- 审查价值：高
- 通用性：高
- 推荐优先级：高

## 5. security-and-hardening

### 职责

- 强制把安全作为默认边界
- 约束输入、鉴权、密钥、外部服务、输出编码等关键点

### 我的理解

这是 Review 阶段里最不能妥协的 skill。  
它不像代码风格可以讨论，有很多规则本身就是红线。

### 优点

- 可执行性强
- 规则边界清楚
- 很适合 web 系统

### 缺点

- 偏 web 场景
- 有些组织要求需要具体化到项目基础设施

### 判断

- 审查价值：极高
- 通用性：高
- 推荐优先级：涉及外部输入时必须启用

## 6. performance-optimization

### 职责

- 要求先测量再优化
- 识别真实瓶颈而不是拍脑袋
- 优化后再回测并加守护

### 我的理解

它最大的价值不是优化本身，而是遏制“玄学调优”。

### 优点

- 思路非常健康
- 前后端都覆盖
- 容易和监控、profiling、budget 体系联动

### 缺点

- 工具要求高
- 没有基线时不容易严格落地

### 判断

- 审查价值：高
- 通用性：中高
- 推荐优先级：出现性能约束时必须启用

## Verify 与 Review 阶段的最佳组合

### 普通功能合并前

- `code-review-and-quality`
- `debugging-and-error-recovery`
- `code-simplification`

### 前端交互问题

- 上面组合
- 加 `browser-testing-with-devtools`

### 高风险功能

- 上面组合
- 加 `security-and-hardening`
- 加 `performance-optimization`

## 这一阶段最常见的虚假完成

- 没复现 bug 就说修好了
- 没看运行态就说页面正常
- 只看代码风格，不看系统正确性
- 没安全检查就说“只是小改”
- 没性能基线却声称“已经优化”
- 复杂度越来越高但没人做减法

## 我的总评

如果说 Build 阶段在防止 agent 写歪，  
那 Verify 与 Review 阶段就是在防止 agent 自信地交付错误结果。
