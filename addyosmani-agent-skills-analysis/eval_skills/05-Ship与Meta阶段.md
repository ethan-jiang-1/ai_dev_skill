# Ship 与 Meta 阶段

## 这一阶段在解决什么

很多工程问题不是出在写代码时，而是出在：

- 提交历史混乱
- CI 不守门
- 老系统下不来
- 决策没记录
- 上线无灰度、无回滚、无监控

Ship 阶段负责把这些“后期灾难源”提前制度化。  
另外还有一个 Meta skill，专门决定什么时候该调用哪个 skill。

## 1. git-workflow-and-versioning

### 职责

- 规范提交粒度
- 规范分支策略
- 让提交历史成为安全网和可追溯记录

### 我的理解

AI 生成代码很快，所以越快越需要历史纪律。  
这个 skill 的真正价值是把“速度”变成“可回退的速度”。

### 优点

- 原子提交非常适合 agent
- save point 思想很实战
- 对 code review、debug、回滚都直接有帮助

### 缺点

- 对随手改随手堆的人有门槛
- 与现有 branching model 可能需要磨合

### 判断

- 工程价值：极高
- 通用性：极高
- 推荐优先级：默认开启

## 2. ci-cd-and-automation

### 职责

- 把 lint、typecheck、test、build、安全检查自动化
- 把 quality gate 变成机器约束

### 我的理解

这是整个 skill 体系的“组织级执行器”。  
没有 CI，很多规范都只是一种愿望。

### 优点

- 把流程要求转成硬门槛
- 对团队一致性很重要
- 非常适合长期项目

### 缺点

- 建设成本较高
- 如果 pipeline 太重，反馈速度会下降

### 判断

- 工程价值：极高
- 通用性：高
- 推荐优先级：正式项目极高

## 3. deprecation-and-migration

### 职责

- 管理旧系统退场
- 规划迁移路径
- 把删除代码当成工程成就

### 我的理解

这是一种非常成熟的大型工程视角。  
大多数 skill 库都在讲怎么新增功能，很少讲怎么安全地下线旧功能。

### 优点

- 适合长期演进
- 对系统债务治理很有价值
- advisory / compulsory 的区分很实用

### 缺点

- 小项目使用频率低
- 依赖 usage 观测、替代方案、迁移能力

### 判断

- 工程价值：中高
- 通用性：中等
- 推荐优先级：大型系统很高

## 4. documentation-and-adrs

### 职责

- 记录决策原因
- 维护架构记忆
- 给未来的人和 agent 留下 why

### 我的理解

这个 skill 的重要性常常被低估。  
代码能告诉你现在长什么样，ADRs 才能告诉你为什么会长成这样。

### 优点

- 长期维护价值高
- 对新人和 agent 都很友好
- 能减少重复解释成本

### 缺点

- 容易被拖延
- 写得空会变成文档噪音

### 判断

- 工程价值：高
- 通用性：高
- 推荐优先级：架构与公共接口变化时很高

## 5. shipping-and-launch

### 职责

- 管理上线前检查、灰度发布、监控、回滚
- 保证上线不是盲投

### 我的理解

这不是“怎么部署”的教程，而是“怎么降低发布风险”的框架。

### 优点

- 很完整
- feature flag 生命周期写得很实战
- 兼顾安全、性能、文档、基础设施

### 缺点

- 对没有灰度和观测能力的团队会显得理想化
- 小项目会觉得重

### 判断

- 工程价值：极高
- 通用性：高
- 推荐优先级：生产上线必须高

## 6. using-agent-skills

### 职责

- 判断任务来了应该先用哪个 skill
- 规定跨 skill 的串联顺序
- 作为整个仓库的调度层

### 我的理解

它不是直接创造业务价值的 skill，  
但它能显著降低“明明有 skill 却不用”的失误。

### 优点

- 让 skill 发现更清楚
- 让新人更容易上手整套体系
- 把完整生命周期串起来了

### 缺点

- 更像交通规则，不是业务技能
- 如果底层 skill 执行不力，它也不能单独兜底

### 判断

- 工程价值：高
- 通用性：极高
- 推荐优先级：默认保留

## Ship 阶段最重要的现实意义

- 它把“工程完成”从代码提交延伸到真正可发布
- 它把“历史质量”和“发布质量”纳入同一套 discipline
- 它提醒团队：新增功能只是开始，迁移、记录、交付、回滚同样是工程

## 我的总评

Ship 阶段和 Meta 层让这套仓库从“开发规范”升级成“完整工程生命周期规范”。  
如果没有这一层，这个项目会更像 coding workflow；有了这一层，它更像 production engineering workflow。
